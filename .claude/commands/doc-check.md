# /doc-check

Проверить полноту документа и получить рекомендацию: продолжать, остановиться или подождать.

## Использование

```
/doc-check [путь к документу]
/doc-check                      # проверить активный документ
```

## Примеры

```
/doc-check docs/02-features/F007-update-check.adoc
/doc-check docs/05-decisions/ADR-001-kotlinx-serialization.adoc
```

## Инструкции для агента

$ARGUMENTS содержит путь к документу. Если пустой — запросить путь у пользователя.

### Шаги выполнения

1. **Прочитать документ** по указанному пути

2. **Определить тип документа:**
   - Feature (в `02-features/`)
   - ADR (в `05-decisions/`)
   - Concept (в `01-concepts/`)
   - Interface (в `03-interfaces/`)

3. **Определить текущий уровень зрелости:**
   - L0 Draft — идея есть, структура неполная
   - L1 Minimal Viable — цель ясна, основные user stories
   - L2 Development Ready — AC есть, edge cases описаны
   - L3 Complete — всё тестируемо, все вопросы отвечены

4. **Проверить stakeholder questions** (в зависимости от типа):

   **Для Features:**
   | Stakeholder | Вопрос |
   |-------------|--------|
   | Product Owner | Какую проблему решает? |
   | Product Owner | Какова ценность? |
   | Разработчик | Что делать? |
   | Разработчик | Какие AC? |
   | Разработчик | Какие edge cases? |
   | Тестировщик | Как проверить? |
   | Архитектор | Какие модули затрагивает? |

   **Для ADR:**
   | Stakeholder | Вопрос |
   |-------------|--------|
   | Будущий разработчик | Почему выбрано это решение? |
   | Будущий разработчик | Какие альтернативы рассматривались? |
   | Будущий разработчик | Какие trade-offs приняты? |

5. **Проверить testability** (для Features):
   - Каждый AC должен быть в формате Given/When/Then
   - Не должно быть неоднозначных формулировок ("правильно", "быстро", "удобно")

6. **Выдать рекомендацию:**

   **STOP** если:
   - Достигнут целевой уровень
   - Все stakeholder questions отвечены
   - Нет блокирующих вопросов

   **ITERATE** если:
   - Stakeholder не может работать по документу
   - Есть конкретные проблемы которые нужно исправить
   - Нужен переход на следующий уровень

   **WAIT** если:
   - Нужен input который недоступен
   - Зависимость от непринятого решения

### Формат ответа

```markdown
## Анализ документа: [название файла]

**Тип:** [Feature / ADR / Concept / Interface]
**Текущий уровень:** [L0 / L1 / L2 / L3] ([название уровня])

### Stakeholder Questions

| Stakeholder | Вопрос | Статус |
|-------------|--------|--------|
| [роль] | [вопрос] | ✅ / ❌ [причина] |

### Testability (для Features)

| AC | Статус | Проблема |
|----|--------|----------|
| [ID] | ✅ / ❌ | [что не так] |

### Найденные проблемы

1. **[Краткое описание]**
   Проблема: [детали]
   Решение: [как исправить]

### Рекомендация

**[STOP / ITERATE / WAIT]**

[Обоснование в 2-3 предложениях]

[Если ITERATE — список конкретных действий]
[Если WAIT — что нужно подождать]
```
