= ADR-002: Pipeline Architecture для обработки метаданных
:icons: font

== Метаданные

[cols="1,2"]
|===
| Дата | 2026-01-15
| Авторы | Dependanger Team
| Затронутые компоненты | dependanger-effective, dependanger-api
|===

== Контекст

Обработка DSL конфигурации в готовые артефакты (TOML, BOM) требует множества этапов:

1. Конвертация DSL → промежуточное представление
2. Импорт версий из внешних BOM
3. Фильтрация библиотек по дистрибуции
4. Резолюция версий (version references → конкретные версии)
5. Фильтрация бандлов
6. Валидация результата
7. Проверка обновлений
8. Анализ совместимости

Необходим способ организации этих этапов, который обеспечит:

* Чёткий порядок выполнения
* Возможность добавления/удаления этапов
* Изоляцию логики каждого этапа
* Поддержку асинхронных операций

== Решение

Выбрана **Pipeline Architecture** с упорядоченными процессорами.

[source]
----
DSL → [Processor 1] → [Processor 2] → ... → [Processor N] → EffectiveMetadata
----

Каждый процессор:

* Имеет уникальный `name`
* Имеет числовой `order` для определения порядка
* Получает `EffectiveMetadata` и возвращает модифицированную версию
* Может добавлять диагностику (errors, warnings)

== Обоснование

=== Рассмотренные альтернативы

**1. Monolithic processor:**

[source,kotlin]
----
fun process(metadata: DependangerMetadata): EffectiveMetadata {
    // Вся логика в одном месте
}
----

*Недостаток:* Нарушение SRP, сложность тестирования, невозможность кастомизации.

**2. Chain of Responsibility (без order):**

*Недостаток:* Неявный порядок, сложность управления последовательностью.

**3. Event-driven pipeline:**

*Недостаток:* Избыточная сложность для синхронной обработки.

=== Преимущества выбранного подхода

[cols="1,2"]
|===
| Критерий | Обоснование

| Explicit ordering
| Числовой `order` явно определяет последовательность

| Modularity
| Каждый процессор изолирован и тестируется отдельно

| Extensibility
| Новые процессоры добавляются через SPI без изменения core

| Optional features
| Процессоры можно включать/выключать (BOM import, updates, analysis)

| Diagnostics
| Каждый процессор может сообщать о проблемах через `ProcessingContext`

| Async support
| `suspend fun process()` поддерживает I/O операции
|===

== Последствия

=== Положительные

* Чёткое разделение ответственности между процессорами
* Лёгкое тестирование каждого процессора в изоляции
* Пользователи могут добавлять кастомные процессоры
* Feature processors можно отключать через `withoutX()` API

=== Отрицательные

* Накладные расходы на создание промежуточных копий `EffectiveMetadata`
* Необходимость следить за order чтобы избежать конфликтов
* Сложнее отслеживать изменения, вносимые каждым процессором

=== Нейтральные

* Все процессоры должны работать с immutable data
* Диагностика собирается в `ProcessingContext`

== Реализация

=== SPI Interface

[source,kotlin]
----
interface EffectiveMetadataProcessor {
    val name: String
    val order: Int
    
    suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext
    ): EffectiveMetadata
}
----

=== Order Convention

Полный список процессоров и их порядок выполнения описан в xref:../02-features/F003-processing-pipeline.adoc#standard-processors[F003: Processing Pipeline — Стандартные процессоры].

=== Пример использования

[source,kotlin]
----
val dependanger = Dependanger.fromDsl { /* ... */ }
    .addProcessor(MyCustomProcessor())
    .disableProcessor("update-check")  // Отключить проверку обновлений
    .build()

val result = dependanger.process()
----
