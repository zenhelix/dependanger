= Архитектура системы
:toc: left
:toclevels: 3
:sectnums:
:icons: font

[abstract]
--
Документ описывает архитектуру системы Dependanger:
архитектурные слои, модули, потоки данных и ключевые SPI интерфейсы.

Системный контекст: xref:../00-overview.adoc#system-context[Overview → Контекст].
--

== Принципы архитектуры

[cols="1,3",options="header"]
|===
| Принцип | Описание

| Направление зависимостей
| Зависимости направлены от периферии к ядру: interfaces → api → features → core

| Изоляция домена
| Core модуль не зависит от инфраструктуры (HTTP, файловая система, форматы)

| Расширяемость через SPI
| Новый функционал добавляется через реализацию интерфейсов, без изменения core

| Единая точка входа
| Весь функционал доступен через API фасад, интерфейсы (CLI, Gradle) — тонкие обёртки

| Абстракция форматов
| Сериализация модели абстрагирована через `SerializationFormat<T>` SPI

| Расширяемая генерация
| Генераторы артефактов добавляются через `ArtifactGenerator<T>` SPI

| Экспортный слой на границе
| Генераторы — точка экспорта. Артефакты (TOML, BOM) публикуются для внешних проектов
|===

== Архитектурные слои

Система организована в 4 архитектурных слоя с дополнительными SPI-реализациями:

[mermaid,target=architecture-layers,format=svg]
----
flowchart TB
    subgraph L1["1. INTERFACES"]
        I1["CLI, Gradle Plugin"]
    end
    subgraph L2["2. API"]
        A1["Фасад Dependanger, координация"]
    end
    subgraph L3["3. MODEL"]
        M1["core (DSL), metadata, effective"]
    end
    subgraph L4["4. FEATURES"]
        F1["Процессоры: resolver, updates, etc."]
    end
    subgraph SPI["SPI Implementations"]
        S1["Генераторы, Резолверы"]
    end
    L1 --> L2
    L2 --> L3
    L3 --> L4
    L2 -.->|"координирует через SPI"| SPI
    L3 -.->|"использует через SPI"| SPI
----

NOTE: Слои определяют направление основных зависимостей (от периферии к ядру).
SPI-реализации (генераторы, резолверы) не являются отдельным слоем — это точки расширения, которые API и MODEL координируют через абстрактные интерфейсы.
Зависимость идёт от реализаций к интерфейсам, а не наоборот.

=== Описание слоёв

[cols="1,2,2",options="header"]
|===
| Слой | Описание | Модули

| INTERFACES
| Два равноправных входа: CLI выполняет операции с metadata.json, Gradle Plugin парсит Kotlin DSL.
| `dependanger-gradle-plugin`, `dependanger-cli`

| API
| Фасад и координация. Единая точка входа для всего функционала. Координирует SPI-реализации через абстрактные интерфейсы.
| `dependanger-api`

| MODEL
| Доменные модели: Core (DSL, сущности), metadata.json, EffectiveMetadata. Определяет SPI интерфейсы.
| `dependanger-core`, `dependanger-metadata-json`, `dependanger-effective`

| FEATURES
| Опциональные процессоры pipeline. Могут быть отключены. Реализуют `EffectiveMetadataProcessor` SPI.
| `dependanger-maven-resolver`, `dependanger-updates`, `dependanger-analysis`
|===

=== SPI-реализации (не слой)

Генераторы артефактов — это SPI-реализации интерфейса `ArtifactGenerator<T>`, а не отдельный архитектурный слой:

[cols="1,2,2",options="header"]
|===
| Тип | Описание | Модули

| Generators
| Преобразуют `EffectiveMetadata` в TOML/BOM. Реализуют `ArtifactGenerator` SPI.
| `dependanger-generator-toml`, `dependanger-generator-bom`
|===

Такой подход соответствует принципу E1 (расширяемость через SPI): новые генераторы добавляются без изменения API или MODEL слоёв.

== SPI Интерфейсы

Система использует Service Provider Interface (SPI) для расширяемости.

=== SerializationFormat

Абстракция формата сериализации `DependangerMetadata`.

[source,kotlin]
----
interface SerializationFormat<T> {
    val formatId: String  // "json", "yaml"
    val fileExtension: String  // ".json", ".yaml"

    fun serialize(metadata: DependangerMetadata): T
    fun deserialize(input: T): DependangerMetadata
    fun write(metadata: DependangerMetadata, path: Path)
    fun read(path: Path): DependangerMetadata
}
----

**Встроенные реализации:** `JsonSerializationFormat` (`dependanger-metadata-json`)

'''

=== SchemaMigrator

Миграция `DependangerMetadata` между версиями схемы.
Миграция отделена от сериализации (Single Responsibility) и поддерживает цепочку: 1.0 → 1.1 → 2.0.

[source,kotlin]
----
interface SchemaMigrator {
    val fromVersion: String   // e.g. "1.0"
    val toVersion: String     // e.g. "1.1"

    fun migrate(metadata: DependangerMetadata): DependangerMetadata
}
----

Обнаружение — через ServiceLoader.
Несколько мигратров автоматически выстраиваются в цепочку по `fromVersion` → `toVersion`.

'''

=== EffectiveMetadataProcessor

Процессор в pipeline обработки.

[source,kotlin]
----
interface EffectiveMetadataProcessor {
    val id: String          // Уникальный идентификатор
    val phase: ProcessingPhase
    val order: Int          // По умолчанию = phase.order, можно переопределить
    val isOptional: Boolean // Можно ли отключить

    suspend fun process(metadata: EffectiveMetadata, context: ProcessingContext): EffectiveMetadata
}
----

==== ProcessingPhase

`ProcessingPhase` — open class с предопределёнными фазами в companion object.
Третьи стороны могут создавать собственные фазы:

[source,kotlin]
----
// Встроенные фазы доступны через companion object
ProcessingPhase.PROFILE           // order=5
ProcessingPhase.METADATA_CONVERSION // order=10
ProcessingPhase.VALIDATION        // order=65
// ... и другие

// Создание кастомной фазы
val MY_PHASE = ProcessingPhase("MY_ANALYSIS", 55, ExecutionMode.PARALLEL_IO)
----

**Встроенные процессоры:** см. xref:../02-features/F003-processing-pipeline.adoc[F003: Processing Pipeline]

'''

==== ProcessingCallback и ProcessingEvent

Pipeline уведомляет о ходе обработки через event-based callback.
Новые типы событий добавляются без нарушения существующих реализаций — `when` с exhaustive matching подскажет необработанные варианты.

[source,kotlin]
----
// События pipeline
sealed class ProcessingEvent {
    data class PhaseStarted(val phase: ProcessingPhase) : ProcessingEvent()
    data class PhaseCompleted(val phase: ProcessingPhase, val duration: Duration) : ProcessingEvent()
    data class PhaseError(val phase: ProcessingPhase, val error: Throwable) : ProcessingEvent()
    data class DiagnosticAdded(val message: DiagnosticMessage) : ProcessingEvent()
    data class Progress(val phase: ProcessingPhase, val current: Int, val total: Int) : ProcessingEvent()
}

// Единый callback — fun interface для удобства
fun interface ProcessingCallback {
    fun onEvent(event: ProcessingEvent)
}
----

**Пример использования:**

[source,kotlin]
----
val result = dependanger.process(distribution = "backend") { event ->
    when (event) {
        is ProcessingEvent.PhaseStarted -> println("▸ ${event.phase.name}")
        is ProcessingEvent.PhaseCompleted -> println("✓ ${event.phase.name} (${event.duration})")
        is ProcessingEvent.PhaseError -> println("✗ ${event.phase.name}: ${event.error.message}")
        is ProcessingEvent.DiagnosticAdded -> println("  ⚠ ${event.message.message}")
        is ProcessingEvent.Progress -> println("  ${event.current}/${event.total}")
    }
}
----

'''

=== ArtifactGenerator

Генератор выходных артефактов.

[source,kotlin]
----
interface ArtifactGenerator<T> {
    val generatorId: String  // "toml", "bom"

    fun generate(effective: EffectiveMetadata): T

----

**Встроенные реализации:** `TomlGenerator`, `BomGenerator`

'''

=== ReportProvider

Генератор отчётов.
Отделён от `ArtifactGenerator`, так как report — feature-модуль с собственной логикой (форматы, секции, маппинг данных из extensions).

[source,kotlin]
----
interface ReportProvider {
    val providerId: String  // "report"

    fun generate(
        effective: EffectiveMetadata,
        settings: ReportSettings,
        originalMetadata: DependangerMetadata?,
    ): GeneratedReport

    fun generateToFile(
        effective: EffectiveMetadata,
        settings: ReportSettings,
        originalMetadata: DependangerMetadata?,
    ): GeneratedReport
}

data class GeneratedReport(
    val format: ReportFormat,
    val content: String,
    val outputPath: Path?,
)
----

**Встроенные реализации:** `ReportGenerator` (`dependanger-report`)

'''

== SPI Discovery

Модули регистрируют реализации через `ServiceLoader`:

[source]
----
META-INF/services/
├── io.github.zenhelix.dependanger.core.spi.SerializationFormat
├── io.github.zenhelix.dependanger.core.spi.SchemaMigrator
├── io.github.zenhelix.dependanger.effective.spi.EffectiveMetadataProcessor
├── io.github.zenhelix.dependanger.effective.spi.ReportProvider
└── io.github.zenhelix.dependanger.generator.spi.ArtifactGenerator
----

=== Пример: создание кастомного процессора с ServiceLoader

**1. Реализация интерфейса:**

[source,kotlin]
----
// my-processor/src/main/kotlin/MyProcessor.kt
package com.example

// Кастомная фаза — выполняется после PLUGIN (50), перед USED_VERSIONS (60)
val MY_PHASE = ProcessingPhase("MY_ANALYSIS", 55, ExecutionMode.PARALLEL_IO)

// Типобезопасный ключ для хранения результатов в EffectiveMetadata.extensions
val MY_ANALYSIS_KEY = ExtensionKey<MyAnalysisResult>("my-analysis")

class MyCustomProcessor : EffectiveMetadataProcessor {
    override val id = "my-custom"
    override val phase = MY_PHASE
    override val isOptional = true

    override suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext
    ): EffectiveMetadata {
        val result = MyAnalysisResult(...)
        // Результат сохраняется в extensions (runtime-only, типобезопасно)
        return metadata.withExtension(MY_ANALYSIS_KEY, result)
    }
}

// Чтение результата (например, в генераторе или после pipeline)
val analysis = effectiveMetadata.getExtension(MY_ANALYSIS_KEY)
----

**2. Регистрация в ServiceLoader:**

[source]
----
// my-processor/src/main/resources/META-INF/services/
// io.github.zenhelix.dependanger.effective.spi.EffectiveMetadataProcessor

com.example.MyCustomProcessor
----

**3. Подключение модуля:**

[source,kotlin]
----
// build.gradle.kts
dependencies {
    runtimeOnly("com.example:my-processor:1.0.0")
}
----

После добавления модуля в classpath, система автоматически обнаружит и подключит `MyCustomProcessor` в pipeline.

== Программное использование

=== Базовое использование

[source,kotlin]
----
val result = Dependanger.fromDsl {
    versions {
        version("kotlin", "2.1.20")
    }
    libraries {
        library("kotlin-stdlib", "org.jetbrains.kotlin:kotlin-stdlib",
            version = versionRef("kotlin"))
    }
    bomImports {
        bom("spring-boot-bom", "org.springframework.boot:spring-boot-dependencies:3.4.0")
    }
    settings {
        toml { filename = "libs.versions.toml" }
    }
}.build()

// Генерация артефактов — extension functions на DependangerResult
// (определены в DependangerResultToml.kt и DependangerResultBom.kt)
val toml = result.toToml()
val bom = result.toBom()

// Запись в файлы
result.writeTomlTo(Path.of("gradle/libs.versions.toml"))
result.writeBomTo(Path.of("build/dependanger-bom.pom"))

// Generic API — для любого ArtifactGenerator (включая custom)
val customGenerator = MyCustomGenerator(config)
val output = result.generate(customGenerator)
result.writeTo(customGenerator, Path.of("output/custom-artifact"))
----

=== Builder Pattern

[source,kotlin]
----
val dependanger = Dependanger.fromDsl {
    // DSL конфигурация...
}
.preset(ProcessingPreset.STRICT)         // Включить все optional процессоры
.jdkVersion(17)                          // JDK версия для fallback/compatibility
.disableProcessor("license-check")       // Отключить конкретный процессор
.build()

val result = dependanger.process(distribution = "android")

// Проверка результата
if (result.diagnostics.hasErrors) {
    result.diagnostics.errors.forEach { println(it.message) }
}
----

val dependanger = Dependanger.fromJson(File("metadata.json").readText())
