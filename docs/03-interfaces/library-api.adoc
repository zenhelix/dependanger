[[overview]]
== Обзор

Library API предоставляет программный доступ к функциональности Dependanger для интеграции с другими инструментами и создания кастомных решений.

[cols="1,2",options="header"]
|===
| Характеристика | Значение

| Главный класс
| `io.github.zenhelix.dependanger.api.Dependanger`

| Минимальная версия JDK
| 17

| Целевая аудитория
| Разработчики инструментов
|===

[[design-principles]]
=== Принципы дизайна

API построен по принципу **тонкого фасада с extension-функциями**:

* `Dependanger` — минимальный фасад: конструирование pipeline и выполнение `process()` / `validate()`.
* `DependangerResult` — data class для результата с `effective` (nullable) и `diagnostics`.
* Генераторы (TOML, BOM) и extraction-методы вынесены в **extension-функции** на `DependangerResult`, что позволяет добавлять новые генераторы и extractors без изменения ядра.

Это избегает жёсткой связи фасада с конкретными генераторами и фичами.

[[core-concepts]]
== Основные концепции

[[concept-dependanger]]
=== Dependanger

Главный фасад для всех операций.
Создаётся через factory-функции `Dependanger(dslBlock) \{ ... \}` или `Dependanger(metadata) \{ ... \}`, либо через companion-методы `Dependanger.fromDsl()`, `Dependanger.fromMetadata()`, `Dependanger.fromJson()`.

[[concept-builder]]
=== DependangerBuilder

Builder для настройки pipeline: preset, environment, кастомные процессоры, отключение процессоров.
Возвращается из `Dependanger.fromDsl()` / `fromMetadata()` / `fromJson()`.
Не thread-safe — предназначен для однопоточной конфигурации.

[[concept-result]]
=== DependangerResult

Data class с двумя полями:

* `effective: EffectiveMetadata?` — `null` при ошибке обработки (например, `validate()` с невалидным metadata)
* `diagnostics: Diagnostics` — всегда присутствует (может быть `Diagnostics.EMPTY`)

Не sealed class — единый тип для успеха и ошибки.
Computed property `isSuccess` = `effective != null && !diagnostics.hasErrors`.

[[concept-thread-safety]]
=== Thread Safety

[cols="1,2",options="header"]
|===
| Компонент | Thread Safety

| `Dependanger`
| **Thread-safe.** Все поля — immutable. Операции `process()` / `validate()` stateless — состояние передаётся через параметры.

| `DependangerBuilder`
| **Не thread-safe.** Builder предназначен для однопоточной конфигурации. После `build()` результирующий `Dependanger` thread-safe.

| `DependangerResult`
| **Immutable.** Data class, безопасен для sharing между потоками.

| `Metadata`, `EffectiveMetadata`
| **Immutable.** Все data-классы неизменяемы.

| Processors
| **Stateless.** Кастомные процессоры должны быть stateless.
|===

[NOTE]
====
Для параллельной обработки нескольких дистрибуций вызывайте `process()` с разными `distribution` из coroutine scope.
Каждый вызов `process()` создаёт свой `ProcessingContext` — вызовы не делят mutable-состояние.
====

[[quick-start]]
== Быстрый старт

=== Базовое использование (factory-функции)

[source,kotlin]
----
import io.github.zenhelix.dependanger.api.Dependanger
import io.github.zenhelix.dependanger.api.toToml
import io.github.zenhelix.dependanger.api.writeTomlTo
import io.github.zenhelix.dependanger.api.writeBomTo
import io.github.zenhelix.dependanger.generators.bom.BomConfig
import java.nio.file.Path
import kotlinx.coroutines.runBlocking

fun main() = runBlocking {
    val dependanger = Dependanger(
        dslBlock = {
            versions {
                version("kotlin", "2.1.20")
            }
            libraries {
                library("kotlin-stdlib", "org.jetbrains.kotlin:kotlin-stdlib") {
                    version = versionRef("kotlin")
                }
            }
        },
        builderAction = {
            jdkVersion(17)
        }
    )

    // Обработка (default distribution)
    val result = dependanger.process()

    // Генерация TOML (extension function)
    val tomlContent: String = result.toToml()

    // Запись TOML в файл
    result.writeTomlTo(Path.of("gradle/libs.versions.toml"))

    // Генерация BOM (extension function, требует config)
    val bomConfig = BomConfig(
        groupId = "com.example",
        artifactId = "my-bom",
        version = "1.0.0",
        name = null,
        description = null,
        filename = BomConfig.DEFAULT_FILENAME,
        includeOptionalDependencies = false,
        prettyPrint = true,
        includeDeprecationComments = true,
    )
    result.writeBomTo(Path.of("build/bom.pom.xml"), bomConfig)
}
----

[[dependanger-class]]
== Dependanger

[[dependanger-creation]]
=== Создание

Три способа создания `Dependanger`:

==== Factory-функции (рекомендуется для inline-использования)

[source,kotlin]
----
// Из DSL-блока
val d1 = Dependanger(
    dslBlock = {
        versions { version("kotlin", "2.1.20") }
        libraries { /* ... */ }
    },
    builderAction = {
        preset(ProcessingPreset.DEFAULT)
        jdkVersion(17)
    }
)

// Из готового DependangerMetadata
val d2 = Dependanger(
    metadata = existingMetadata,
    builderAction = {
        disableProcessor("update-check")
    }
)
----

==== Companion-методы (рекомендуется для поэтапной конфигурации)

[source,kotlin]
----
// fromDsl — возвращает DependangerBuilder
val builder1 = Dependanger.fromDsl { /* DSL */ }

// fromMetadata — из готового объекта
val builder2 = Dependanger.fromMetadata(metadata)

// fromJson — парсит JSON, оборачивает в builder
val builder3 = Dependanger.fromJson(jsonString)

// Все три → DependangerBuilder → .build() → Dependanger
val dependanger = builder1.jdkVersion(17).build()
----

[NOTE]
====
`fromJson()` бросает `DependangerConfigurationException`, если JSON невалиден.
Builder без metadata и без DSL-блока бросит `DependangerConfigurationException` при `build()`.
====

[[dependanger-operations]]
=== Операции

Оба метода — `suspend`:

[source,kotlin]
----
// Полная обработка через pipeline
public suspend fun process(
    distribution: String? = null,     // null → settings.defaultDistribution
    callback: ProcessingCallback? = null,
): DependangerResult

// Валидация (MINIMAL preset + validation processor)
public suspend fun validate(): DependangerResult
----

[[dependanger-process-behavior]]
==== process()

1. Собирает pipeline: core-процессоры + discovered-процессоры (закэшированы при `build()`) + additionalProcessors.
2. Применяет preset и disabledProcessorIds.
3. Применяет `pipelineCustomizer` (если задан).
4. Создаёт `ProcessingContext` с `distribution`, `settings`, `environment`.
5. Запускает pipeline.
Процессоры группируются по `ExecutionMode`: sequential выполняются последовательно (fold), parallel — через `coroutineScope { async }`.
Каждый процессор возвращает immutable `ProcessorOutput` (metadata + executedId); shared mutable state отсутствует.
6. Возвращает `DependangerResult(effective, diagnostics)`.

**Обработка ошибок:**

* `CancellationException` — **перебрасывается** (через `coroutineContext.ensureActive()`), не оборачивается.
Structured concurrency сохраняется.
* `DependangerException` — пробрасывается as-is.
* `PipelineConfigurationException` — оборачивается в `DependangerConfigurationException`.
* Любой другой `Exception` — оборачивается в `DependangerProcessingException`.

[IMPORTANT]
====
`process()` **бросает исключение** при фатальной ошибке pipeline.
Если pipeline отработал, но с ошибками в diagnostics — вернётся `DependangerResult` с `effective != null` и `diagnostics.hasErrors == true`.
Проверяйте `result.isSuccess` перед генерацией, либо используйте `result.diagnostics.errors` для деталей.
====

[[dependanger-validate-behavior]]
==== validate()

1. Использует `ProcessingPreset.MINIMAL` с включённым `validation` процессором.
2. `CancellationException` — **перебрасывается** (через `coroutineContext.ensureActive()`), structured concurrency сохраняется.
3. При любом другом исключении — **не бросает**, а возвращает `DependangerResult(effective = null, diagnostics = error(...))`.

Это позволяет вызывающему коду безопасно проверить `result.isSuccess` без try/catch (кроме отмены корутины).

[NOTE]
====
`validate()` — lightweight quick-check.
Он **намеренно игнорирует** настройки builder'а (`preset`, `additionalProcessors`, `disabledProcessorIds`, `pipelineCustomizer`).
Всегда использует `ProcessingPreset.MINIMAL` + `validation` processor для быстрой и предсказуемой проверки.

Для полной валидации с учётом всех настроек — используйте `process()` и проверяйте `result.diagnostics`.
====

[[dependanger-builder]]
== DependangerBuilder

=== Конфигурация

[source,kotlin]
----
public class DependangerBuilder {
    // Все методы возвращают this (fluent API)

    // Processing preset (DEFAULT, MINIMAL, STRICT, DISTRIBUTION)
    public fun preset(preset: ProcessingPreset): DependangerBuilder

    // Environment
    public fun environment(environment: ProcessingEnvironment): DependangerBuilder
    public fun jdkVersion(version: Int): DependangerBuilder
    public fun kotlinVersion(version: String): DependangerBuilder
    public fun gradleVersion(version: String): DependangerBuilder

    // Custom processors
    public fun addProcessor(processor: EffectiveMetadataProcessor): DependangerBuilder
    public fun disableProcessor(processorId: String): DependangerBuilder

    // Pipeline customization
    public fun configureProcessing(block: PipelineBuilder.() -> Unit): DependangerBuilder

    // Build
    public fun build(): Dependanger
}
----

[cols="1,2,1",options="header"]
|===
| Метод | Описание | Default

| `preset()`
| Processing preset, определяющий набор активных процессоров
| `ProcessingPreset.DEFAULT`

| `environment()`
| Полная замена `ProcessingEnvironment` (JDK, Kotlin, Gradle версии + env variables)
| `ProcessingEnvironment.DEFAULT` (version fields = null, environmentVariables = emptyMap)

| `jdkVersion()`
| Устанавливает JDK-версию в environment (через `copy()`)
| `null`

| `kotlinVersion()`
| Устанавливает Kotlin-версию в environment (через `copy()`)
| `null`

| `gradleVersion()`
| Устанавливает Gradle-версию в environment (через `copy()`)
| `null`

| `addProcessor()`
| Добавляет кастомный процессор в pipeline
| —

| `disableProcessor()`
| Отключает процессор по его `id`
| —

| `configureProcessing()`
| Низкоуровневая настройка `PipelineBuilder`. Аккумулируется: несколько вызовов выполняются последовательно в порядке добавления.
| —
|===

[NOTE]
====
Все методы возвращают `this` — поддержка fluent-цепочек.

`jdkVersion()`, `kotlinVersion()`, `gradleVersion()` — convenience-обёртки над `environment.copy()`.
Вызов `environment()` после них перезапишет все ранее установленные значения:

[source,kotlin]
----
// НЕПРАВИЛЬНО: jdkVersion потеряна
builder.jdkVersion(17).environment(ProcessingEnvironment.DEFAULT)

// ПРАВИЛЬНО: environment первым, потом точечные
builder.environment(baseEnv).jdkVersion(17).kotlinVersion("2.1.20")
----

Используйте либо `environment()`, либо точечные методы — не смешивайте.
====

=== Пример: строгий режим с кастомными процессорами

[source,kotlin]
----
val dependanger = Dependanger.fromDsl {
    versions { version("kotlin", "2.1.20") }
    libraries { /* ... */ }
}
    .preset(ProcessingPreset.STRICT)
    .jdkVersion(17)
    .kotlinVersion("2.1.20")
    .addProcessor(MyAuditProcessor())
    .disableProcessor("license-check")  // отключить конкретный feature
    .configureProcessing {
        // низкоуровневая настройка PipelineBuilder
        enableOptional("validation")
    }
.build()
----

[[result-api]]
== DependangerResult

=== Структура

[source,kotlin]
----
public data class DependangerResult(
    val effective: EffectiveMetadata?,
    val diagnostics: Diagnostics,
) {
    public val isSuccess: Boolean
        get() = effective != null && !diagnostics.hasErrors

    // Safe — бросает если !isSuccess
    public fun <T> generate(generator: ArtifactGenerator<T>): T
    public fun <T> writeTo(generator: ArtifactGenerator<T>, path: Path): Path

    // Unsafe — бросает только если effective == null, игнорирует diagnostics.hasErrors
    public fun <T> generateUnsafe(generator: ArtifactGenerator<T>): T
    public fun <T> writeToUnsafe(generator: ArtifactGenerator<T>, path: Path): Path
}
----

=== Использование

[source,kotlin]
----
val result = dependanger.process(distribution = "production")

// Проверка успеха
if (result.isSuccess) {
    val toml = result.toToml()  // extension function (делегирует в generate())
    println(toml)
}

// Проверка диагностики
result.diagnostics.errors.forEach { error ->
    println("ERROR [${error.processorId}] ${error.code}: ${error.message}")
}
result.diagnostics.warnings.forEach { warning ->
    println("WARNING [${warning.processorId}] ${warning.code}: ${warning.message}")
}
----

=== generate() / writeTo() — safe vs unsafe

[source,kotlin]
----
// generate() — safe: бросает если !isSuccess (effective == null ИЛИ diagnostics.hasErrors)
val tomlContent = result.generate(TomlGenerator(TomlConfig.DEFAULT))

// generateUnsafe() — бросает только если effective == null
// Используйте для best-effort генерации при наличии ошибок в diagnostics
val bestEffort = result.generateUnsafe(TomlGenerator(TomlConfig.DEFAULT))

// writeTo() / writeToUnsafe() — аналогично, но с записью в файл
val outputPath = result.writeTo(TomlGenerator(TomlConfig.DEFAULT), Path.of("libs.versions.toml"))
----

[IMPORTANT]
====
* `generate()` / `writeTo()` — **бросают `DependangerProcessingException`** если `!isSuccess` (pipeline не отработал или есть ошибки в diagnostics).
Это safe-path по умолчанию.
* `generateUnsafe()` / `writeToUnsafe()` — бросают только если `effective == null`.
Позволяют генерировать best-effort результат несмотря на ошибки в diagnostics.
Используйте осознанно.
====

=== Edge Cases

[cols="1,2",options="header"]
|===
| Сценарий | Поведение

| `effective != null`, `diagnostics.hasErrors == false`
| `isSuccess = true`. `generate()` и `generateUnsafe()` работают одинаково.

| `effective != null`, `diagnostics.hasErrors == true`
| `isSuccess = false`. `generate()` **бросает**, `generateUnsafe()` **работает** (best-effort результат).

| `effective == null`, `diagnostics.hasErrors == true`
| `isSuccess = false`. И `generate()`, и `generateUnsafe()` **бросают**.

| `effective == null`, `diagnostics.hasErrors == false`
| Теоретически невозможно при текущей реализации, но оба метода **бросают** по null-check.

| Вызов `generate()` при `effective == null`
| Бросает `DependangerProcessingException("Cannot generate: processing failed")`.
|===

[[generator-extensions]]
== Extension-функции для генераторов

Convenience-обёртки над `generate()` / `writeTo()` — вынесены в отдельные файлы, не создают связи фасада с генераторами.

=== TOML

[source,kotlin]
----
import io.github.zenhelix.dependanger.api.toToml
import io.github.zenhelix.dependanger.api.writeTomlTo

// Генерация строки (default config)
val toml: String = result.toToml()

// С кастомным конфигом
val toml2 = result.toToml(TomlConfig(
    filename = "libs.versions.toml",
    includeComments = true,
    sortSections = true,
    useInlineVersions = false,
    includeDeprecationComments = true,
))

// Запись в файл
val path: Path = result.writeTomlTo(Path.of("gradle/libs.versions.toml"))
val path2 = result.writeTomlTo(Path.of("output"), TomlConfig.DEFAULT)
----

=== BOM

[source,kotlin]
----
import io.github.zenhelix.dependanger.api.toBom
import io.github.zenhelix.dependanger.api.writeBomTo

val config = BomConfig(
    groupId = "com.example",
    artifactId = "my-bom",
    version = "1.0.0",
    name = "My BOM",
    description = "Centralized dependency versions",
    filename = BomConfig.DEFAULT_FILENAME,
    includeOptionalDependencies = false,
    prettyPrint = true,
    includeDeprecationComments = true,
)

// Генерация строки
val bom: String = result.toBom(config)

// Запись в файл
result.writeBomTo(Path.of("build/bom.pom.xml"), config)
----

[NOTE]
====
`toToml()` имеет default-параметр `TomlConfig.DEFAULT`.
`toBom()` **не имеет** default — `BomConfig` требует обязательные `groupId`, `artifactId`, `version`.
====

=== Универсальный generate()

Для кастомных генераторов (реализующих `ArtifactGenerator<T>`):

[source,kotlin]
----
// Любой ArtifactGenerator<T>
val artifact = result.generate(myCustomGenerator)

[[extraction-api]]
== Extraction API

Extension-свойства на `DependangerResult` для извлечения feature-специфичных данных из `EffectiveMetadata.extensions`.
Определены в `DependangerResultExtensions.kt`.

[source,kotlin]
----
import io.github.zenhelix.dependanger.api.updates
import io.github.zenhelix.dependanger.api.vulnerabilities
import io.github.zenhelix.dependanger.api.licenseViolations
import io.github.zenhelix.dependanger.api.transitives
import io.github.zenhelix.dependanger.api.compatibilityIssues
import io.github.zenhelix.dependanger.api.versionConflicts

val result = dependanger.process(distribution = "production")

// Доступные обновления (из UpdateCheckProcessor)
val updates: List<UpdateAvailableInfo> = result.updates
updates.forEach { println("${it.alias}: ${it.currentVersion} -> ${it.latestVersion}") }

// Уязвимости (из SecurityCheckProcessor)
val vulnerabilities: List<VulnerabilityInfo> = result.vulnerabilities
vulnerabilities.forEach { println("${it.cveId} (CVSS: ${it.cvssScore}): ${it.summary}") }

// Нарушения лицензий (из LicenseCheckProcessor)
val licenseIssues: List<LicenseViolation> = result.licenseViolations

// Транзитивные зависимости (из TransitiveResolverProcessor)
val trees: List<TransitiveTree> = result.transitives

// Проблемы совместимости (из CompatibilityCheckProcessor)
val issues: List<CompatibilityIssue> = result.compatibilityIssues

// Конфликты версий (из TransitiveResolverProcessor)
val conflicts: List<VersionConflict> = result.versionConflicts
----

[cols="1,2,1",options="header"]
|===
| Extension-свойство | Описание | Возвращаемый тип

| `updates`
| Доступные обновления версий
| `List<UpdateAvailableInfo>`

| `vulnerabilities`
| Найденные уязвимости (CVE)
| `List<VulnerabilityInfo>`

| `licenseViolations`
| Нарушения лицензионных политик
| `List<LicenseViolation>`

| `transitives`
| Деревья транзитивных зависимостей
| `List<TransitiveTree>`

| `compatibilityIssues`
| Проблемы совместимости
| `List<CompatibilityIssue>`

| `versionConflicts`
| Конфликты версий транзитивных зависимостей
| `List<VersionConflict>`
|===

[NOTE]
====
Все extraction-свойства безопасны: при `effective == null` или отсутствии данных возвращают пустой список.
Данные доступны только если соответствующий процессор был включён (через preset или `addProcessor()`).
Если процессор отключён или не в classpath — свойство вернёт пустой список без ошибок.

**Как отличить "процессор не нашёл данных" от "процессор не запускался":**
используйте `ProcessingInfo.processorIds` — список ID процессоров, которые фактически выполнились:

[source,kotlin]
----
val processorRan = result.effective
    ?.processingInfo
    ?.processorIds
    ?.contains("update-check") == true

if (!processorRan) {
    println("Update check was not executed (disabled or not in classpath)")
} else if (result.updates.isEmpty()) {
    println("All libraries are up to date")
}
----
====

[[distributions]]
== Работа с дистрибуциями

=== Одна дистрибуция

[source,kotlin]
----
// Default distribution (из settings.defaultDistribution или null → все библиотеки)
val result1 = dependanger.process()

// Конкретная дистрибуция
val result2 = dependanger.process(distribution = "android")
----

=== Несколько дистрибуций

[source,kotlin]
----
// Последовательная обработка
val distributions = listOf("android", "ios", "backend")
val results = distributions.map { dist ->
    dist to dependanger.process(distribution = dist)
}

// Параллельная обработка (Dependanger thread-safe)
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

suspend fun processAll(
    dependanger: Dependanger,
    distributions: List<String>
): Map<String, DependangerResult> = coroutineScope {
    distributions.associateWith { dist ->
        async { dependanger.process(distribution = dist) }
    }.mapValues { (_, deferred) -> deferred.await() }
}
----

[IMPORTANT]
====
`distribution = null` не означает "все дистрибуции".
Это означает "использовать `settings.defaultDistribution`".
Если `defaultDistribution` не задан — pipeline работает без фильтрации (все библиотеки включены).
Для обработки всех дистрибуций — итерируйте явно.

Несуществующая дистрибуция (не объявленная в `distributions {}`) обрабатывается `ProfileProcessor` в pipeline — добавляет ошибку в diagnostics.
====

[[processing-callback]]
== Processing Callback

Для отслеживания прогресса pipeline:

[source,kotlin]
----
import io.github.zenhelix.dependanger.effective.pipeline.ProcessingCallback
import io.github.zenhelix.dependanger.effective.pipeline.ProcessingEvent

val result = dependanger.process(
    distribution = "production",
    callback = ProcessingCallback { event ->
        when (event) {
            is ProcessingEvent.PhaseStarted ->
                println("Starting: ${event.phase}")
            is ProcessingEvent.PhaseCompleted ->
                println("Completed: ${event.phase} in ${event.duration}")
            is ProcessingEvent.PhaseError ->
                println("Error in ${event.phase}: ${event.error.message}")
            is ProcessingEvent.DiagnosticAdded ->
                println("Diagnostic: [${event.message.severity}] ${event.message.message}")
            is ProcessingEvent.Progress ->
                println("Progress [${event.phase}]: ${event.current}/${event.total}")
        }
    }
)
----

`ProcessingCallback` — `fun interface` с единственным методом `onEvent(event: ProcessingEvent)`.
Можно передать лямбду.

[NOTE]
====
Callback — наблюдатель.
Исключения из callback'а **не влияют на pipeline**: они перехватываются, логируются через `kotlin-logging` на уровне `WARN` и подавляются.
Callback не должен бросать исключения — но если бросит, pipeline продолжит работу корректно.
====

[[custom-processors]]
== Custom Processors

=== Создание

[source,kotlin]
----
import io.github.zenhelix.dependanger.effective.pipeline.EffectiveMetadataProcessor
import io.github.zenhelix.dependanger.effective.pipeline.ProcessingPhase
import io.github.zenhelix.dependanger.effective.model.EffectiveMetadata
import io.github.zenhelix.dependanger.effective.pipeline.ProcessingContext

class MyAuditProcessor : EffectiveMetadataProcessor {
    override val id: String = "my-audit"
    override val phase: ProcessingPhase = ProcessingPhase.COMPATIBILITY_ANALYSIS
    override val order: Int = 115  // после стандартного compatibility (110)
    override val isOptional: Boolean = true
    override val description: String = "Custom audit checks"

    override fun supports(context: ProcessingContext): Boolean = true

    override suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext,
    ): EffectiveMetadata {
        // Возвращаем НОВЫЙ объект — immutability contract
        return metadata.copy(
            diagnostics = metadata.diagnostics + myDiagnostics,
        )
    }
}
----

=== Регистрация

[source,kotlin]
----
// Через builder
val dependanger = Dependanger.fromDsl { /* ... */ }
    .addProcessor(MyAuditProcessor())
    .build()

// Через ServiceLoader (META-INF/services)
// Файл: META-INF/services/io.github.zenhelix.dependanger.effective.pipeline.EffectiveMetadataProcessor
// Содержимое: com.example.MyAuditProcessor
----

[[error-handling]]
== Обработка ошибок

=== Иерархия исключений

[source,kotlin]
----
DependangerException(message, cause)           // базовый
├── DependangerConfigurationException          // ошибки конфигурации (build, JSON parse, unknown distribution)
├── DependangerValidationException(errors)     // ошибки валидации (список строк)
├── DependangerProcessingException(phase?)     // ошибки pipeline (phase: String?, null для generic errors)
└── DependangerGenerationException(generatorId) // ошибки генерации
----

=== Когда что бросается

[cols="1,2",options="header"]
|===
| Ситуация | Поведение

| Невалидный JSON в `fromJson()`
| `DependangerConfigurationException`

| Исключение в DSL-блоке при `build()`
| `DependangerConfigurationException` (оборачивает оригинальное исключение в `cause`)

| Builder без metadata и DSL
| `DependangerConfigurationException` при `build()`

| Pipeline configuration error
| `DependangerConfigurationException` (обёрнутый `PipelineConfigurationException`)

| Ошибка в процессоре при `process()`
| `DependangerProcessingException` с `phase`

| `generate()` при `!isSuccess`
| `DependangerProcessingException` (safe-check перед генерацией)

| `generateUnsafe()` при `effective == null`
| `DependangerProcessingException`

| Ошибка в генераторе (`generate()` / `generateUnsafe()`)
| `DependangerGenerationException` с `generatorId` (оборачивает исключение генератора)

| Ошибка записи файла (`writeTo()` / `writeToUnsafe()`)
| `DependangerGenerationException` с `generatorId` (оборачивает IO/другие исключения)

| Ошибка при `validate()`
| **Не бросает** — возвращает `DependangerResult(effective = null, diagnostics = error(...))`

| Отмена корутины (`CancellationException`)
| **Перебрасывается** в обоих методах (`process()` и `validate()`) через `coroutineContext.ensureActive()`
|===

[IMPORTANT]
====
**Асимметрия `process()` vs `validate()`:**

* `process()` бросает исключения при фатальных ошибках.
* `validate()` **не бросает** бизнес-ошибки — возвращает через `diagnostics`.
* Оба метода **перебрасывают `CancellationException`** — structured concurrency сохраняется.
====

=== Паттерн обработки

[source,kotlin]
----
// process() — нужен try/catch для фатальных ошибок
try {
    val result = dependanger.process(distribution = "production")

    if (result.isSuccess) {
        result.writeTomlTo(Path.of("libs.versions.toml"))
    } else {
        result.diagnostics.errors.forEach { e ->
            System.err.println("[${e.code}] ${e.message}")
        }
    }
} catch (e: DependangerConfigurationException) {
    System.err.println("Configuration error: ${e.message}")
} catch (e: DependangerProcessingException) {
    System.err.println("Processing failed at phase ${e.phase}: ${e.message}")
}

// validate() — try/catch не нужен
val validationResult = dependanger.validate()
if (!validationResult.isSuccess) {
    validationResult.diagnostics.errors.forEach { e ->
        System.err.println("Validation: [${e.code}] ${e.message}")
    }
}
----

[[diagnostics]]
== Диагностика

`Diagnostics` — immutable контейнер для сообщений трёх уровней:

[source,kotlin]
----
public data class Diagnostics(
    val errors: List<DiagnosticMessage>,
    val warnings: List<DiagnosticMessage>,
    val infos: List<DiagnosticMessage>,
) {
    val hasErrors: Boolean     // errors.isNotEmpty()
    val isValid: Boolean       // errors.isEmpty()
    operator fun plus(other: Diagnostics): Diagnostics
}

public data class DiagnosticMessage(
    val code: String,          // e.g. "UNRESOLVED_VERSION"
    val message: String,       // человекочитаемое описание
    val severity: Severity,    // ERROR, WARNING, INFO
    val processorId: String?,  // какой процессор выдал (null для system-level)
    val context: Map<String, String>,  // доп. контекст (alias, version и т.д.)
)
----

Стандартные коды диагностики определены в xref:../02-features/F003-processing-pipeline.adoc[F003 Processing Pipeline].

[[full-example]]
== Полный пример

[source,kotlin]
----
import io.github.zenhelix.dependanger.api.*
import io.github.zenhelix.dependanger.core.model.ProcessingPreset
import io.github.zenhelix.dependanger.generators.bom.BomConfig
import kotlinx.coroutines.runBlocking
import java.nio.file.Path

fun main() = runBlocking {
    // 1. Создание Dependanger через DSL
    val dependanger = Dependanger(
        dslBlock = {
            versions {
                version("kotlin", "2.1.20")
                version("spring-boot", "3.4.1")
            }

            libraries {
                library("kotlin-stdlib", "org.jetbrains.kotlin:kotlin-stdlib") {
                    version = versionRef("kotlin")
                    tags("core", "kotlin")
                }
                library("spring-boot-starter", "org.springframework.boot:spring-boot-starter") {
                    version = versionRef("spring-boot")
                    tags("spring", "core")
                }
            }

            distributions {
                distribution("core") {
                    spec {
                        filter {
                            byTags {
                                include { anyOf("core") }
                            }
                        }
                    }
                }
            }

            compatibility {
                jdkRequirement("spring-boot-3x") {
                    matches = "spring-*"
                    minJdk = 17
                }
            }
        },
        builderAction = {
            preset(ProcessingPreset.STRICT)
            jdkVersion(17)
            kotlinVersion("2.1.20")
        }
    )

    // 2. Валидация (безопасно, без исключений)
    val validation = dependanger.validate()
    if (!validation.isSuccess) {
        validation.diagnostics.errors.forEach { e ->
            println("Validation error: [${e.code}] ${e.message}")
        }
        return@runBlocking
    }

    // 3. Обработка с callback
    val result = dependanger.process(
        distribution = "core",
        callback = { event ->
            println("Event: $event")
        }
    )

    if (!result.isSuccess) {
        result.diagnostics.errors.forEach { e ->
            println("Error: [${e.code}] ${e.message}")
        }
        return@runBlocking
    }

    // 4. Предупреждения (не блокируют, но стоит показать)
    result.diagnostics.warnings.forEach { w ->
        println("Warning: [${w.code}] ${w.message}")
    }

    // 5. Генерация TOML
    result.writeTomlTo(Path.of("build/libs.versions.toml"))

    // 6. Генерация BOM
    result.writeBomTo(
        Path.of("build/bom.pom.xml"),
        BomConfig(
            groupId = "com.example",
            artifactId = "my-bom",
            version = "1.0.0",
            name = "My BOM",
            description = null,
            filename = BomConfig.DEFAULT_FILENAME,
            includeOptionalDependencies = false,
            prettyPrint = true,
            includeDeprecationComments = true,
        )
    )

    // 7. Анализ feature-данных (через extraction extensions)
    val updates = result.updates
    if (updates.isNotEmpty()) {
        println("\nAvailable updates:")
        updates.forEach { println("  ${it.alias}: ${it.currentVersion} -> ${it.latestVersion}") }
    }

    val vulns = result.vulnerabilities
    if (vulns.isNotEmpty()) {
        println("\nVulnerabilities found:")
        vulns.forEach { println("  ${it.cveId} [${it.severity}]: ${it.summary}") }
    }

    println("\nGeneration completed!")
}
----
