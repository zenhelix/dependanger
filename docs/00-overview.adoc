= Обзор проекта: Dependanger
:toc: left
:toclevels: 3
:sectnums:
:icons: font

[abstract]
--
Dependanger — инструмент для управления зависимостями Kotlin/Java проектов.
Позволяет создавать централизованные каталоги библиотек с метаданными, формировать готовые наборы (бандлы) для разных ролей разработчиков, и генерировать стандартные артефакты (Gradle Version Catalog, Maven BOM).
--

== Цели и видение

=== Проблема

Разработчики Kotlin/Java сталкиваются с комплексом проблем при управлении зависимостями:

**1. Разрозненность**

* Каждый проект самостоятельно определяет версии библиотек
* Нет единого источника истины для команды/организации
* Дублирование усилий по исследованию и выбору библиотек

**2. Сложность выбора**

* Новым разработчикам трудно понять, какие библиотеки использовать для конкретной задачи
* Нет контекста: почему выбрана именно эта библиотека, какие альтернативы рассматривались
* Отсутствует информация о совместимости между библиотеками

**3. Устаревание и совместимость**

* Трудно отслеживать выход новых версий
* Сложно понять, какие версии библиотек совместимы друг с другом
* Обновление одной библиотеки может сломать совместимость с другими

=== Решение

Dependanger предлагает:

**Централизованный каталог с метаданными**

* Единое место для определения всех зависимостей через Kotlin DSL
* Описания, теги, информация о совместимости для каждой библиотеки
* Версии с возможностью условного выбора (JDK версия, Kotlin версия, distribution profile)

**Ролевые бандлы и distribution profiles**

* **Bundles** — именованные группы библиотек для переиспользования (например: `kotlin-core`, `spring-web`)
* **Distributions** — профили генерации с фильтрацией (например: `android`, `backend`, `minimal`)
* Проверенные комбинации версий через compatibility rules

**Импорт внешних BOM (Platforms)**

* Импорт версий из внешних Maven BOM/Platform (Spring Boot BOM, Kotlin BOM)
* Автоматическое наследование версий без дублирования
* Приоритет: явно указанные версии > импортированные из BOM

**Два интерфейса с разными сценариями использования**

* **Gradle Plugin** — Kotlin DSL в `build.gradle.kts`, интеграция с Gradle lifecycle, инкрементальность
* **CLI** — standalone команды для CI/CD, работа с metadata.json напрямую, скриптовая автоматизация

**Многоэтапный pipeline обработки**

[source]
----
1. Вход: Kotlin DSL (Gradle Plugin) или CLI команды
        ↓
2. DependangerMetadata → metadata.json
        ↓
3. Processing Pipeline:
   - Version resolution (из aliases, platforms, Maven)
   - Filtering (по distribution profile)
   - Validation (правила совместимости)
   - Optional: Update Check, Compatibility Analysis
        ↓
4. EffectiveMetadata → effective-{distribution}.json
        ↓
5. Генерация: TOML (Version Catalog) + BOM (Maven POM)
        ↓
6. Публикация артефактов для использования в других проектах
----

См. xref:02-features/F003-processing-pipeline.adoc[F003: Processing Pipeline] для деталей.

**Автоматизация**

* Генерация стандартных артефактов (TOML, BOM)
* Проверка доступных обновлений
* Анализ совместимости

**Расширяемость через SPI**

* `SerializationFormat<T>` — абстракция формата хранения (JSON, YAML, etc.)
* `EffectiveMetadataProcessor` — процессоры pipeline с приоритетами
* `ArtifactGenerator<T>` — генераторы выходных артефактов

== Глоссарий

[cols="1,3",options="header"]
|===
| Термин | Определение

| DependangerDsl
| Точка входа Kotlin DSL для конфигурации. Содержит блоки `versions`, `libraries`, `plugins`, `bundles`, `platforms`, `distributions`, `compatibility`. Существует в памяти во время выполнения DSL.

| DependangerMetadata
| Сериализуемое представление конфигурации. Промежуточный формат для хранения и передачи (по умолчанию JSON). Создаётся через `SerializationFormat` SPI. Файл: `metadata.json`.

| EffectiveMetadata
| Результат обработки pipeline. Все версии разрешены (с указанием источника), фильтры применены, валидация выполнена. Готов для генерации артефактов. Файл: `effective-{distribution}.json`.

| Library
| Зависимость с Maven координатами (`group:artifact`), версией и метаданными (описание, теги, KMP таргеты). Определяется через DSL или импортируется из BOM.

| Plugin
| Gradle plugin — определяется через alias, plugin ID и версию. Публикуется в Version Catalog секции `[plugins]`.

| Bundle
| Именованная группа библиотек для переиспользования. Может включать другие бандлы. Публикуется в Version Catalog секции `[bundles]`. Пример: `kotlin-core`, `spring-web`.

| Platform
| Внешний Maven BOM/Platform для импорта версий. Например: `org.springframework.boot:spring-boot-dependencies`. Версии из Platform имеют приоритет ниже явно указанных.

| Distribution
| Профиль генерации артефактов с фильтрацией библиотек. Определяет какие библиотеки/бандлы включить, какие исключить, какие версии переопределить. Пример: `android`, `backend`, `minimal`.

| Version Alias
| Именованная версия для переиспользования несколькими библиотеками. Поддерживает fallback условия. Пример: `kotlin = "2.1.0"`.

| Version Fallback
| Условная альтернативная версия. Применяется при выполнении условия (JDK версия, Kotlin версия, distribution). Пример: `fallback("1.9.24") { jdkBelow(11) }`.

| Version Source
| Источник версии библиотеки. Приоритет: `DECLARED` (явно в DSL, включая inline в coordinates) > `BOM_IMPORT` (из Platform).

| Processor
| Компонент pipeline обработки с приоритетом (order). Выполняет трансформацию данных на определённом этапе. Реализует `EffectiveMetadataProcessor` SPI.

| Compatibility Rule
| Правило проверки совместимости между библиотеками или с окружением (JDK, Kotlin). Severity: ERROR, WARNING, INFO.

| Version Catalog (TOML)
| Стандартный формат Gradle 7.0+ для объявления зависимостей. Выходной артефакт Dependanger. Секции: `[versions]`, `[libraries]`, `[bundles]`, `[plugins]`.

| BOM (Bill of Materials)
| Maven POM-артефакт с `<dependencyManagement>` для централизованного управления версиями. Выходной формат для Maven-экосистемы и Gradle `platform()`.
|===

=== Системы-потребители

* **Gradle Build System** — использует сгенерированный Version Catalog (TOML)
* **Maven** — использует сгенерированный BOM
* **CI/CD Pipelines** — запускает проверки обновлений и совместимости
* **IDE (IntelliJ IDEA)** — автодополнение из Version Catalog

== Границы системы

=== В области (In Scope)

**Определение зависимостей (Kotlin DSL):**

* [x] Libraries — библиотеки с Maven координатами, тегами, метаданными
* [x] Plugins — Gradle plugins с версиями
* [x] Versions — именованные версии с fallback условиями
* [x] Bundles — группы библиотек для переиспользования
* [x] Platforms — импорт версий из внешних Maven BOM
* [x] Distributions — профили генерации с фильтрацией
* [x] Compatibility rules — правила проверки совместимости

**Генерация артефактов:**

* [x] Gradle Version Catalog (TOML) — основной формат для Gradle
* [x] Maven BOM (XML) — для Maven и Gradle `platform()`

**Автоматизация и проверки:**

* [x] Update Check — проверка доступных обновлений в Maven репозиториях
* [x] Compatibility Analysis — анализ совместимости на основе правил
* [x] Report Generation — единый отчёт в JSON формате

**Интерфейсы:**

* [x] Gradle Plugin — Kotlin DSL, интеграция с Gradle lifecycle
* [x] CLI — standalone команды для CI/CD и скриптов

**Расширяемость (SPI):**

* [x] `SerializationFormat<T>` — форматы сериализации (JSON, YAML)
* [x] `EffectiveMetadataProcessor` — процессоры pipeline
* [x] `ArtifactGenerator<T>` — генераторы артефактов

=== Расширяемость (SPI)

[cols="1,2,2",options="header"]
|===
| SPI Interface | Назначение | Встроенные реализации

| `SerializationFormat<T>`
| Форматы сериализации DependangerMetadata
| JSON (`dependanger-metadata-json`)

| `EffectiveMetadataProcessor`
| Этапы обработки в pipeline
| BomImport, UpdateCheck, CompatibilityAnalysis

| `ArtifactGenerator<T>`
| Генераторы выходных артефактов
| TOML (`dependanger-generator-toml`), BOM (`dependanger-generator-bom`)
|===

**Примеры кастомных расширений:**

* `YamlSerializationFormat` — YAML вместо JSON для metadata
* `SecurityScanProcessor` — проверка CVE уязвимостей
* `BazelBuildGenerator` — генерация Bazel BUILD файлов

=== Вне области (Out of Scope)

* [ ] Автоматическое обновление версий (только отчет, решение за пользователем)
* [ ] Хостинг/публикация артефактов (Dependanger генерирует TOML/BOM, но не управляет их публикацией — это задача CI/CD или пользователя)
* [ ] Графический интерфейс (только CLI и Gradle Plugin)
* [ ] Поддержка не-JVM языков (только Kotlin/Java экосистема)
* [ ] Разрешение конфликтов транзитивных зависимостей (это задача Gradle/Maven)

=== Ограничения

[cols="1,2,1",options="header"]
|===
| ID | Ограничение | Причина

| C-001
| JDK 17+ обязателен
| Использование современных API, соответствие Gradle 8.x требованиям

| C-002
| Только Maven-совместимые репозитории
| Стандарт в Kotlin/Java экосистеме

| C-003
| TOML формат по спецификации Gradle 7.0+
| Совместимость со стандартным Version Catalog
|===

== Масштабирование

Dependanger проектируется для масштабирования от личного использования до организации:

[cols="1,2,2",options="header"]
|===
| Уровень | Описание | Пример использования

| Личный
| Один разработчик, свои проекты
| Персональный реестр любимых библиотек

| Команда
| Несколько разработчиков, общие проекты
| Стандартизированный реестр команды в Git

| Организация
| Много команд, много проектов
| Публикуемый BOM, переиспользуемые бандлы
|===
