= ADR-004: SPI-based расширяемость системы
:icons: font

== Метаданные

[cols="1,2"]
|===
| Статус | 🟢 Принято
| Дата | 2026-01-15
| Авторы | Dependanger Team
| Затронутые компоненты | Все модули
|===

== Контекст

Dependanger должен поддерживать расширение функциональности без модификации core кода:

* Новые форматы сериализации (YAML, TOML input)
* Новые генераторы артефактов (Gradle Catalog, Bazel)
* Кастомные процессоры pipeline
* Альтернативные стратегии фильтрации

Необходим механизм расширения, который:

1. Не требует форка проекта
2. Позволяет добавлять функциональность через зависимости
3. Имеет стабильные контракты

== Решение

Выбран **Service Provider Interface (SPI)** подход с явными интерфейсами расширения.

[source]
----
┌─────────────────────────────────────────────────────────┐
│                    dependanger-api                       │
│  ┌─────────────────────────────────────────────────────┐│
│  │   Dependanger (Facade)                              ││
│  │     ↓                                               ││
│  │   DependangerBuilder                                ││
│  │     ↓ uses                                          ││
│  │   SPI Interfaces                                    ││
│  └─────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────┘
                          ↑ implements
    ┌─────────────┬───────────────┬───────────────┐
    │             │               │               │
┌───┴───┐   ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
│ TOML  │   │   BOM     │   │  Updates  │   │  Custom   │
│ Gen   │   │   Gen     │   │  Check    │   │  Module   │
└───────┘   └───────────┘   └───────────┘   └───────────┘
----

== Обоснование

=== Рассмотренные альтернативы

**1. Plugin system с runtime discovery:**

[source,kotlin]
----
ServiceLoader.load(DependangerPlugin::class.java)
----

*Недостаток:* Сложность отладки, неявные зависимости, проблемы с classpath.

**2. Callback/Hook based:**

[source,kotlin]
----
dependanger.onLibraryProcessed { library -> ... }
----

*Недостаток:* Ограниченные возможности, сложность управления порядком.

**3. Inheritance based:**

[source,kotlin]
----
class MyDependanger : Dependanger() { override fun ... }
----

*Недостаток:* Fragile base class problem, сложность версионирования.

=== Преимущества выбранного подхода

[cols="1,2"]
|===
| Критерий | Обоснование

| Explicit contracts
| Интерфейсы явно определяют точки расширения

| Compile-time safety
| Ошибки обнаруживаются при компиляции

| Testability
| Легко создавать моки и тестовые реализации

| Versioning
| Интерфейсы могут эволюционировать с deprecation

| No magic
| Все зависимости явные, без runtime discovery
|===

== Последствия

=== Положительные

* Чёткие контракты для расширения
* Простое тестирование с mock реализациями
* Возможность создания сторонних расширений
* Стабильный API для публичного использования

=== Отрицательные

* Больше интерфейсов для поддержки
* Изменения в интерфейсах требуют careful versioning
* Нет автоматического discovery (явная регистрация)

=== Нейтральные

* Default реализации предоставляются в core модулях
* Builder pattern для конфигурации расширений

== Реализация

=== Основные SPI интерфейсы

==== SerializationFormat<T>

Формат сериализации для метаданных.

[source,kotlin]
----
interface SerializationFormat<T> {
    val formatName: String
    val fileExtension: String
    
    fun serialize(value: T): String
    fun deserialize(content: String): T
}
----

Реализации:

* `JsonMetadataFormat` — JSON для DependangerMetadata
* `JsonEffectiveMetadataFormat` — JSON для EffectiveMetadata

==== EffectiveMetadataProcessor

Процессор в pipeline обработки.

[source,kotlin]
----
interface EffectiveMetadataProcessor {
    val name: String
    val order: Int
    
    suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext
    ): EffectiveMetadata
}
----

Реализации:

* `BomImportProcessor` — импорт версий из BOM
* `UpdateCheckProcessor` — проверка обновлений
* `CompatibilityCheckProcessor` — анализ совместимости

==== ArtifactGenerator

Генератор выходных артефактов.

[source,kotlin]
----
interface ArtifactGenerator {
    val formatName: String
    
    fun generate(metadata: EffectiveMetadata): String
    fun generate(metadata: EffectiveMetadata, config: GeneratorConfig): String
}
----

Реализации:

* `EffectiveTomlGenerator` — TOML Version Catalog
* `EffectiveBomGenerator` — Maven BOM

==== LibraryFilter

Стратегия фильтрации библиотек.

[source,kotlin]
----
interface LibraryFilter {
    fun filter(
        libraries: List<Library>,
        filterSpec: FilterSpec?
    ): List<Library>
}
----

Реализации:

* `DefaultLibraryFilter` — tag-based фильтрация
* `RegexLibraryFilter` — regex-based фильтрация

=== Регистрация расширений

[source,kotlin]
----
// Через Builder API
val dependanger = Dependanger.builder()
    .withProcessor(MyCustomProcessor())
    .withLibraryFilter(MyLibraryFilter())
    .withMetadataFormat(YamlMetadataFormat())
    .build()
----

=== Пример кастомного процессора

[source,kotlin]
----
class SecurityScanProcessor(
    private val vulnerabilityDb: VulnerabilityDatabase
) : EffectiveMetadataProcessor {
    
    override val name = "security-scan"
    override val order = 85  // После compatibility analysis
    
    override suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext
    ): EffectiveMetadata {
        metadata.libraries.forEach { library ->
            val vulns = vulnerabilityDb.findVulnerabilities(
                library.group, library.artifact, library.version
            )
            if (vulns.isNotEmpty()) {
                context.addWarning(
                    name,
                    "Library ${library.alias} has ${vulns.size} known vulnerabilities"
                )
            }
        }
        return metadata
    }
}

// Использование
val dependanger = Dependanger.builder()
    .withProcessor(SecurityScanProcessor(MyVulnDb()))
    .build()
----

== Версионирование API

=== Stability markers

[source,kotlin]
----
@RequiresOptIn(
    message = "This API is experimental and may change",
    level = RequiresOptIn.Level.WARNING
)
annotation class ExperimentalDependangerApi

@ExperimentalDependangerApi
interface NewFeatureProcessor : EffectiveMetadataProcessor
----

=== Deprecation policy

1. Deprecated API сохраняется минимум 2 minor версии
2. Deprecated API помечается `@Deprecated` с указанием замены
3. Удаление только в major версиях
