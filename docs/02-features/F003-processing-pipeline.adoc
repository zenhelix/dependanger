== Описание

Processing Pipeline выполняет многоэтапную обработку метаданных:

* Резолвинг версий (из alias, BomImport, Maven)
* Применение фильтров (Distribution)
* Валидация
* Опциональные проверки (updates, compatibility)

Результат — `EffectiveMetadata`, готовый для генерации артефактов.

[[effective-metadata]]
== EffectiveMetadata

=== Отличия от DependangerMetadata

[cols="1,2,2",options="header"]
|===
| Аспект | DependangerMetadata | EffectiveMetadata

| Версии
| `versionRef("kotlin")` — ссылка
| `resolvedVersion = "2.1.20"`, `versionSource = DECLARED`

| Библиотеки
| Все определённые библиотеки
| Только отфильтрованные по distribution

| Бандлы
| Ссылки на библиотеки
| Раскрытые с учётом `extends`

| Deprecated
| Информация о deprecation
| + флаг `isDeprecated`, проверено `removalVersion`

| Плагины
| Все плагины
| Отфильтрованные по distribution

| Диагностика
| —
| `Diagnostics` с ошибками/предупреждениями
|===

=== Структура EffectiveMetadata

[source,kotlin]
----
data class EffectiveMetadata(
    val schemaVersion: String,
    val distribution: String?,
    val libraries: Map<String, EffectiveLibrary>,
    val plugins: Map<String, EffectivePlugin>,
    val bundles: Map<String, EffectiveBundle>,
    val versions: Map<String, ResolvedVersion>,
    val diagnostics: Diagnostics,
    val processingInfo: ProcessingInfo? = null
)

data class EffectiveLibrary(
    val alias: String,
    val group: String,
    val artifact: String,
    val resolvedVersion: String?,
    val versionSource: VersionSource,
    val tags: Set<String>,
    val isDeprecated: Boolean,
    val deprecation: DeprecationInfo?
)

data class ResolvedVersion(
    val alias: String,
    val value: String,
    val source: VersionSource,
    val originalRef: String?  // для BOM_IMPORT — ссылка на BOM
)

enum class VersionSource {
    DECLARED,    // Явно указана в DSL
    BOM_IMPORT   // Импортирована из BOM
}
----

[[processing-diagnostics]]
== Diagnostics

Структура для сбора ошибок и предупреждений во время выполнения pipeline.

=== Уровни диагностики

[cols="1,2,2",options="header"]
|===
| Уровень | Поведение | Примеры

| ERROR
| Pipeline останавливается (в strict mode) или продолжается с пометкой
| Неразрешённая версия, циклические бандлы, невалидные координаты

| WARNING
| Pipeline продолжается, информация сохраняется
| Deprecated библиотека, неиспользуемая версия, пустой distribution

| INFO
| Информационное сообщение
| Импортирована версия из BOM, применён fallback
|===

=== Структура

[source,kotlin]
----
data class Diagnostics(
    val errors: List<DiagnosticMessage>,
    val warnings: List<DiagnosticMessage>,
    val infos: List<DiagnosticMessage>
) {
    fun hasErrors(): Boolean = errors.isNotEmpty()
    fun hasWarnings(): Boolean = warnings.isNotEmpty()
}

data class DiagnosticMessage(
    val code: String,           // e.g., "UNRESOLVED_VERSION", "DEPRECATED_LIBRARY"
    val message: String,        // Human-readable сообщение
    val processorId: String,    // Процессор, который добавил сообщение
    val context: Map<String, Any> = emptyMap()  // Дополнительный контекст
)
----

=== Коды диагностики

[cols="1,2,1",options="header"]
|===
| Код | Описание | Уровень

| `UNRESOLVED_VERSION`
| Версия не разрешена (нет в DSL, нет в BOM)
| ERROR/WARNING

| `UNKNOWN_VERSION_REF`
| Ссылка на несуществующий version alias
| ERROR

| `CIRCULAR_BUNDLE`
| Циклическая зависимость бандлов
| ERROR

| `UNKNOWN_LIBRARY_IN_BUNDLE`
| Бандл ссылается на несуществующую библиотеку
| ERROR

| `INVALID_COORDINATES`
| Невалидные Maven координаты
| ERROR

| `DEPRECATED_LIBRARY`
| Библиотека помечена как deprecated
| WARNING

| `DEPRECATED_PAST_REMOVAL`
| Deprecated библиотека прошла removalVersion
| ERROR

| `EMPTY_DISTRIBUTION`
| Distribution filter исключил все библиотеки
| WARNING

| `UNUSED_VERSION`
| Версия не используется ни одной библиотекой
| INFO

| `BOM_VERSION_IMPORTED`
| Версия импортирована из BOM
| INFO

| `FALLBACK_APPLIED`
| Применена fallback версия
| INFO

| `COMPATIBILITY_VIOLATION`
| Нарушено правило совместимости
| ERROR/WARNING
|===

== SPI: EffectiveMetadataProcessor

=== Описание интерфейса

Интерфейс `EffectiveMetadataProcessor` предоставляет точку расширения для добавления кастомных процессоров в pipeline.

**Основные возможности:**

* `order` — порядок выполнения (меньше = раньше)
* `processorId` — уникальный идентификатор процессора
* `isOptional` — можно ли отключить процессор (по умолчанию `false`)
* `process()` — suspend-функция обработки метаданных

[[compat-rules-processor]]
=== Процессор compat-rules

Процессор `compat-rules` (order: 66) проверяет правила совместимости, определённые в блоке `compatibility {}`.

**Типы правил:**

[cols="1,3",options="header"]
|===
| Тип | Описание

| `jdkRequirement`
| Проверяет требования к версии JDK для указанных библиотек

| `mutualExclusion`
| Проверяет что указанные библиотеки не используются одновременно

| `versionConstraint`
| Проверяет что версия библиотеки соответствует диапазону

| `customRule`
| Кастомная проверка через callback
|===

**Связь с DSL:**

[source,kotlin]
----
dependanger {
    compatibility {
        jdkRequirement("spring-jdk17") {
            libraries("spring-web", "spring-data-jpa")
            minJdk = 17
        }

        mutualExclusion("logging") {
            libraries("slf4j-simple", "logback-classic", "log4j-core")
            message = "Only one logging implementation should be used"
        }

        versionConstraint("kotlin-compat") {
            library = "kotlinx-coroutines-core"
            versionRange = "[1.7.0,)"
        }
    }
}
----

NOTE: Правила проверяются процессором `compat-rules` (order: 66) после валидации (order: 65).

[[deprecated-handling]]
=== Обработка deprecated библиотек

Deprecated библиотеки обрабатываются на нескольких этапах pipeline:

1. **library-filter (order: 20)** — если distribution имеет `byDeprecated { exclude() }`, deprecated исключаются
2. **validation (order: 65)** — проверка `removalVersion`, генерация WARNING/ERROR согласно `onDeprecatedLibrary`
3. **Генерация артефактов** — добавление комментариев о deprecation

**Настройка поведения:**

[source,kotlin]
----
dependanger {
    settings {
        validation {
            onDeprecatedLibrary = WARN  // WARN (default), FAIL, IGNORE
        }
    }
}
----

== Processing Context

`ProcessingContext` содержит информацию об окружении, которая используется процессорами для принятия решений.

=== Параметры контекста

* `jdkVersion` — версия JDK (для условий fallback)
* `kotlinVersion` — версия Kotlin (опционально)
* `gradleVersion` — версия Gradle (опционально)
* `distribution` — активный Distribution
* `environment` — переменные окружения
* `strictVersionResolution` — режим строгой резолюции версий
* `enabledProcessors` / `disabledProcessors` — управление процессорами

== Version Source Priority

При резолвинге версий применяется приоритет:

[source]
----
DECLARED > BOM_IMPORT
----

[cols="1,1,3",options="header"]
|===
| Приоритет | Source | Описание

| 1 (высший)
| DECLARED
| Явно указана в DSL (включая inline версию в coordinates)

| 2 (низший)
| BOM_IMPORT
| Импортирована из BomImport через BomImportProcessor
|===

== Strict Mode

[source,kotlin]
----
dependanger {
    settings {
        strictVersionResolution = true  // ERROR при неразрешённых версиях
    }
}
----

При `strictVersionResolution = true`:

* Неразрешённая версия → ERROR (остановка pipeline)
* При `false` → WARNING (продолжение с пустой версией)

== Parallel Pipeline Execution

Pipeline группирует последовательные процессоры с одинаковым `ExecutionMode` и выполняет каждую группу соответствующим образом:

* **SEQUENTIAL** — строго последовательное выполнение, каждый процессор получает результат предыдущего
* **PARALLEL_IO** — параллельно I/O-bound операции (сетевые запросы) через `coroutineScope` + `async`
* **PARALLEL_COMPUTE** — параллельно CPU-bound операции (валидация) через `coroutineScope` + `async`

Параллельные процессоры получают одну и ту же базовую `EffectiveMetadata`, а результаты объединяются через merge-стратегию: новые diagnostics, updates, vulnerabilities и другие enrichment-поля каждого процессора добавляются к общему результату.

=== Фазы выполнения

[cols="1,2,2",options="header"]
|===
| Фаза | Процессоры | Описание

| SEQUENTIAL
| profile, metadata-conversion, extracted-versions, version-fallback, version-resolver
| Требуют последовательного выполнения (зависят от результатов предыдущих)

| PARALLEL_IO
| bom-import, update-check, security-check, license-check, transitive-resolver
| I/O-bound операции (сетевые запросы к Maven Central, OSV API, ClearlyDefined API)

| PARALLEL_COMPUTE
| validation, compat-rules, compatibility-analysis
| CPU-bound операции (могут выполняться параллельно)
|===

== Pipeline Presets

Предустановленные конфигурации pipeline:

[cols="1,2,2",options="header"]
|===
| Preset | Описание | Включённые процессоры

| `DEFAULT`
| Стандартная конфигурация
| Все обязательные + bom-import

| `MINIMAL`
| Минимальный набор без опциональных
| Только обязательные процессоры

| `STRICT`
| Все проверки включены
| Все процессоры + strictVersionResolution

| `DISTRIBUTION`
| Для генерации артефактов
| Обязательные + profile фильтрация
|===

=== Способы настройки

Есть два эквивалентных способа настройки опциональных процессоров:

[source,kotlin]
----
dependanger {
    processing {
        // Способ 1: Property-style
        importBoms = true
        checkUpdates = true
        analyzeCompatibility = false

        // Способ 2: Builder-style (эквивалентно)
        withBomImport()
        withUpdateCheck()
        withoutCompatibilityAnalysis()

        // Preset + кастомизация
        preset = ProcessingPreset.STRICT
        withoutSecurityCheck()  // отключить один процессор из preset

        // Отключение процессоров по ID
        disabledProcessors = listOf("license-check")
    }
}
----

NOTE: Property-style (`importBoms = true`) и Builder-style (`withBomImport()`) эквивалентны.
Используйте тот стиль, который предпочитаете.

== Pipeline Flow

Pipeline группирует процессоры по `ExecutionMode` (последовательные группы одного режима объединяются) и выполняет каждую группу соответствующим образом:

* **SEQUENTIAL** — процессоры выполняются последовательно, каждый получает результат предыдущего
* **PARALLEL_IO / PARALLEL_COMPUTE** — процессоры запускаются параллельно через `coroutineScope` + `async`, результаты объединяются через merge-стратегию

== Edge Cases

[cols="1,3,2",options="header"]
|===
| Сценарий | Описание | Ожидаемое поведение

| Пустой metadata
| `DependangerMetadata` без библиотек, версий, бандлов
| Pipeline завершается успешно, EffectiveMetadata пустой но валидный

| Несуществующий versionRef
| Библиотека ссылается на `versionRef("unknown")`
| ERROR: "Unknown version reference 'unknown' in library 'xxx'"

| Циклические бандлы
| Бандл A extends B, бандл B extends A
| ERROR: "Circular bundle dependency detected: A → B → A"

| Несуществующая библиотека в бандле
| Бандл ссылается на `libraries("unknown-lib")`
| ERROR: "Unknown library 'unknown-lib' in bundle 'xxx'"

| Несуществующий бандл в extends
| Бандл с `extends("unknown-bundle")`
| ERROR: "Unknown bundle 'unknown-bundle' in extends"

| Несуществующий distribution
| Запрос генерации для `distribution = "unknown"`
| ERROR: "Unknown Distribution 'unknown'"

| Дубликат processorId
| Два процессора с одинаковым `processorId`
| ERROR: "Duplicate processor ID 'xxx'"

| Процессор выбрасывает исключение
| `process()` выбрасывает RuntimeException
| ERROR: "Processor 'xxx' failed: [message]", pipeline останавливается

| Таймаут процессора
| I/O процессор превышает таймаут (30s default)
| ERROR: "Processor 'xxx' timed out after 30s"

| Неразрешённая версия в strict mode
| Библиотека без версии, BOM не содержит, `strictVersionResolution = true`
| ERROR: "Unresolved version for library 'xxx'"

| Неразрешённая версия в non-strict mode
| Библиотека без версии, BOM не содержит, `strictVersionResolution = false`
| WARNING: "Unresolved version for library 'xxx'", `resolvedVersion = null`

| Конфликт fallback условий
| Две fallback версии с перекрывающимися условиями
| Используется первая по порядку определения, WARNING в логе

| Пустой distribution spec
| `distribution("empty") { }` без фильтров
| Все библиотеки включаются (эквивалент отсутствия фильтрации)

| Фильтр исключает все библиотеки
| Distribution фильтр не matching ни одной библиотеки
| WARNING: "Distribution 'xxx' filtered out all libraries", пустой EffectiveMetadata

| Отключённый обязательный процессор
| `disabledProcessors = ["validation"]` (обязательный)
| ERROR: "Cannot disable required processor 'validation'"

| Опциональный процессор без включения
| `bom-import` процессор не включён через `withBomImport()`
| Процессор пропускается, версии из BOM не импортируются

| Параллельные процессоры с общим состоянием
| Два PARALLEL_IO процессора модифицируют один объект
| Использовать thread-safe структуры данных, либо ERROR при data race

| Context без jdkVersion
| `jdkVersion = null`, версия имеет `jdkBelow()` условие
| Fallback условия с JDK не применяются, используется основная версия

| Очень большой metadata
| 10000+ библиотек
| Pipeline завершается (возможно медленнее), рекомендуется PARALLEL фазы

| Прерывание pipeline
| Ctrl+C или kill во время выполнения
| Graceful shutdown, частичные результаты не сохраняются

| Concurrent pipeline executions
| Два pipeline запущены параллельно для одного metadata
| Независимое выполнение, каждый создаёт свой EffectiveMetadata
|===

== Gradle Tasks

=== dependangerGenerateEffective

Основная задача для запуска pipeline и генерации EffectiveMetadata.

**Inputs:**

* `metadata.json` (от `dependangerGenerateMetadata`)
* Настройки из блока `processing { }`

**Outputs:**

* `build/dependanger/effective-{distribution}.json`

**Зависимости:**

* Зависит от `dependangerGenerateMetadata`

=== Связь задач с процессорами

[cols="1,2,2",options="header"]
|===
| Gradle Task | Процессоры | Описание

| `dependangerGenerateEffective`
| Все включённые процессоры
| Полный pipeline

| `dependangerValidate`
| profile → metadata-conversion → validation
| Только валидация

| `dependangerCheckUpdates`
| profile → metadata-conversion → update-check
| Проверка обновлений

| `dependangerAnalyze`
| profile → metadata-conversion → compat-rules → compatibility-analysis
| Анализ совместимости
|===
