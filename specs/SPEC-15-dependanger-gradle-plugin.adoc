[[overview]]
== Обзор

Модуль `dependanger-gradle-plugin` интегрирует Dependanger в Gradle build lifecycle.
Плагин предоставляет DSL-конфигурацию через `DependangerExtension` и 14 Gradle-задач для генерации, валидации и анализа зависимостей.

Plugin ID: `io.github.zenhelix.dependanger`

Зависимость: единственная зависимость -- `dependanger-api` (implementation), которая транзитивно тянет все необходимые модули.

[[common-pattern]]
== Общий паттерн задач

Все задачи следуют единому шаблону:

[source,text]
----
1. Получить DependangerExtension из project
2. Извлечь metadata через extension.dsl.toMetadata()
3. Создать Dependanger через API фасад
4. Выполнить операцию (process/validate/generate)
5. Записать результат в extension.outputDirectory
6. Обработать ошибки с учётом extension.failOnError
----

=== Базовый код получения extension и metadata

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile
    val failOnError = extension.failOnError.get()

    outputDir.mkdirs()

    // ... специфичная логика задачи
}
----

=== runBlocking для suspend-функций

Все suspend-функции Dependanger API вызываются через `runBlocking`:

[source,kotlin]
----
import kotlinx.coroutines.runBlocking

val result = runBlocking {
    dependanger.process(distribution)
}
----

=== Обработка ошибок

[source,kotlin]
----
// Gradle-специфичная обработка ошибок
try {
    // ... операция
} catch (e: DependangerException) {
    if (failOnError) {
        throw GradleException("Dependanger: ${e.message}", e)
    } else {
        logger.warn("Dependanger: ${e.message}")
    }
}

[[gradle-annotations]]
== Аннотации Gradle для кэширования

Для поддержки инкрементальных билдов и Gradle Build Cache задачи должны декларировать inputs и outputs.

=== Рекомендуемая структура задачи с аннотациями

[source,kotlin]
----
import org.gradle.api.DefaultTask
import org.gradle.api.file.DirectoryProperty
import org.gradle.api.file.RegularFileProperty
import org.gradle.api.provider.Property
import org.gradle.api.tasks.*

public abstract class GenerateMetadataTask : DefaultTask() {
    init {
        group = "dependanger"
        description = "Generate metadata.json from DSL"
    }

    @get:Internal
    public val extension: DependangerExtension
        get() = project.extensions.getByType(DependangerExtension::class.java)

    @get:OutputDirectory
    public val outputDirectory: DirectoryProperty
        get() = extension.outputDirectory

    @TaskAction
    public fun execute() {
        // реализация
    }
}
----

NOTE: DSL-конфигурация как `@Input` проблематична, т.к. `DependangerDsl` не `Serializable` для Gradle.
Для полноценного кэширования можно использовать `@Internal` для extension и объявить hashCode/fingerprint DSL через кастомный `@Input`:

[source,kotlin]
----
@get:Input
public val metadataFingerprint: String
    get() {
        val metadata = extension.dsl.toMetadata()
        return metadata.hashCode().toString()
    }
----

На первом этапе реализации можно использовать `@Internal` для extension и не кэшировать задачи.
Кэширование -- отдельная задача оптимизации.

[[dsl-fingerprinting]]
=== DSL fingerprinting: подход первой итерации

В первой итерации для up-to-date проверок используется простой hashCode от metadata.
Это не идеально для incremental builds (hashCode может давать коллизии, а Gradle предпочитает детерминированную сериализацию inputs), но работает корректно для определения изменений DSL-конфигурации.

[source,kotlin]
----
@Input
public fun getDslHashCode(): Int = dsl.toMetadata().hashCode()
----

NOTE: Это решение первой итерации.
В будущем для более надёжной поддержки incremental builds рекомендуется перейти на полноценный fingerprint -- например, сериализовать `DependangerMetadata` в JSON-строку и использовать её как `@Input`, что даст Gradle точную информацию об изменениях без коллизий hashCode.
Пометить как area для улучшения.

[[spec-generate-metadata]]
== SPEC-15.1: GenerateMetadataTask

=== Назначение

Конвертировать DSL-конфигурацию из Gradle build скрипта в сериализованный `metadata.json`.

=== Алгоритм

[source,text]
----
1. Получить extension из project
2. Вызвать extension.dsl.toMetadata() -> DependangerMetadata
3. Создать JsonSerializationFormat
4. Записать metadata в outputDir/metadata.json
5. Логировать результат
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile

    outputDir.mkdirs()

    val format = JsonSerializationFormat()
    val outputFile = outputDir.resolve("metadata.json").toPath()
    format.write(metadata, outputFile)

    logger.lifecycle("Dependanger: Generated metadata.json -> $outputFile")
    logger.lifecycle("  Versions: ${metadata.versions.size}")
    logger.lifecycle("  Libraries: ${metadata.libraries.size}")
    logger.lifecycle("  Plugins: ${metadata.plugins.size}")
    logger.lifecycle("  Bundles: ${metadata.bundles.size}")
}
----

=== Edge cases

[cols="1,2"]
|===
| Ситуация | Поведение

| Пустой DSL | Создаёт metadata.json с пустыми списками (валидный сценарий)
| DSL с ошибками (невалидные координаты) | Ошибка в toMetadata(), оборачивается в GradleException
| Output-директория не существует | Создаётся через mkdirs()
|===

[[spec-generate-effective]]
== SPEC-15.2: GenerateEffectiveTask

=== Назначение

Обработать metadata через pipeline и создать effective-metadata с разрешёнными версиями, применёнными фильтрами и собранной диагностикой.

=== Алгоритм

[source,text]
----
1. Получить extension
2. Построить metadata из DSL
3. Создать Dependanger через fromMetadata(metadata)
4. Вызвать process() через runBlocking
5. Сериализовать effective metadata в outputDir/effective.json
6. Вывести диагностику (warnings/errors)
7. Если failOnError && diagnostics.hasErrors -> throw GradleException
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile
    val failOnError = extension.failOnError.get()

    outputDir.mkdirs()

    val dependanger = Dependanger.fromMetadata(metadata).build()
    val result = runBlocking { dependanger.process() }

    // Вывод диагностики
    result.diagnostics.errors.forEach { logger.error("Dependanger ERROR: ${it.message}") }
    result.diagnostics.warnings.forEach { logger.warn("Dependanger WARN: ${it.message}") }
    result.diagnostics.infos.forEach { logger.info("Dependanger INFO: ${it.message}") }

    if (result.diagnostics.hasErrors && failOnError) {
        throw GradleException(
            "Dependanger processing failed with ${result.diagnostics.errors.size} error(s). " +
            "Set dependanger { failOnError = false } to continue despite errors."
        )
    }

    if (result.effective != null) {
        // Сериализация effective metadata
        // TODO: Зависит от наличия EffectiveJsonFormat или аналога
        val outputFile = outputDir.resolve("effective.json")
        // effectiveFormat.write(result.effective, outputFile.toPath())
        logger.lifecycle("Dependanger: Generated effective.json -> $outputFile")
    }
}

[[spec-generate-toml]]
== SPEC-15.3: GenerateTomlTask

=== Назначение

Сгенерировать TOML Version Catalog из effective metadata.

=== Алгоритм

[source,text]
----
1. Получить extension, metadata, outputDir
2. Создать Dependanger -> process()
3. Извлечь TomlConfig из settings metadata
4. Вызвать result.toToml(config)
5. Записать в outputDir/libs.versions.toml
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile

    outputDir.mkdirs()

    val dependanger = Dependanger.fromMetadata(metadata).build()
    val result = runBlocking { dependanger.process() }

    if (!result.isSuccess) {
        handleProcessingErrors(result, extension.failOnError.get())
        return
    }

    // Создание TomlConfig из settings metadata
    val tomlSettings = metadata.settings?.toml
    val tomlConfig = TomlConfig(
        includeComments = tomlSettings?.includeComments ?: true,
        sortSections = tomlSettings?.sortSections ?: true,
    )

    val tomlFilename = tomlSettings?.filename ?: "libs.versions.toml"
    val tomlPath = outputDir.resolve(tomlFilename).toPath()
    result.writeTomlTo(tomlPath, tomlConfig)

    logger.lifecycle("Dependanger: Generated $tomlFilename -> $tomlPath")
}
----

== SPEC-15.4: GenerateBomTask

=== Назначение

Сгенерировать Maven BOM (pom.xml с dependencyManagement) из effective metadata.

=== Алгоритм

[source,text]
----
1. Получить extension, metadata, outputDir
2. Создать Dependanger -> process()
3. Извлечь BomConfig из settings metadata
4. Вызвать result.toBom(config)
5. Записать в outputDir/bom.pom.xml

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile

    outputDir.mkdirs()

    val dependanger = Dependanger.fromMetadata(metadata).build()
    val result = runBlocking { dependanger.process() }

    if (!result.isSuccess) {
        handleProcessingErrors(result, extension.failOnError.get())
        return
    }

    val bomSettings = metadata.settings?.bom
    val bomConfig = BomConfig(
        groupId = bomSettings?.groupId ?: project.group.toString(),
        artifactId = bomSettings?.artifactId ?: "${project.name}-bom",
        version = bomSettings?.version ?: project.version.toString(),
    )

    val bomPath = outputDir.resolve("bom.pom.xml").toPath()
    result.writeBomTo(bomPath, bomConfig)

    logger.lifecycle("Dependanger: Generated bom.pom.xml -> $bomPath")
}
----

NOTE: BomConfig для `toBom()` не имеет значения по умолчанию (в отличие от `toToml()`), поэтому groupId и artifactId обязательны.
Они берутся из settings или из project properties.

[[spec-analyze-task]]
== SPEC-15.9: AnalyzeTask

=== Назначение

Анализ совместимости зависимостей.

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val failOnError = extension.failOnError.get()

    val dependanger = Dependanger.fromMetadata(metadata)
        .preset(ProcessingPreset.STRICT)
        .jdkVersion(Runtime.version().feature()) // текущая JDK
        .build()

    val result = runBlocking { dependanger.process() }

    // Compatibility issues отражаются в diagnostics
    val errors = result.diagnostics.errors.filter { it.code?.startsWith("COMPAT") == true }
    val warnings = result.diagnostics.warnings.filter { it.code?.startsWith("COMPAT") == true }

    if (errors.isEmpty() && warnings.isEmpty()) {
        logger.lifecycle("Dependanger: No compatibility issues found.")
    } else {
        errors.forEach { logger.error("Dependanger COMPAT ERROR: ${it.message}") }
        warnings.forEach { logger.warn("Dependanger COMPAT WARN: ${it.message}") }
        logger.lifecycle("Dependanger: Analysis complete: ${errors.size} errors, ${warnings.size} warnings")
    }

    if (errors.isNotEmpty() && failOnError) {
        throw GradleException("Dependanger: Compatibility analysis found ${errors.size} error(s).")
    }
}

[[spec-list-versions-task]]
== SPEC-15.11: ListVersionsTask

=== Назначение

Вывод всех версий из effective metadata в консоль.
Информационная задача для отладки.

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()

    val dependanger = Dependanger.fromMetadata(metadata).build()
    val result = runBlocking { dependanger.process() }

    if (result.effective == null) {
        logger.error("Dependanger: Failed to resolve versions")
        return
    }

    val versions = result.effective!!.versions

    logger.lifecycle("Dependanger: Resolved versions (${versions.size}):")
    logger.lifecycle("")

    val maxAliasLen = versions.keys.maxOfOrNull { it.length } ?: 0
    versions.entries.sortedBy { it.key }.forEach { (alias, resolved) ->
        val paddedAlias = alias.padEnd(maxAliasLen)
        logger.lifecycle("  $paddedAlias = ${resolved.value}")
    }
}

    val metadata = extension.dsl.toMetadata()
    val failOnError = extension.failOnError.get()

        logger.error("Dependanger: Cannot run security check -- processing failed")
        return
    }

    if (vulnerabilities.isEmpty()) {
        logger.lifecycle("Dependanger: No known vulnerabilities found.")
    } else {
        logger.lifecycle("Dependanger: Found ${vulnerabilities.size} vulnerabilities:")
        vulnerabilities.forEach { vuln ->
            logger.warn("  ${vuln.library} (${vuln.cveId}): ${vuln.severity} - ${vuln.summary}")
        }

        outputDir.mkdirs()
        val reportFile = outputDir.resolve("security-report.json")
        // Сериализация vulnerabilities в JSON
        logger.lifecycle("Dependanger: Security report -> $reportFile")
    }

    if (vulnerabilities.any { it.severity == "CRITICAL" || it.severity == "HIGH" } && failOnError) {
        throw GradleException(
            "Dependanger: Found ${vulnerabilities.size} vulnerabilities (HIGH or CRITICAL)."
        )
    }

    outputDir.mkdirs()
    val reportFile = outputDir.resolve("license-report.json")
    logger.lifecycle("Dependanger: License report -> $reportFile")

    if (denied.isNotEmpty() && failOnError) {
        throw GradleException("Dependanger: Found ${denied.size} denied license(s).")
    }
}

// DependangerTaskHelper.kt (новый файл)
internal object DependangerTaskHelper {

    fun getExtension(project: Project): DependangerExtension =
        project.extensions.getByType(DependangerExtension::class.java)

    fun buildMetadata(extension: DependangerExtension): DependangerMetadata =
        extension.dsl.toMetadata()

    fun handleProcessingErrors(
        result: DependangerResult,
        failOnError: Boolean,
        logger: Logger,
    ) {
        result.diagnostics.errors.forEach { logger.error("Dependanger ERROR: ${it.message}") }
        result.diagnostics.warnings.forEach { logger.warn("Dependanger WARN: ${it.message}") }

        if (result.diagnostics.hasErrors && failOnError) {
            throw GradleException(
                "Dependanger processing failed with ${result.diagnostics.errors.size} error(s)."
            )
        }
    }

    fun ensureOutputDir(extension: DependangerExtension): File {
        val dir = extension.outputDirectory.get().asFile
        dir.mkdirs()
        return dir
    }
}

[[error-handling]]
== Обработка ошибок

[cols="1,2,2"]
|===
| Ситуация | Поведение | Обоснование

| DSL.toMetadata() выбрасывает исключение | GradleException с сообщением об ошибке DSL | Ошибки конфигурации должны быть чётко видны
| Pipeline processing failure | Если failOnError -> GradleException; иначе -> warning в лог | Конфигурируемое поведение
| Генератор (TOML/BOM) fails | GradleException всегда | Ошибка генерации -- критическая
| Output directory не writable | GradleException | Ошибка файловой системы
| Network error (updates/security) | Если failOnError -> GradleException; иначе -> warning + пропуск feature | Сетевые ошибки не должны блокировать build по умолчанию
|===

=== Gradle Exception Types

[cols="1,2"]
|===
| Тип | Использование

| `GradleException` | Стандартное прерывание build при ошибках
| `StopExecutionException` | Мягкое прерывание задачи без ошибки build (UP-TO-DATE)

[[incremental-builds]]
== Поддержка инкрементальных билдов

=== Первый этап (текущая реализация)

Все задачи выполняются заново при каждом запуске.
Это приемлемо для первой итерации.

=== Второй этап (оптимизация)

Для полноценной поддержки инкрементальных билдов:

1. **GenerateMetadataTask**: `@Input` = fingerprint DSL, `@OutputFile` = metadata.json
2. **GenerateEffectiveTask**: `@InputFile` = metadata.json, `@OutputFile` = effective.json
3. **GenerateTomlTask**: `@InputFile` = effective.json, `@OutputFile` = libs.versions.toml
4. **GenerateBomTask**: `@InputFile` = effective.json, `@OutputFile` = bom.pom.xml
5. **ValidateTask**: `@InputFile` = metadata.json (output -- только лог)
6. **Analysis tasks**: `@InputFile` = effective.json (output -- report file)

NOTE: Для корректной работы incremental build необходимо чтобы задачи читали/писали файлы, а не пересоздавали pipeline при каждом запуске.
Это потребует рефакторинга -- разделения DSL -> metadata -> effective -> artifacts на отдельные шаги с файловыми inputs/outputs.
