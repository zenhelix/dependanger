[[spec-to-metadata]]
== SPEC-01.1: DependangerDsl.toMetadata()

[[purpose-to-metadata]]
=== Назначение

Конвертировать in-memory DSL-конфигурацию в сериализуемую модель `DependangerMetadata`.
Это ключевая точка перехода из «мира DSL» в «мир данных».

[[dsl-data-access]]
=== Доступ к данным DSL

Списковые DSL хранят данные в `MutableList` полях (напр. `versionsDsl.versions`).
`SettingsDsl` предоставляет `toSettings(): Settings`.
В `toMetadata()` данные копируются через `.toList()` для иммутабельности.

==== VersionsDsl: versions: MutableList<Version>

Доступ: `versionsDsl.versions.toList()`

*Алгоритм:*

1. Итерировать по внутреннему списку зарегистрированных версий
2. Для каждой записи создать `Version(name, value, fallbacks)`
3. Fallbacks берутся из `VersionDsl.fallbacks` — список `VersionFallback(value, condition)`
4. Condition строится из `FallbackConditionDsl`

*Валидация (опциональная, может быть перенесена в ValidationProcessor):*

- Имена версий уникальны
- Значения версий не пустые

*Input/Output пример:*

[source,kotlin]
----
// DSL
versions {
    version("kotlin", "2.1.20") {
        fallback("1.9.25") { jdkBelow(17) }
    }
    version("spring-boot", "3.4.0")
}

// Output
listOf(
    Version("kotlin", "2.1.20", listOf(
        VersionFallback("1.9.25", FallbackCondition.JdkBelow(17))
    )),
    Version("spring-boot", "3.4.0", emptyList()),
)
----

==== LibrariesDsl: libraries: MutableList<Library>

Доступ: `librariesDsl.libraries.toList()`

*Алгоритм:*

1. Итерировать по зарегистрированным библиотекам
2. Для каждой создать `Library` из данных `LibraryDsl`
3. `group` и `artifact` разбираются из строки координат `"group:artifact"` (уже реализовано в DSL через `parseCoordinates`)
4. `version` — опциональный `VersionReference` (Literal, Reference или Range)
5. Собрать tags, requires, deprecation, license, constraints, isPlatform из LibraryDsl

*Парсинг координат:*

- `"org.jetbrains.kotlin:kotlin-stdlib"` → group=`org.jetbrains.kotlin`, artifact=`kotlin-stdlib`
- `"org.jetbrains.kotlin:kotlin-stdlib:2.1.20"` → group, artifact + version=Literal("2.1.20")

*Input/Output пример:*

[source,kotlin]
----
// DSL
libraries {
    library("kotlin-stdlib", "org.jetbrains.kotlin:kotlin-stdlib") {
        version = versionRef("kotlin")
        tags("kotlin", "stdlib")
    }
}

// Output
listOf(
    Library(
        alias = "kotlin-stdlib",
        group = "org.jetbrains.kotlin",
        artifact = "kotlin-stdlib",
        version = VersionReference.Reference("kotlin"),
        tags = setOf("kotlin", "stdlib"),
    ),
)
----

=== buildExtensionsMap(): Map<String, JsonElement>

[source,kotlin]
----
private fun buildExtensionsMap(): Map<String, JsonElement> {
    val result = mutableMapOf<String, JsonElement>()
    for ((key, value) in allExtensions()) {
        // Попытка сериализовать через kotlinx-serialization
        // Если значение @Serializable — сериализовать в JsonElement
        // Если нет — пропустить с warning или выбросить исключение
        result[key.name] = Json.encodeToJsonElement(serializer(value::class.starProjectedType), value)
    }
    return result
}
----

*Edge cases:*

- Extension не @Serializable → выбрасывать `IllegalStateException` с сообщением о несериализуемом расширении
- Пустой map extensions → пустой map в metadata (OK)
- Extension с одинаковым именем → последний побеждает (уже обеспечивается Map)

[[spec-apply-preset]]
== SPEC-01.2: DependangerDsl.applyPreset(name: String)

[[purpose-apply-preset]]
=== Назначение

Найти зарегистрированный пресет по имени и применить его настройки к текущему DSL.

[[algorithm-apply-preset]]
=== Алгоритм

[source,text]
----
1. Найти preset по name в presetsDsl.presets
2. Если не найден → throw IllegalArgumentException("Preset '$name' not found")
3. Применить preset.settings → merge с текущими settingsDsl (preset значения перезаписывают текущие)
----

NOTE: `preset.bundles` и `preset.distributions` содержат имена (ссылки) на бандлы и дистрибуции, определённые в DSL.
При вызове `applyPreset()` они не мержатся в DSL-списки (типы несовместимы: `List<String>` vs `MutableList<Bundle/Distribution>`).
Эти данные доступны через `metadata.presets` и используются потребителями (API facade, CLI, Gradle Plugin) для определения scope обработки.
Например, API facade при `process(preset="android")` читает `preset.distributions` и запускает pipeline для каждой дистрибуции, а `preset.bundles` определяет, какие бандлы включить.

[[implementation-apply-preset]]
=== Детальная реализация

Используется двухуровневый подход к merge:

* **DSL-путь** (основной): `PresetsDsl` хранит `PresetDsl`-объекты рядом с `Preset` data classes.
При `applyPreset()` используется `PresetDsl.settingsDsl`, где каждое поле обёрнуто в `Trackable<T>` delegate.
`Trackable.isSet` отслеживает, было ли поле явно установлено пользователем.
Применяются только те поля, для которых `isSet == true`.

* **JSON-путь** (fallback): Если preset загружен из JSON (нет DSL-объекта), `mergeFrom(Settings)` сравнивает поля с `Settings()` defaults.
Отличающиеся от default значения перезаписывают текущие.

[source,kotlin]
----
public fun applyPreset(name: String) {
    // bundles и distributions — строковые ссылки на определённые в DSL сущности.
    // Доступны через metadata.presets для потребителей (API, CLI, Gradle Plugin),
    // которые используют их для определения scope обработки.

    val presetDsl = presetsDsl.findDslByName(name)
    if (presetDsl != null) {
        // DSL-путь: точный merge через Trackable — применяются только явно установленные поля
        presetDsl.settingsDsl?.applyTo(settingsDsl)
        return
    }

    // JSON-путь (fallback): preset загружен из файла, DSL-объекта нет
    val preset = presetsDsl.findByName(name)
        ?: throw IllegalArgumentException(
            "Preset '$name' not found. Available presets: ${presetsDsl.availableNames()}"
        )
    preset.settings?.let { presetSettings ->
        settingsDsl.mergeFrom(presetSettings)
    }
}
----

*Edge cases:*

- Preset не найден → `IllegalArgumentException` с перечислением доступных
- Множественный applyPreset → каждый последующий мержит свои настройки поверх предыдущих
- Пустой пресет (без settings) → no-op (bundles/distributions всё равно доступны через metadata.presets)
- Поле явно установлено в default значение через DSL → `Trackable.isSet == true` → применяется корректно
- Поле не тронуто в preset → `Trackable.isSet == false` → текущее значение сохраняется

[[dsl-build-contract]]
== Контракт доступа к данным DSL

Списковые DSL (VersionsDsl, LibrariesDsl, etc.) хранят данные в публичных `MutableList` полях.
`SettingsDsl` предоставляет `toSettings(): Settings`.
В `toMetadata()` списки копируются через `.toList()`.

=== Требования:

1. **Иммутабельность результата** — `toMetadata()` создаёт defensive copies через `.toList()`
2. **Нет валидации** — `toMetadata()` конвертирует «as is», валидация происходит в pipeline
3. **Данные — data class** — все возвращаемые модели неизменяемые

=== Требования к merge настроек

==== DSL-путь: `SettingsDsl.applyTo()` через Trackable delegates

Каждое поле в `PresetDsl`-версии settings DSL использует `Trackable<T>` delegate:

[source,kotlin]
----
public class Trackable<T>(default: T) : ReadWriteProperty<Any?, T> {
    public var isSet: Boolean = false; private set
    private var value: T = default

    override fun getValue(thisRef: Any?, property: KProperty<*>): T = value
    override fun setValue(thisRef: Any?, property: KProperty<*>, value: T) {
        this.value = value
        isSet = true
    }
}
----

Метод `applyTo(target: SettingsDsl)` применяет к target только поля с `isSet == true`.
Вложенные sub-settings (validation, toml, bom, etc.) трекаются целиком: если блок `validation { }` был вызван, весь `ValidationSettings` считается установленным.
Списки (allowedLicenses, excludePatterns) заменяются целиком, не дополняются.

==== JSON-путь (fallback): `SettingsDsl.mergeFrom(Settings)`

[source,kotlin]
----
public fun mergeFrom(settings: Settings)
----

1. Сравнивает каждое поле `settings` с `Settings()` (конструктор без аргументов)
2. Если значение отличается от default → перезаписывает текущее
3. Вложенные settings (ValidationSettings, TomlSettings, etc.) сравниваются целиком (`!=`)
4. Списки заменяются целиком, не дополняются

NOTE: JSON-путь не может отличить "явно установлено в default" от "не установлено".
Это допустимое ограничение: при создании preset из JSON пользователь задаёт только нужные поля, остальные отсутствуют и получают default при десериализации.

=== GlobMatcher

Утилита для glob-matching паттернов по Maven-координатам.
Используется в: LibraryFilterProcessor (SPEC-03), CompatRulesProcessor (SPEC-03), CompatibilityCheckProcessor (SPEC-08), UpdateCheckProcessor (SPEC-07).

[source,kotlin]
----
package io.github.zenhelix.dependanger.core.util

/**
 * Glob-matcher для Maven-координат и строковых паттернов.
 *
 * Поддерживаемые wildcard-символы:
 * - `*` — любая подстрока (включая пустую)
 * - `?` — любой одиночный символ
 *
 * Примеры паттернов:
 * - `"org.springframework:*"` — все артефакты группы org.springframework
 * - `"org.springframework*:*"` — org.springframework И org.springframework.boot
 * - `"*:spring-*"` — все группы, артефакты начинающиеся с "spring-"
 * - `"org.jetbrains.kotlin:kotlin-stdlib"` — точное совпадение
 */
public object GlobMatcher {

    /**
     * Проверяет совпадение координат "group:artifact" с glob-паттерном.
     *
     * @param pattern glob-паттерн в формате "groupPattern:artifactPattern"
     * @param group Maven groupId библиотеки
     * @param artifact Maven artifactId библиотеки
     * @return true если координаты соответствуют паттерну
     */
    public fun matches(pattern: String, group: String, artifact: String): Boolean {
        val (patternGroup, patternArtifact) = pattern.split(":", limit = 2)
            .let { if (it.size == 2) it[0] to it[1] else it[0] to "*" }

        return matchesGlob(patternGroup, group) && matchesGlob(patternArtifact, artifact)
    }

    /**
     * Проверяет совпадение строки с glob-паттерном.
     *
     * @param pattern glob-паттерн (может содержать * и ?)
     * @param value проверяемая строка
     * @return true если строка соответствует паттерну
     */
    public fun matchesGlob(pattern: String, value: String): Boolean {
        if (pattern == "*") return true

        // Экранируем ВСЕ regex-спецсимволы кроме * и ?
        val regex = buildString {
            append("^")
            for (ch in pattern) {
                when (ch) {
                    '*' -> append(".*")
                    '?' -> append(".")
                    '.', '+', '^', '$', '{', '}', '(', ')', '|', '[', ']', '\\' ->
                        append("\\").append(ch)
                    else -> append(ch)
                }
            }
            append("$")
        }
        return Regex(regex).matches(value)
    }

    /**
     * Проверяет совпадение строки координат "group:artifact" с glob-паттерном.
     * Удобный метод для случаев, когда координаты уже объединены.
     */
    public fun matchesCoordinate(pattern: String, coordinate: String): Boolean {
        val parts = coordinate.split(":", limit = 2)
        return if (parts.size == 2) {
            matches(pattern, parts[0], parts[1])
        } else {
            matchesGlob(pattern, coordinate)
        }
    }
}

==== Отличия от предыдущих наивных реализаций

IMPORTANT: Предыдущие реализации в SPEC-03 и SPEC-08 содержали баг: regex-спецсимволы (`.`, `+`, `^`, `$`, etc.) не экранировались корректно.
Например, паттерн `"org.springframework:*"` при наивной замене `"."` → `"\\."` и `"*"` → `".*"` мог некорректно обрабатывать символы вроде `+` или `$` в group/artifact.

Каноническая реализация `GlobMatcher` экранирует все 12 regex-спецсимволов: `. + ^ $ { } ( ) | [ ] \`

==== Тестовые сценарии GlobMatcher

[cols="1,2,1"]
|===
| Паттерн | Вход (group:artifact) | Результат

| `org.springframework:*` | `org.springframework:spring-core` | true
| `org.springframework:*` | `org.springframework.boot:spring-boot` | false
| `org.springframework*:*` | `org.springframework.boot:spring-boot` | true
| `*:spring-*` | `org.springframework:spring-web` | true
| `*:*` | любые координаты | true
| `org.jetbrains.kotlin:kotlin-stdlib` | `org.jetbrains.kotlin:kotlin-stdlib` | true
| `org.jetbrains.kotlin:kotlin-stdlib` | `org.jetbrains.kotlin:kotlin-test` | false
| `org.example+plus:artifact` | `org.example+plus:artifact` | true (+ экранирован)
| `*:*-internal` | `com.company:utils-internal` | true

=== VersionComparator и SemanticVersion

Каноническая реализация сравнения семантических версий.
Используется в: VersionFallbackProcessor (SPEC-03), CompatRulesProcessor (SPEC-03), UpdateCheckProcessor (SPEC-07), CompatibilityCheckProcessor (SPEC-08), TransitiveResolverProcessor (SPEC-12).

[source,kotlin]
----
package io.github.zenhelix.dependanger.core.util

/**
 * Парсированная семантическая версия.
 *
 * Поддерживаемые форматы:
 * - `1.0.0`, `1.0`, `1`
 * - `1.0.0-RC1`, `1.0.0.RELEASE`, `1.0.0-beta.2`
 * - `2.1.20`, `3.4.0.Final`
 *
 * Суффиксы RELEASE, FINAL, GA удаляются при парсинге (считаются стабильными).
 */
@Serializable
public data class SemanticVersion(
    val major: Int,
    val minor: Int,
    val patch: Int,
    val qualifier: String? = null,
) : Comparable<SemanticVersion> {

    override fun compareTo(other: SemanticVersion): Int {
        if (major != other.major) return major.compareTo(other.major)
        if (minor != other.minor) return minor.compareTo(other.minor)
        if (patch != other.patch) return patch.compareTo(other.patch)
        // stable (qualifier=null) > prerelease (qualifier!=null)
        return when {
            qualifier == null && other.qualifier == null -> 0
            qualifier == null -> 1   // this is stable, other is prerelease
            other.qualifier == null -> -1
            else -> qualifier.compareTo(other.qualifier)
        }
    }

    override fun toString(): String = buildString {
        append("$major.$minor.$patch")
        qualifier?.let { append("-$it") }
    }
}
----

NOTE: Compound qualifiers (например `"RC1.2"`) парсятся как единая строка `qualifier="RC1.2"`.
Дальнейшая декомпозиция qualifier не выполняется.

==== UpdateType enum

Перечисление типов обновлений.
Ordinal используется для фильтрации (PATCH < MINOR < MAJOR).

[source,kotlin]
----
package io.github.zenhelix.dependanger.core.util

@Serializable
public enum class UpdateType { PRERELEASE, PATCH, MINOR, MAJOR }
----

[source,kotlin]
----
package io.github.zenhelix.dependanger.core.util

/**
 * Утилита для парсинга и сравнения семантических версий.
 */
public object VersionComparator {

    private val STABLE_SUFFIXES = setOf("release", "final", "ga")

    private val PRERELEASE_PATTERNS = listOf(
        "alpha", "beta", "rc", "cr", "snapshot", "dev", "pre", "ea",
        "m\\d+",  // M1, M2, ...
    )

    private val prereleaseRegex: Regex = Regex(
        PRERELEASE_PATTERNS.joinToString("|") { "(?i)[-.]?$it" }
    )

    /**
     * Парсит строку версии в [SemanticVersion].
     * Возвращает null если парсинг невозможен (нечисловой major).
     */
    public fun parse(version: String): SemanticVersion? {
        // Удаляем стабильные суффиксы: 1.0.0.RELEASE -> 1.0.0
        val cleaned = version.replace(
            Regex("[-.]?(${STABLE_SUFFIXES.joinToString("|")})$", RegexOption.IGNORE_CASE), ""
        )
        val parts = cleaned.split(Regex("[.-]"), limit = 4)
        val major = parts.getOrNull(0)?.toIntOrNull() ?: return null
        val minor = parts.getOrNull(1)?.toIntOrNull() ?: 0
        val patch = parts.getOrNull(2)?.toIntOrNull() ?: 0
        val qualifier = parts.getOrNull(3)
        return SemanticVersion(major, minor, patch, qualifier)
    }

    /**
     * Определяет, является ли версия предварительным релизом.
     */
    public fun isPrerelease(version: String): Boolean =
        prereleaseRegex.containsMatchIn(version)

    /**
     * Сравнивает две строковые версии.
     * Если парсинг невозможен — fallback на строковое сравнение.
     */
    public fun compare(a: String, b: String): Int {
        val va = parse(a)
        val vb = parse(b)
        return when {
            va != null && vb != null -> va.compareTo(vb)
            else -> a.compareTo(b)
        }
    }

    /**
     * Классифицирует тип обновления между двумя версиями.
     */
    public fun classifyUpdate(current: SemanticVersion, latest: SemanticVersion): UpdateType =
        when {
            latest.major > current.major -> UpdateType.MAJOR
            latest.minor > current.minor -> UpdateType.MINOR
            latest.patch > current.patch -> UpdateType.PATCH
            else -> UpdateType.PRERELEASE
        }

    /**
     * Извлекает major-компонент версии.
     */
    public fun parseMajor(version: String): String =
        version.split(".").firstOrNull() ?: version

    /**
     * Извлекает major.minor компонент версии.
     */
    public fun parseMajorMinor(version: String): String =
        version.split(".").take(2).joinToString(".")

    /**
     * Выбирает наивысшую версию из списка.
     */
    public fun selectHighest(versions: List<String>): String? =
        versions.maxWithOrNull(Comparator { a, b -> compare(a, b) })
}
----

=== CustomRuleHandler SPI

Интерфейс для обработки пользовательских правил совместимости (`CompatibilityRule.CustomRule`).
Регистрация через ServiceLoader: `META-INF/services/io.github.zenhelix.dependanger.effective.spi.CustomRuleHandler`.

NOTE: Интерфейс находится в модуле `dependanger-effective` (пакет `effective.spi`), т.к. зависит от типов `EffectiveLibrary`, `ProcessingContext` и `CompatibilityIssue`.

[source,kotlin]
----
package io.github.zenhelix.dependanger.effective.spi

/**
 * SPI для обработки пользовательских правил совместимости.
 * Реализации обнаруживаются через ServiceLoader.
 */
public interface CustomRuleHandler {
    /** Уникальный идентификатор типа правила. */
    public val ruleType: String

    /**
     * Выполнить проверку правила и вернуть найденные проблемы.
     */
    public fun evaluate(
        rule: CompatibilityRule.CustomRule,
        libraries: Map<String, EffectiveLibrary>,
        context: ProcessingContext,
    ): List<CompatibilityIssue>
}
----

Используется в:

* xref:SPEC-08-dependanger-analysis.adoc[SPEC-08] -- `CompatibilityCheckProcessor` для обработки `CustomRule` через SPI
* xref:SPEC-03-dependanger-effective.adoc[SPEC-03] -- `CompatRulesProcessor` пропускает `CustomRule` с INFO `COMPAT_CUSTOM_RULE_DEFERRED` и делегирует обработку в `CompatibilityCheckProcessor` (order=110)
