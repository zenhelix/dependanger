# /doc-check

Проверить полноту документа и получить рекомендацию: продолжать, остановиться или подождать.

## Использование

```
/doc-check [путь]               # файл или папка
/doc-check                      # проверить активный документ
```

**Режимы:**

- **Файл:** проверка одного документа
- **Папка:** последовательная проверка всех `.adoc` файлов в папке (и подпапках)

## Примеры

```
/doc-check docs/02-features/F007-update-check.adoc   # один файл
/doc-check docs/02-features/                         # вся папка features
/doc-check docs/                                     # весь docs
```

## Инструкции для агента

$ARGUMENTS содержит путь к документу или папке. Если пустой — запросить путь у пользователя.

### Определение режима

1. **Проверить, что указано в $ARGUMENTS:**
    - Если путь заканчивается на `.adoc` → режим одного файла
    - Если путь — директория → режим папки
    - Если пустой → запросить путь

2. **Режим папки:**
    - Найти все `.adoc` файлы в указанной папке (рекурсивно)
    - Отсортировать по имени (F001, F002... или ADR-001, ADR-002...)
    - Обработать каждый файл по очереди
    - Показать сводный отчёт в конце

### Шаги выполнения

1. **Прочитать документ** по указанному пути

2. **Определить тип документа:**
   - Feature (в `02-features/`)
   - ADR (в `05-decisions/`)
   - Concept (в `01-concepts/`)
   - Interface (в `03-interfaces/`)

3. **Определить текущий уровень зрелости:**
   - L0 Draft — идея есть, структура неполная
   - L1 Minimal Viable — цель ясна, основные user stories
   - L2 Development Ready — AC есть, edge cases описаны
   - L3 Complete — всё тестируемо, все вопросы отвечены

4. **Проверить stakeholder questions** (в зависимости от типа):

   **Для Features:**
   | Stakeholder | Вопрос |
   |-------------|--------|
   | Product Owner | Какую проблему решает? |
   | Product Owner | Какова ценность? |
   | Разработчик | Что делать? |
   | Разработчик | Какие AC? |
   | Разработчик | Какие edge cases? |
   | Тестировщик | Как проверить? |
   | Архитектор | Какие модули затрагивает? |

   **Для ADR:**
   | Stakeholder | Вопрос |
   |-------------|--------|
   | Будущий разработчик | Почему выбрано это решение? |
   | Будущий разработчик | Какие альтернативы рассматривались? |
   | Будущий разработчик | Какие trade-offs приняты? |

5. **Проверить testability** (для Features):
   - Каждый AC должен быть в формате Given/When/Then
   - Не должно быть неоднозначных формулировок ("правильно", "быстро", "удобно")

6. **Каскадная проверка консистентности:**

   Найти связанные документы и проверить их актуальность:

   | Тип документа | Где искать связи |
      |---------------|------------------|
   | Feature | → Interfaces (описывают фичу)<br>→ ADR (решения по фиче)<br>→ Другие Features (зависимости)<br>→ Concepts (базовые понятия) |
   | ADR | → Features (используют решение)<br>→ Concepts (архитектурные изменения) |
   | Concept | → Features (опираются на концепт) |
   | Interface | → Features (реализует) |

   **Как искать связи:**
    - Найти все `xref:` ссылки В документе (forward links)
    - Поискать `xref:` на этот документ в других файлах (backlinks)
    - Проверить секцию "Зависимости" / "Dependencies"

   **Что проверять:**
    - Ссылки не битые?
    - Терминология согласована?
    - Нет противоречий?
    - Статусы согласованы? (например, ADR не Superseded)

7. **Выдать рекомендацию:**

   **STOP** если:
   - Достигнут целевой уровень
   - Все stakeholder questions отвечены
   - Нет блокирующих вопросов

   **ITERATE** если:
   - Stakeholder не может работать по документу
   - Есть конкретные проблемы которые нужно исправить
   - Нужен переход на следующий уровень

   **WAIT** если:
   - Нужен input который недоступен
   - Зависимость от непринятого решения

### Формат ответа

```markdown
## Анализ документа: [название файла]

**Тип:** [Feature / ADR / Concept / Interface]
**Текущий уровень:** [L0 / L1 / L2 / L3] ([название уровня])

### Stakeholder Questions

| Stakeholder | Вопрос | Статус |
|-------------|--------|--------|
| [роль] | [вопрос] | ✅ / ❌ [причина] |

### Testability (для Features)

| AC | Статус | Проблема |
|----|--------|----------|
| [ID] | ✅ / ❌ | [что не так] |

### Каскадная проверка

| Связанный документ | Тип связи | Статус |
|--------------------|-----------|--------|
| [путь] | xref / зависимость | ✅ / ⚠️ [проблема] |

**Требуют внимания:**
- [документ] — [причина]

### Найденные проблемы

1. **[Краткое описание]**
   Проблема: [детали]
   Решение: [как исправить]

### Рекомендация

**[STOP / ITERATE / WAIT]**

[Обоснование в 2-3 предложениях]

[Если ITERATE — список конкретных действий]
[Если WAIT — что нужно подождать]

### Каскадные действия (если есть)

После изменения этого документа проверить:
1. `[путь к связанному документу]` — [причина]
2. ...
```

## Режим папки: пакетная проверка

### Шаги выполнения (режим папки)

1. **Найти файлы:** `find [путь] -name "*.adoc" | sort`
2. **Для каждого файла:**
    - Выполнить полный анализ (как для одного файла)
    - Записать результат в сводку
3. **Показать сводный отчёт**

### Формат ответа (режим папки)

```markdown
## Пакетная проверка: [путь к папке]

**Найдено документов:** N

---

### 1. [имя файла 1]

**Тип:** Feature | **Уровень:** L2 | **Рекомендация:** STOP

<details>
<summary>Подробности</summary>

[Полный отчёт по файлу — stakeholder questions, testability, проблемы]

</details>

---

### 2. [имя файла 2]

**Тип:** ADR | **Уровень:** L1 | **Рекомендация:** ITERATE

<details>
<summary>Подробности</summary>

[Полный отчёт по файлу]

</details>

---

## Сводка

| Документ | Тип | Уровень | Рекомендация | Проблем |
|----------|-----|---------|--------------|---------|
| F001-... | Feature | L2 | STOP | 0 |
| F002-... | Feature | L1 | ITERATE | 3 |
| ADR-001-... | ADR | L0 | ITERATE | 5 |

### Статистика

- **STOP:** N документов (готовы)
- **ITERATE:** N документов (требуют доработки)
- **WAIT:** N документов (ждут внешних данных)

### Приоритетные действия

1. **[документ]** — [что сделать]
2. **[документ]** — [что сделать]

### Каскадные проверки

После исправлений проверить связанные документы:

- `[путь]` ← зависит от [документ]
```

### Опции для пользователя (режим папки)

После показа сводки предложить:

- `детали [N]` — показать полный отчёт по документу N
- `iterate [N]` — перейти к итерации документа N
- `все детали` — показать все отчёты развёрнуто
