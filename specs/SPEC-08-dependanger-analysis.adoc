= SPEC-08: dependanger-analysis -- Расширенный анализ совместимости
:toc: left
:toclevels: 3
:sectnums:

[[overview]]
== Обзор

Модуль `dependanger-analysis` выполняет расширенный анализ совместимости, который ДОПОЛНЯЕТ базовый `CompatRulesProcessor` (order=66) из `dependanger-effective`.

Разделение ответственности:

* **`CompatRulesProcessor`** (order=66, core, non-optional) -- статические декларативные проверки: `JdkRequirement`, `MutualExclusion`, `VersionConstraint`.
Не требуют I/O, всегда включены.
См. xref:SPEC-03-dependanger-effective.adoc#spec-compat-rules[SPEC-03.12].
* **`CompatibilityCheckProcessor`** (order=110, feature, optional) -- расширенный анализ: `CustomRule` handlers через SPI, bytecode-анализ (будущее).
Может требовать I/O, SPI-зависимости, ASM.

Результаты обоих процессоров хранятся в `EffectiveMetadata.extensions` через `CompatibilityIssuesExtensionKey`.
`CompatibilityCheckProcessor` читает существующие issues (от `CompatRulesProcessor`) и добавляет свои.

Приоритет: **P3** (желательно).

[[spec-compatibility-check]]
== SPEC-08.1: CompatibilityCheckProcessor.process()

[[purpose-compat]]
=== Назначение

Выполнить расширенный анализ совместимости, который дополняет базовый `compat-rules` процессор (order=66).
Обрабатывает ТОЛЬКО типы проверок, которые НЕ покрыты core-процессором:

* **CustomRule** -- вызов пользовательских handler'ов через SPI (`CustomRuleHandler`)
* **Bytecode-анализ** (будущее) -- определение JDK class file version, бинарная совместимость

НЕ обрабатывает (уже покрыто `CompatRulesProcessor`):

* `JdkRequirement` -- статическая проверка числа JDK version
* `MutualExclusion` -- проверка наличия библиотек в map
* `VersionConstraint` -- сравнение строк версий

[[algorithm-compat]]
=== Алгоритм

[source,text]
----
1. Получить правила совместимости из context.originalMetadata.compatibility
2. Отфильтровать только CustomRule правила (остальные обработаны CompatRulesProcessor)
3. ЕСЛИ CustomRule правил нет -> вернуть metadata, INFO "No custom rules to process"

4. ДЛЯ КАЖДОГО CustomRule:
   a. Найти handler по ruleType через SPI:
      ServiceLoader: `META-INF/services/io.github.zenhelix.dependanger.effective.spi.CustomRuleHandler`
      handlers загружаются через `associateBy { it.ruleType }`, lookup по `handlers[rule.ruleId]`
   b. ЕСЛИ handler не найден:
      - WARNING "Custom rule handler not found for '{ruleId}'"
      - Продолжить к следующему правилу
   c. ЕСЛИ handler найден:
      - Вызвать `handler.evaluate(rule, metadata.libraries, context)` в try-catch (синхронный вызов, НЕ suspend)
      - ЕСЛИ exception -> ERROR "Custom rule '{ruleId}' failed: {message}", продолжить pipeline
      - ЕСЛИ результат содержит issues -> собрать CompatibilityIssue

5. Прочитать существующие issues из metadata (от CompatRulesProcessor):
   val existingIssues = metadata.compatibilityIssues

6. Объединить: allIssues = existingIssues + newIssues
7. Записать через withExtension(CompatibilityIssuesExtensionKey, allIssues)
8. Добавить diagnostics: INFO/WARNING/ERROR в зависимости от severity
----

=== Детальная реализация

==== CustomRuleHandler SPI

[source,kotlin]
----
// Определён в dependanger-effective/spi/
public interface CustomRuleHandler {
    /** Уникальный идентификатор типа правила, которое обрабатывает этот handler */
    public val ruleType: String

    /**
     * Выполнить проверку правила и вернуть найденные проблемы.
     * Метод синхронный (НЕ suspend).
     * @param rule Правило из DSL
     * @param libraries Текущие effective-библиотеки
     * @param context Контекст обработки
     * @return Список обнаруженных issues (пустой если проверка пройдена)
     */
    public fun evaluate(
        rule: CompatibilityRule.CustomRule,
        libraries: Map<String, EffectiveLibrary>,
        context: ProcessingContext,
    ): List<CompatibilityIssue>
}

Регистрация:

==== Основная реализация

[source,kotlin]
----
override suspend fun process(
    metadata: EffectiveMetadata,
    context: ProcessingContext,
): EffectiveMetadata {
    val rules = context.originalMetadata.compatibility
    val customRules = rules.filterIsInstance<CompatibilityRule.CustomRule>()

    if (customRules.isEmpty()) {
        return metadata.copy(
            diagnostics = metadata.diagnostics + Diagnostics.info(
                code = "COMPAT_NO_CUSTOM_RULES",
                message = "No custom compatibility rules to process",
                processorId = id,
            )
        )
    }

    // Загрузка handlers через SPI
    val handlers = ServiceLoader.load(CustomRuleHandler::class.java)
        .associateBy { it.ruleType }

    val newIssues = mutableListOf<CompatibilityIssue>()
    var diagnostics = metadata.diagnostics

    for (rule in customRules) {
        val handler = handlers[rule.ruleId]
        if (handler == null) {
            diagnostics = diagnostics + Diagnostics.warning(
                code = "COMPAT_CUSTOM_HANDLER_NOT_FOUND",
                message = "Custom rule handler not found for '${rule.ruleId}'",
                processorId = id,
            )
            continue
        }

        try {
            val issues = handler.evaluate(rule, metadata.libraries, context)
            newIssues.addAll(issues)

            for (issue in issues) {
                diagnostics = diagnostics + when (issue.severity) {
                    Severity.ERROR -> Diagnostics.error(
                        code = "COMPAT_CUSTOM_RULE",
                        message = "[${rule.ruleId}] ${issue.message}",
                        processorId = id,
                    )
                    Severity.WARNING -> Diagnostics.warning(
                        code = "COMPAT_CUSTOM_RULE",
                        message = "[${rule.ruleId}] ${issue.message}",
                        processorId = id,
                    )
                    Severity.INFO -> Diagnostics.info(
                        code = "COMPAT_CUSTOM_RULE",
                        message = "[${rule.ruleId}] ${issue.message}",
                        processorId = id,
                    )
                }
            }
        } catch (e: Exception) {
            diagnostics = diagnostics + Diagnostics.error(
                code = "COMPAT_CUSTOM_RULE_FAILED",
                message = "Custom rule '${rule.ruleId}' failed: ${e.message}",
                processorId = id,
            )
            newIssues.add(CompatibilityIssue(
                ruleId = rule.name,
                message = "Custom rule evaluation failed: ${e.message}",
                severity = Severity.ERROR,
                suggestion = "Check custom rule handler implementation for '${rule.ruleId}'",
            ))
        }
    }

    // Merge с существующими issues (от CompatRulesProcessor)
    val existingIssues = metadata.compatibilityIssues
    val allIssues = existingIssues + newIssues

    val result = metadata.withExtension(CompatibilityIssuesExtensionKey, allIssues)
    return result.copy(diagnostics = diagnostics)
}
----

[[bytecode-analysis]]
== Bytecode-анализ (будущее, опционально)

Модуль имеет зависимость на `org.ow2.asm:asm:9.7.1` для потенциального bytecode-анализа.
В текущей версии не используется.
Возможные будущие применения:

- Определение минимальной версии JDK по class file version
- Анализ бинарной совместимости библиотек
- Обнаружение split packages

При реализации bytecode-анализа он будет добавлен в `CompatibilityCheckProcessor` как дополнительный шаг после обработки `CustomRule`.

[[edge-cases]]
== Edge Cases и обработка ошибок

[cols="1,2,2"]
|===
| Ситуация | Поведение | Диагностический код

| Нет CustomRule правил | Процессор возвращает metadata с INFO | `COMPAT_NO_CUSTOM_RULES`
| Handler не найден для ruleId | WARNING, правило пропускается | `COMPAT_CUSTOM_HANDLER_NOT_FOUND`
| Exception в handler.evaluate() | ERROR, issue с severity=ERROR, pipeline продолжается | `COMPAT_CUSTOM_RULE_FAILED`
| Handler возвращает пустой список | Правило прошло проверку, без issues | --
| Существующие issues от CompatRulesProcessor | Merge: existingIssues + newIssues | --
| Несколько handlers с одним ruleType | ServiceLoader: последний загруженный побеждает (associateBy) | --
| Strict mode + ERROR severity | ERROR в diagnostics | --
|===

[[diagnostic-codes]]
== Диагностические коды

[cols="1,2,1"]
|===
| Код | Описание | Уровень

| `COMPAT_NO_CUSTOM_RULES` | Нет CustomRule правил для обработки | INFO
| `COMPAT_CUSTOM_RULE` | Результат выполнения custom rule | varies

[[testing]]
== Тестовые сценарии

[cols="1,3"]
|===
| Тест | Описание

| `no custom rules returns unchanged` | Нет CustomRule -> INFO, metadata без изменений (кроме diagnostics)
| `custom rule handler found and passes` | Handler возвращает emptyList() -> без issues
| `custom rule handler found and triggers` | Handler возвращает issues -> merge в extensions
| `custom rule handler not found` | ruleId без handler -> WARNING
| `custom rule handler throws exception` | handler.evaluate() бросает -> ERROR + issue
| `merge with existing issues` | CompatRulesProcessor issues сохраняются + новые добавлены
| `multiple custom rules` | Несколько CustomRule -> все обработаны последовательно
| `jdk/mutual/version rules ignored` | JdkRequirement/MutualExclusion/VersionConstraint не обрабатываются
| `strict mode with ERROR` | ERROR severity + strict -> diagnostics ERROR
|===
