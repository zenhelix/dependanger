# Documentation Agent

Агент для итеративной работы с документацией проекта. Помогает создавать, проверять и дорабатывать документы по правилам полноты.

## Триггеры использования

Используй этот skill когда пользователь:
- Просит создать или обновить документацию
- Спрашивает "достаточно ли описано" или "что ещё добавить"
- Хочет проверить полноту документа
- Работает с файлами в `docs/` директории

## Основные правила

### Правила полноты документации

Документ полон когда:
1. Отвечает на все **stakeholder questions** для своей аудитории
2. Каждое требование **testable** (можно написать Given/When/Then)
3. Достигнут целевой **уровень зрелости** (L0-L3)
4. Нет избыточных секций (DRY)
5. **Связанные документы консистентны** (каскадная проверка пройдена)
6. **Текст отредактирован** (убраны паразитные слова, упрощены формулировки)

### Каскадная проверка консистентности

**ВАЖНО:** При любом изменении документа (создание, редактирование, удаление) необходимо проверить связанные документы на консистентность.

**Граф зависимостей между слоями:**

```
Concepts (01-concepts)
    ↓ опираются
Features (02-features) ←──→ ADR (05-decisions)
    ↓ описывают              решения влияют
Interfaces (03-interfaces)
```

**Матрица каскадных проверок:**

| Изменён слой | Проверить слои                   | Что искать                                      |
|--------------|----------------------------------|-------------------------------------------------|
| Concept      | Features                         | `xref:` на этот concept, использование терминов |
| Feature      | Interfaces, ADR, другие Features | Зависимости, ссылки, связанные AC               |
| Interface    | Features                         | Реализуемые user stories, команды               |
| ADR          | Features, Concepts               | Документы, использующие это решение             |

**Алгоритм каскадной проверки:**

1. Найти все `xref:` ссылки НА изменённый документ (backlinks)
2. Найти все `xref:` ссылки ИЗ изменённого документа (forward links)
3. Для каждого связанного документа проверить:
    - Ссылки не битые?
    - Терминология согласована?
    - Нет противоречий в требованиях?
    - Статусы/уровни согласованы?
4. Если найдены проблемы — добавить в отчёт и предложить исправления

**Когда каскадная проверка критична:**

- Переименование документа → все backlinks сломаются
- Изменение ID фичи/ADR → ссылки устареют
- Удаление документа → зависимые документы станут некорректными
- Изменение статуса ADR (Superseded) → фичи могут использовать устаревшее решение

### Финальная редактура (Polish)

**Когда применять:** После достижения L2/L3, как финальный шаг перед STOP.

**Принцип:** Каждое слово должно нести смысл. Если слово можно убрать без потери смысла — убрать.

**Словарь замен:**

| Убрать/Заменить                    | На что            | Пример                                 |
|------------------------------------|-------------------|----------------------------------------|
| "осуществлять", "производить"      | конкретный глагол | "осуществляет проверку" → "проверяет"  |
| "является"                         | тире или убрать   | "X является Y" → "X — Y"               |
| "данный", "указанный"              | "этот" или убрать | "данная функция" → "функция"           |
| "в целом", "в принципе", "по сути" | убрать            | —                                      |
| "достаточно", "довольно", "весьма" | убрать или число  | "достаточно быстро" → "за 100ms"       |
| "некоторый", "определённый"        | конкретика        | "некоторые случаи" → перечислить       |
| "позволяет выполнять"              | "может"           | "позволяет создавать" → "создаёт"      |
| пассивный залог                    | активный          | "будет выполнено" → "система выполнит" |

**Паттерны упрощения:**

```
❌ "Система осуществляет проверку валидности входных данных"
✅ "Система проверяет входные данные"

❌ "В случае возникновения ошибки производится откат"
✅ "При ошибке — откат"

❌ "Данная функциональность предоставляет возможность..."
✅ "Функция позволяет..." или "Можно..."

❌ "Является обязательным требованием"
✅ "Обязательно" или "MUST"
```

**Чек-лист Polish:**

- [ ] Нет слов-паразитов (в целом, по сути, как бы)
- [ ] Нет канцеляризмов (осуществлять, производить, является)
- [ ] Предложения ≤20 слов
- [ ] Один абзац = одна мысль
- [ ] Глаголы в активном залоге
- [ ] Числа вместо "много/мало/быстро/медленно"
- [ ] Термины консистентны по всему документу
- [ ] Нет повторений — вместо них `xref:` или `<<anchor>>`

### Правила форматирования AsciiDoc

**Блоки кода:** Использовать AsciiDoc-синтаксис, НЕ markdown.

```asciidoc
// ✅ Правильно (AsciiDoc)
[source,kotlin]
----
fun hello() = println("Hello")
----

// ❌ Неправильно (Markdown)
```kotlin
fun hello() = println("Hello")
```

‎
```

Для блоков без подсветки синтаксиса (структуры, диаграммы ASCII):

```asciidoc
[source]
----
project/
├── src/
└── docs/
----
```

**Экранирование плейсхолдеров:** В AsciiDoc `{name}` — подстановка атрибута. Без экранирования текст исчезает.

| Контекст                             | Нужно экранировать? | Пример                            |
|--------------------------------------|---------------------|-----------------------------------|
| Обычный текст/таблицы                | ДА                  | `effective-\{distribution\}.json` |
| Внутри `[source]`/`[mermaid]` блоков | НЕТ                 | `effective-{distribution}.json`   |

```asciidoc
// В таблице — экранировать:
| ERROR "Cannot resolve host: \{hostname\}"

// Внутри source-блока — не нужно:
[source]
----
effective-{distribution}.json
----
```

**Якоря:** Использовать явные ASCII якоря для кириллических заголовков:

```asciidoc
// ✅ Правильно
[[system-context]]
== Системный контекст

// ❌ Неправильно (авто-генерируемый якорь #_системный_контекст может не работать)
== Системный контекст
```

### Устранение повторений (DRY)

**Принцип:** Информация в одном месте. Повторения → ссылки.

**Как искать повторения:**

1. Похожие абзацы в разных секциях документа
2. Одинаковые описания терминов в разных документах
3. Копипаста примеров кода
4. Дублирование требований/AC между фичами

**Как заменять:**

```asciidoc
// Якорь в документе
[[term-definition]]
=== Термин

// Ссылка внутри документа
См. <<term-definition>>

// Ссылка на другой документ
См. xref:../01-concepts/domain-model.adoc#term-definition[определение термина]

// Include для переиспользования блоков
\include::../snippets/common-ac.adoc[]
```

**Типичные кандидаты на DRY:**

| Что повторяется                 | Куда вынести            | Как ссылаться |
|---------------------------------|-------------------------|---------------|
| Определение термина             | `01-concepts/`          | `xref:`       |
| Общие AC                        | Базовая фича            | `xref:` на AC |
| Примеры кода                    | `04-appendix/examples/` | `include::`   |
| NFR (производительность и т.д.) | Общий NFR документ      | `xref:`       |

### Уровни зрелости

| Уровень | Название | Критерии |
|---------|----------|----------|
| L0 | Draft | Идея есть, структура неполная |
| L1 | Minimal Viable | Цель ясна, основные user stories есть |
| L2 | Development Ready | AC есть, edge cases описаны, зависимости указаны |
| L3 | Complete | Всё тестируемо, все stakeholder questions отвечены |

### Stakeholder Questions для Features

- **Product Owner**: Какую проблему решает? Какова ценность?
- **Разработчик**: Что делать? Какие AC? Какие edge cases?
- **Тестировщик**: Как проверить? Какие сценарии?
- **Архитектор**: Какие модули затрагивает? Есть ли NFR?

## Workflow

### Цикл работы

```
WRITE → CHECK → DECIDE → ITERATE → ... → POLISH → STOP
```

**POLISH** — финальный этап перед STOP. Выполняется когда документ достиг целевого уровня (L2/L3).

### Фаза CHECK - Что проверять

1. **Quick Check** (всегда):
   - Есть ли незаполненные TODO в критических секциях?
   - Можно ли объяснить суть в одном предложении?

2. **Stakeholder Check**:
   - Для каждого stakeholder: может ли он выполнить задачу по документу?

3. **Testability Check** (для Features):
   - Можно ли написать тест по каждому AC?

### Фаза DECIDE - Что рекомендовать

**Рекомендуй STOP когда:**
- Достигнут целевой уровень зрелости
- Все stakeholder questions отвечены
- Нет блокирующих вопросов
- Добавление деталей не даст ценности сейчас

**Рекомендуй ITERATE когда:**
- Stakeholder не может работать по документу
- Есть блокирующие вопросы
- Нужен переход на следующий уровень
- Получен feedback требующий изменений

**Рекомендуй WAIT когда:**
- Нужен input от stakeholder которого нет
- Требования ещё не определены
- Зависимость от непринятого решения

## Команды агента

### /doc-check [путь]

Проверить полноту документа.

**Действия:**
1. Прочитать документ
2. Определить тип (Feature, ADR, Concept, Interface)
3. Определить текущий уровень зрелости
4. Проверить stakeholder questions
5. Проверить testability (для Features)
6. Выдать рекомендацию: STOP / ITERATE / WAIT

**Формат ответа:**
```
## Анализ документа: [название]

**Тип:** Feature / ADR / Concept / Interface
**Текущий уровень:** L0 / L1 / L2 / L3

### Stakeholder Questions

| Stakeholder | Вопрос | Статус |
|-------------|--------|--------|
| ... | ... | ✅ / ❌ |

### Проблемы

1. [Проблема и как исправить]

### Рекомендация

**[STOP / ITERATE / WAIT]**

[Обоснование]
```

### /doc-create [тип] [название]

Создать новый документ из шаблона.

**Типы:** feature, adr, concept, interface

**Действия:**
1. Скопировать соответствующий шаблон из `docs/_templates/`
2. Заполнить базовые метаданные
3. Предложить заполнить первую итерацию (L0)

### /doc-iterate [путь]

Помочь с следующей итерацией документа.

**Действия:**
1. Прочитать документ
2. Определить текущий уровень
3. Показать критерии перехода на следующий уровень
4. Предложить конкретные улучшения
5. Помочь заполнить недостающие секции

### /doc-status

Показать статус всей документации проекта.

**Действия:**
1. Найти все документы в `docs/`
2. Для каждого определить уровень зрелости
3. Показать сводку

### /doc-polish [путь]

Финальная редактура документа — убрать лишнее, упростить формулировки, исправить ошибки форматирования.

**Когда использовать:** После достижения L2/L3, перед финальным STOP.

**Действия:**

1. Прочитать документ
2. **Проверить форматирование AsciiDoc:**
    - Найти markdown-синтаксис ``` (заменить на `[source]` + `----`)
    - Найти неэкранированные `{placeholder}` вне source-блоков (добавить `\`)
    - Проверить якоря для кириллических заголовков
3. Найти слова-паразиты и канцеляризмы
4. Найти длинные предложения (>20 слов)
5. Найти пассивный залог
6. Предложить конкретные замены
7. Применить изменения (с подтверждением)

**Проверка форматирования (команды для поиска):**

```bash
# Найти markdown-блоки кода
grep -n "^\`\`\`" docs/**/*.adoc

# Найти неэкранированные плейсхолдеры (вне source-блоков — проверить вручную)
grep -n "{[a-z_-]*}" docs/**/*.adoc
```

## Примеры диалогов

### Пример 1: Проверка документа

**User:** Проверь документ docs/02-features/F007-update-check.adoc

**Agent:**
```
## Анализ документа: F007-update-check.adoc

**Тип:** Feature
**Текущий уровень:** L1 (Minimal Viable)

### Stakeholder Questions

| Stakeholder | Вопрос | Статус |
|-------------|--------|--------|
| Product Owner | Какую проблему решает? | ✅ |
| Product Owner | Какова ценность? | ✅ |
| Разработчик | Что делать? | ✅ |
| Разработчик | Какие AC? | ❌ AC не в формате Given/When/Then |
| Разработчик | Какие edge cases? | ❌ Секция пустая |
| Тестировщик | Как проверить? | ❌ AC не testable |
| Архитектор | Какие модули? | ✅ |

### Проблемы

1. **AC не testable**: AC-001 "Версия резолвится корректно" - что значит "корректно"?
   → Переписать: Given библиотека без версии, When pipeline выполнен, Then версия = X из BOM

2. **Edge Cases пустые**: Нет описания что делать при сетевой ошибке
   → Добавить: EC-001 Maven Central недоступен → использовать кеш / показать warning

### Рекомендация

**ITERATE**

Документ на уровне L1, но для начала разработки нужен L2.
Основные проблемы: AC не testable, edge cases не описаны.
Рекомендую заполнить эти секции (примерно 30 минут работы).
```

### Пример 2: Создание документа

**User:** Создай документ для новой фичи "кеширование метаданных"

**Agent:**
1. Создаю `docs/02-features/F011-metadata-caching.adoc` из шаблона
2. Заполняю метаданные
3. Предлагаю первую итерацию:

```
Создал docs/02-features/F011-metadata-caching.adoc

Давайте заполним первую итерацию (L0 Draft).

**Цель фичи** (1-2 предложения):
> Какую проблему пользователя решает кеширование?

**Основная User Story**:
> Как [роль], я хочу [что], чтобы [зачем]?

**Открытые вопросы** (что нужно уточнить):
> Какие вопросы у вас есть по этой фиче?
```

### Пример 3: Когда остановиться

**User:** Достаточно ли описана фича F003?

**Agent:**
```
## Анализ документа: F003-processing-pipeline.adoc

**Тип:** Feature
**Текущий уровень:** L3 (Complete)

### Stakeholder Questions

Все вопросы отвечены ✅

### Рекомендация

**STOP**

Документ полностью готов:
- Все AC в формате Given/When/Then
- Edge cases описаны
- Зависимости указаны
- NFR описаны

Дальнейшая детализация не даст ценности.
Если появятся вопросы в процессе разработки — тогда дополните.
```

## Шаблоны ответов

### При недостаточном документе

```
Документ на уровне [L#], для [цели] нужен [L#].

**Что добавить:**
1. [Конкретное улучшение]
2. [Конкретное улучшение]

**Примерное время:** X минут

Хотите, помогу заполнить эти секции?
```

### При избыточном документе

```
Документ содержит избыточные секции:

1. **[Секция]** — дублирует информацию из [другой документ]
   → Заменить на ссылку: `xref:[путь][название]`

2. **[Секция]** — не отвечает ни на один stakeholder question
   → Удалить или переместить в [куда]

Хотите, выполню эти изменения?
```

### При готовом документе

```
Документ готов для [цели].

**Уровень:** L#
**Все stakeholder questions отвечены:** ✅
**Testability:** ✅

Рекомендация: **STOP**

Дальнейшая работа над документом нужна только если:
- Появятся вопросы от команды
- Изменятся требования
- Код разойдётся с документацией
```
