[[conventions]]
== Конвенции

- **Формат:** AsciiDoc
- **Язык:** Русский
- **explicitApi():** Все публичные API с `public`
- **@Serializable:** Все модели данных
- **Sealed class иерархии:** Через `@Serializable` на родителе и подтипах
- **Coroutines:** Процессоры — `suspend fun`, CLI — `runBlocking`

=== Доступ к extensions в EffectiveMetadata

Канонический паттерн доступа к типизированным extensions:

[source,kotlin]
----
// Запись (в процессорах):
metadata.withExtension(UpdatesExtensionKey, updates)

// Чтение — через extension properties:
metadata.updates               // List<UpdateAvailableInfo>
metadata.vulnerabilities       // List<VulnerabilityInfo>
metadata.licenseViolations     // List<LicenseViolation>
metadata.compatibilityIssues   // List<CompatibilityIssue>
metadata.transitives           // List<TransitiveTree>
metadata.flatDependencies      // List<FlatDependency>
metadata.versionConflicts      // List<VersionConflict>

// Чтение — через getExtension (nullable):
metadata.getExtension(UpdatesExtensionKey) // List<UpdateAvailableInfo>?
----

Extension keys и extension properties определены в каждом feature-модуле в соответствующем `*Extensions.kt` файле.
Все extension-модели ДОЛЖНЫ быть `@Serializable` для поддержки `EffectiveJsonFormat`.

=== Сериализация EffectiveMetadata

`EffectiveJsonFormat` (определён в xref:SPEC-02-dependanger-metadata-json.adoc#spec-effective-json[SPEC-02.5]) обеспечивает сериализацию `EffectiveMetadata` вместе с extensions.
Используется в:

* SPEC-14 (CLI) — сохранение результатов `process` команды
* SPEC-15 (Gradle Plugin) — задача `dependangerGenerateEffective`
* SPEC-09 (Report) — включение данных в отчёт

=== Дефолты ValidationSettings

[cols="1,1"]
|===
| Настройка | Дефолт

| `onCompatibilityViolation` | `FAIL`
| `onDeprecatedLibrary` | `WARN`
| `onJdkMismatch` | `FAIL`
| `onUnresolvedVersion` | `WARN`
|===

В `STRICT` preset все значения переопределяются на `FAIL`.

=== Дефолты настроек feature-модулей

Все feature-модули (SPEC-06..12) по умолчанию **отключены** (`enabled = false`).
Включение происходит через:

* DSL: `settings { updateCheck { enabled = true } }`
* CLI: флаги конкретных команд (e.g. `check-updates`)
* API: `DependangerBuilder.withUpdateCheck()`, `.withSecurityCheck()`, etc.
* Gradle Plugin: соответствующие задачи (e.g. `dependangerCheckUpdates`)

[[processing-preset-definitions]]
=== Определения ProcessingPreset

`ProcessingPreset.DEFAULT`:

* Все 12 core-процессоров (order 5-66) включены
* Все feature-процессоры отключены (enabled = false)
* ValidationSettings: стандартные дефолты

`ProcessingPreset.STRICT`:

* Все 12 core-процессоров включены
* Все feature-процессоры включены (enabled = true)
* ValidationSettings: все значения = FAIL

`ProcessingPreset.MINIMAL`:

* 5 core-процессоров: ProfileProcessor(5), MetadataConversionProcessor(10), ExtractedVersionsProcessor(12), VersionFallbackProcessor(25), VersionResolverProcessor(30)
* 7 отключены: LibraryFilterProcessor(20), BundleFilterProcessor(40), PluginFilterProcessor(45), PluginProcessor(50), UsedVersionsProcessor(60), ValidationProcessor(65), CompatRulesProcessor(66)
* Все feature-процессоры отключены

`ProcessingPreset.DISTRIBUTION`:

* Все 12 core-процессоров включены (аналогично DEFAULT)
* Все feature-процессоры отключены (аналогично DEFAULT)
* Требует указания `distribution` -- ProfileProcessor применяет фильтрацию библиотек/плагинов по дистрибуции
* Предназначен для CI/CD сценариев, где каталог собирается под конкретную платформу/профиль

[[extension-serializer-spi]]
=== Extension Serializer SPI

Для сериализации `EffectiveMetadata.extensions` используется ServiceLoader-based подход через интерфейс `ExtensionSerializerProvider` (определён в `dependanger-effective`, пакет `spi`):

[source,kotlin]
----
public interface ExtensionSerializerProvider {
    public fun knownKeys(): Map<String, ExtensionKey<*>>
}
----

* Каждый feature-модуль предоставляет провайдер для своих extension-типов
* Регистрация через `META-INF/services/io.github.zenhelix.dependanger.effective.spi.ExtensionSerializerProvider`
* `EffectiveJsonFormat` загружает все провайдеры через ServiceLoader, собирает `knownKeys`, и использует `ExtensionKey.serializer` при сериализации/десериализации (см. xref:SPEC-02-dependanger-metadata-json.adoc#spec-effective-json[SPEC-02.5])
* Неизвестные extensions сохраняются как `JsonNull` с warning

[[repository-convention]]
=== Конвенция конфигурации репозиториев

Репозитории описываются через `sealed interface Repository` (пакет `io.github.zenhelix.dependanger.core.model`).
`MavenRepository` — основная реализация для Maven-репозиториев.
Тип `List<Repository>` используется во всех настройках.

Двухуровневая конфигурация repositories: глобальная `Settings.repositories` + override в feature-settings.

* `Settings.repositories: List<Repository>` — глобальный список репозиториев по умолчанию для всех фич
* Feature-settings (`UpdateCheckSettings.repositories`, `TransitiveResolutionSettings.repositories`) — переопределение для конкретной фичи
* xref:SPEC-06-dependanger-maven-resolver.adoc[SPEC-06], xref:SPEC-07-dependanger-updates.adoc[SPEC-07], xref:SPEC-10-dependanger-security.adoc[SPEC-10], xref:SPEC-11-dependanger-license.adoc[SPEC-11], xref:SPEC-12-dependanger-transitive.adoc[SPEC-12] используют одну merge-логику
* Дефолт: Maven Central

Merge-логика (приоритет: feature → global → default):

[source,kotlin]
----
import io.github.zenhelix.dependanger.core.model.MavenRepository

// Пример для updateCheck (аналогично для остальных фич)
val repositories = featureSettings.repositories.ifEmpty {
    context.settings.repositories.ifEmpty {
        listOf(MavenRepository(url = "https://repo.maven.apache.org/maven2", name = "Maven Central"))
    }
}
----
