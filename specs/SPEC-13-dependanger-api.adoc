[[overview]]
== Обзор

Модуль `dependanger-api` является высокоуровневым фасадом для всех внешних потребителей.
Предоставляет единую точку входа `Dependanger` с builder-паттерном, скрывая внутреннюю сложность pipeline, процессоров и генераторов.

Модуль зависит от всех core-модулей (api), генераторов (api) и всех 7 feature-модулей (implementation).

[[spec-build]]
== SPEC-13.1: DependangerBuilder.build()

[[purpose-build]]
=== Назначение

Создать экземпляр `Dependanger` из накопленной конфигурации builder'а.
Это ключевая точка сборки -- здесь из DSL/metadata + настроек создаётся готовый к использованию фасад.

[[algorithm-build]]
=== Алгоритм

[source,text]
----
1. Получить DependangerMetadata:
   a. Если metadata != null -> использовать его
   b. Если dslBlock != null -> создать DependangerDsl, применить block, вызвать toMetadata()
   c. Иначе -> throw DependangerConfigurationException("No metadata or DSL block provided")
2. Сохранить preset (default: ProcessingPreset.DEFAULT)
3. Сохранить environment (default: ProcessingEnvironment())
4. Создать Dependanger(metadata, preset, environment)
----

NOTE: Дополнительные поля builder'а (`additionalProcessors`, `disabledProcessorIds`, `pipelineCustomizer`) сохраняются в Dependanger для использования при вызове `process()`/`validate()`.
Для этого нужно расширить приватный конструктор Dependanger.

[[implementation-build]]
=== Детальная реализация

[source,kotlin]
----
public fun build(): Dependanger {
    val resolvedMetadata = resolveMetadata()
    return Dependanger(
        metadata = resolvedMetadata,
        preset = preset,
        environment = environment,
        additionalProcessors = additionalProcessors.toList(),
        disabledProcessorIds = disabledProcessorIds.toSet(),
        pipelineCustomizer = pipelineCustomizer,
    )
}

private fun resolveMetadata(): DependangerMetadata {
    // Приоритет: explicit metadata > DSL block
    metadata?.let { return it }

    dslBlock?.let { block ->
        val dsl = DependangerDsl()
        dsl.block()
        return dsl.toMetadata()
    }

    throw DependangerConfigurationException(
        "Cannot build Dependanger: neither metadata nor DSL block provided. " +
        "Use Dependanger.fromMetadata() or Dependanger.fromDsl() to create a builder."
    )
}
----

=== Изменения в Dependanger

Приватный конструктор необходимо расширить для хранения pipeline-конфигурации:

[source,kotlin]
----
public class Dependanger private constructor(
    private val metadata: DependangerMetadata,
    private val preset: ProcessingPreset,
    private val environment: ProcessingEnvironment,
    private val additionalProcessors: List<EffectiveMetadataProcessor> = emptyList(),
    private val disabledProcessorIds: Set<String> = emptySet(),
    private val pipelineCustomizer: (PipelineBuilder.() -> Unit)? = null,
) {
    // ...
}
----

=== Edge cases

[cols="1,2,2"]
|===
| Ситуация | Поведение | Обоснование

| Ни metadata, ни dslBlock не задан | `DependangerConfigurationException` | Fail-fast при некорректной конфигурации
| Заданы оба: metadata и dslBlock | metadata имеет приоритет, dslBlock игнорируется | Explicit > implicit
| DSL.toMetadata() выбрасывает исключение | Пробрасывается наверх как есть | Ошибки DSL должны быть видны пользователю
| Пустой список additionalProcessors | Пустой список передаётся в Dependanger | Допустимый сценарий
|===

[[spec-process]]
== SPEC-13.2: Dependanger.process()

[[purpose-process]]
=== Назначение

Запустить полный pipeline обработки metadata: построить PipelineBuilder, сконфигурировать его по preset, добавить/удалить процессоры, выполнить pipeline, обернуть результат в `DependangerResult`.

[[algorithm-process]]
=== Алгоритм

[source,text]
----
1. Создать PipelineBuilder
2. Зарегистрировать все встроенные процессоры через ServiceLoader или явный список
3. Добавить additionalProcessors
4. Применить preset-конфигурацию (preset.configure(builder))
5. Для каждого disabledProcessorId -> builder.disable(id)
6. Применить pipelineCustomizer, если задан
7. Построить pipeline (builder.build())
8. Создать ProcessingContext(originalMetadata, settings, environment, distribution, callback)
9. Выполнить pipeline.process(context) -> EffectiveMetadata
10. Обернуть в DependangerResult(effective, effective.diagnostics)
11. При исключении -> обернуть в DependangerResult(null, diagnosticsWithError)
----

[[implementation-process]]
=== Детальная реализация

[source,kotlin]
----
public suspend fun process(
    distribution: String? = null,
    callback: ProcessingCallback? = null,
): DependangerResult {
    return try {
        val pipeline = buildPipeline()
        val context = ProcessingContext(
            originalMetadata = metadata,
            settings = metadata.settings,
            environment = environment,
            activeDistribution = distribution,
            callback = callback,
        )

        val effective = pipeline.process(context)

        DependangerResult(
            effective = effective,
            diagnostics = effective.diagnostics,
        )
    } catch (e: DependangerException) {
        throw e
    } catch (e: PipelineConfigurationException) {
        throw DependangerConfigurationException(
            "Pipeline configuration error: ${e.message}", e
        )
    } catch (e: Exception) {
        throw DependangerProcessingException(
            "Processing failed: ${e.message}",
            phase = null,
            cause = e,
        )
    }
}

private fun buildPipeline(): ProcessingPipeline {
    val builder = PipelineBuilder()

    // Регистрация встроенных процессоров через ServiceLoader
    val builtinProcessors = ServiceLoader.load(EffectiveMetadataProcessor::class.java)
    builder.addProcessors(builtinProcessors.toList())

    // Добавление пользовательских процессоров
    builder.addProcessors(additionalProcessors)

    // Конфигурация по preset
    preset.configure(builder)

    // Отключение указанных процессоров
    disabledProcessorIds.forEach { id -> builder.disable(id) }

    // Пользовательская кастомизация pipeline
    pipelineCustomizer?.invoke(builder)

    return builder.build()
}
----

[[preset-configure-behavior]]
=== Поведение ProcessingPreset.configure()

Метод `preset.configure(builder)` определяет, какие процессоры будут включены/отключены в pipeline.
Подробнее см. xref:index.adoc#processing-preset-definitions[определения ProcessingPreset].

[cols="1,3"]
|===
| Preset | Поведение `configure()`

| `DEFAULT`
| Включить все core-процессоры, отключить все feature-процессоры

| `STRICT`
| Включить все core-процессоры + все feature-процессоры, `validationSettings.all = FAIL`

| `MINIMAL`
| Включить только 5 core-процессоров: `profile`, `metadata-conversion`, `extracted-versions`, `version-fallback`, `version-resolver`
|===

=== Регистрация встроенных процессоров

Встроенные процессоры загружаются через `java.util.ServiceLoader`.
Каждый feature-модуль регистрирует свои процессоры через файл `META-INF/services/io.github.zenhelix.dependanger.effective.pipeline.EffectiveMetadataProcessor`.

Альтернативный подход -- явная регистрация в `buildPipeline()`:

[source,kotlin]
----
private fun buildPipeline(): ProcessingPipeline {
    val builder = PipelineBuilder()

    // Явная регистрация встроенных процессоров из feature-модулей
    // Каждый feature-модуль предоставляет свой основной процессор
    builder.addProcessors(discoverBuiltinProcessors())

    // ... остальная конфигурация
    return builder.build()
}

private fun discoverBuiltinProcessors(): List<EffectiveMetadataProcessor> {
    return ServiceLoader.load(EffectiveMetadataProcessor::class.java).toList()
}
----

=== Edge cases

[cols="1,2,2"]
|===
| Ситуация | Поведение | Обоснование

| distribution = null | Обрабатываются все библиотеки без фильтрации | Поведение по умолчанию
| distribution указан, но не определён в metadata | ProcessingPipeline вернёт пустой EffectiveMetadata или ошибку в diagnostics | Валидация в pipeline
| Процессор выбрасывает исключение | Pipeline пробрасывает через PhaseError, Dependanger оборачивает в DependangerProcessingException | Единая стратегия обработки ошибок
| PipelineConfigurationException (дубликаты ID) | Оборачивается в DependangerConfigurationException | Ошибка конфигурации, не runtime
| Дублирующийся processorId между built-in и additional | PipelineBuilder.build() выбросит PipelineConfigurationException | Защита от конфликтов
| callback = null | Pipeline работает без callback'ов | Callback опционален
|===

[[spec-validate]]
== SPEC-13.3: Dependanger.validate()

[[purpose-validate]]
=== Назначение

Запустить только валидацию metadata без полной обработки.
Используется для быстрой проверки корректности конфигурации.

[[algorithm-validate]]
=== Алгоритм

[source,text]
----
1. Создать pipeline с preset = MINIMAL + явно включить ValidationProcessor
2. Вернуть DependangerResult с диагностикой
----

Однако, поскольку preset хранится в экземпляре Dependanger, validate() должен переопределить preset на MINIMAL:

[[implementation-validate]]
=== Детальная реализация

[source,kotlin]
----
public suspend fun validate(): DependangerResult {
    return try {
        val builder = PipelineBuilder()

        // Регистрация только базовых процессоров
        val builtinProcessors = ServiceLoader.load(EffectiveMetadataProcessor::class.java)
        builder.addProcessors(builtinProcessors.toList())

        // MINIMAL preset -- только базовые процессоры
        ProcessingPreset.MINIMAL.configure(builder)
        // Явно включаем валидацию (MINIMAL её отключает)
        builder.enableOptional("validation")

        val pipeline = builder.build()
        val context = ProcessingContext(
            originalMetadata = metadata,
            settings = metadata.settings,
            environment = environment,
            activeDistribution = null,
            callback = null,
        )

        val effective = pipeline.process(context)

        DependangerResult(
            effective = effective,
            diagnostics = effective.diagnostics,
        )
    } catch (e: Exception) {
        val diagnostics = Diagnostics(
            errors = listOf(
                DiagnosticMessage(
                    code = "VALIDATION_FAILED",
                    message = "Validation failed: ${e.message}",
                    severity = Severity.ERROR,
                )
            )
        )
        DependangerResult(effective = null, diagnostics = diagnostics)
    }
}
----

=== Отличие от process()

[cols="1,2,2"]
|===
| Аспект | `process()` | `validate()`

| Preset | Используется preset из builder'а | Всегда MINIMAL
| Distribution | Поддерживается | Не используется (null)
| Callback | Поддерживается | Не используется
| additionalProcessors | Добавляются | Не добавляются
| Цель | Полная обработка с генерацией effective | Только проверка корректности
| Обработка ошибок | Пробрасывает исключения | Оборачивает в DependangerResult с diagnostics
|===

[IMPORTANT]
====
`validate()` НЕ использует `additionalProcessors`, `disabledProcessorIds` и `pipelineCustomizer` из builder'а.
Всегда применяется `MINIMAL` preset.
Это намеренное поведение -- `validate()` проверяет базовую корректность метаданных.
Для кастомной валидации используйте `process()` с нужным preset.
====

[[spec-from-json]]
== SPEC-13.4: Dependanger.fromJson()

[[purpose-from-json]]
=== Назначение

Factory-метод для создания DependangerBuilder из JSON-строки metadata.
Удобен для CLI и других инструментов, работающих с JSON-файлами.

[[algorithm-from-json]]
=== Алгоритм

[source,text]
----
1. Создать экземпляр JsonSerializationFormat
2. Десериализовать JSON в DependangerMetadata через format.deserialize(json)
3. Передать metadata в конструктор DependangerBuilder
4. Вернуть builder для дальнейшей конфигурации
----

[[implementation-from-json]]
=== Детальная реализация

[source,kotlin]
----
public fun fromJson(json: String): DependangerBuilder {
    val format = JsonSerializationFormat()
    val metadata = try {
        format.deserialize(json)
    } catch (e: Exception) {
        throw DependangerConfigurationException(
            "Failed to parse metadata from JSON: ${e.message}", e
        )
    }
    return DependangerBuilder(metadata)
}
----

=== Edge cases

[cols="1,2,2"]
|===
| Ситуация | Поведение | Обоснование

| Пустая строка JSON | `DependangerConfigurationException` | Невалидный JSON
| Невалидный JSON | `DependangerConfigurationException` с cause = SerializationException | Оборачивание низкоуровневых ошибок
| JSON с неизвестными полями | Успешно десериализуется (ignoreUnknownKeys = true) | Forward-compatible
| JSON с отсутствующими обязательными полями | `DependangerConfigurationException` | Ошибка десериализации
| Валидный JSON, но не DependangerMetadata | `DependangerConfigurationException` | Структура не соответствует
|===

[[spec-extraction-methods]]
== SPEC-13.5: Extraction-методы DependangerResult

[[purpose-extraction]]
=== Назначение

Extension properties на `DependangerResult` для type-safe доступа к данным feature-процессоров.
Определены в отдельном файле `DependangerResultExtensions.kt`, аналогично `DependangerResultToml.kt` и `DependangerResultBom.kt`.

Именование свойств совпадает с extension properties на `EffectiveMetadata` (`metadata.updates` / `result.updates`), обеспечивая единообразный доступ на обоих уровнях абстракции.
Отличие: свойства на `DependangerResult` выполняют null-safe доступ к `effective` (`effective?.updates ?: emptyList()`).

Это позволяет CLI и Gradle plugin работать с результатами feature-процессоров через API фасад, **не зависев напрямую от feature-модулей**.
API module видит feature-модули (они `implementation` зависимости), а CLI/plugin видит только API.

[[implementation-extraction]]
=== Детальная реализация

[source,kotlin]
----
// DependangerResultExtensions.kt
package io.github.zenhelix.dependanger.api

import io.github.zenhelix.dependanger.effective.model.getExtension
import io.github.zenhelix.dependanger.features.updates.model.UpdateAvailableInfo
import io.github.zenhelix.dependanger.features.updates.model.UpdatesExtensionKey
import io.github.zenhelix.dependanger.features.security.model.VulnerabilityInfo
import io.github.zenhelix.dependanger.features.security.model.VulnerabilitiesExtensionKey
import io.github.zenhelix.dependanger.features.license.model.LicenseViolation
import io.github.zenhelix.dependanger.features.license.model.LicenseViolationsExtensionKey
import io.github.zenhelix.dependanger.features.transitive.model.TransitiveTree
import io.github.zenhelix.dependanger.features.transitive.model.TransitivesExtensionKey
import io.github.zenhelix.dependanger.effective.model.CompatibilityIssue
import io.github.zenhelix.dependanger.effective.model.CompatibilityIssuesExtensionKey
import io.github.zenhelix.dependanger.features.transitive.model.VersionConflict
import io.github.zenhelix.dependanger.features.transitive.model.VersionConflictsExtensionKey

public val DependangerResult.updates: List<UpdateAvailableInfo>
    get() = effective?.getExtension(UpdatesExtensionKey) ?: emptyList()

public val DependangerResult.vulnerabilities: List<VulnerabilityInfo>
    get() = effective?.getExtension(VulnerabilitiesExtensionKey) ?: emptyList()

public val DependangerResult.licenseViolations: List<LicenseViolation>
    get() = effective?.getExtension(LicenseViolationsExtensionKey) ?: emptyList()

public val DependangerResult.transitives: List<TransitiveTree>
    get() = effective?.getExtension(TransitivesExtensionKey) ?: emptyList()

public val DependangerResult.compatibilityIssues: List<CompatibilityIssue>
    get() = effective?.getExtension(CompatibilityIssuesExtensionKey) ?: emptyList()

public val DependangerResult.versionConflicts: List<VersionConflict>
    get() = effective?.getExtension(VersionConflictsExtensionKey) ?: emptyList()
----

=== Обоснование `effective?.` вместо `effective!!`

Properties используют safe-call `?.` вместо `!!`:

* Если `effective == null` (processing failed), возвращается `emptyList()` -- безопасный default
* Вызывающий код уже проверяет `result.isSuccess` перед extraction, но double-safety лучше crash
* Консистентно с подходом `isSuccess` -- результат всегда доступен, даже если пустой

=== Edge cases

[cols="1,2,2"]
|===
| Ситуация | Поведение | Обоснование

| effective = null (processing failed) | Возвращает `emptyList()` | Безопасный fallback
| Extension не заполнен процессором | Возвращает `emptyList()` | `getExtension()` возвращает null
| Процессор не был включён в pipeline | Возвращает `emptyList()` | Extension просто отсутствует
|===

[[thread-safety]]
== Потокобезопасность

[cols="1,2"]
|===
| Компонент | Гарантии

| `Dependanger` | **Thread-safe**. Все поля immutable (val). Методы `process()` и `validate()` stateless -- создают новый pipeline при каждом вызове. Можно безопасно вызывать из нескольких корутин.
| `DependangerBuilder` | **НЕ thread-safe**. Мутабельные поля. Использовать только в одном потоке/корутине. После `build()` результат thread-safe.
| `DependangerResult` | **Immutable**. Data class, безопасен для sharing.
| `ProcessingPipeline` | **Thread-safe**. Создаётся заново при каждом вызове `process()`.
| `ProcessingContext` | **Immutable**. Безопасен для sharing.
|===

[[error-handling]]
== Стратегия обработки ошибок

=== Иерархия исключений

[source,text]
----
DependangerException (base)
├── DependangerConfigurationException  -- ошибки сборки/конфигурации
│   ├── No metadata or DSL block
│   ├── JSON parse error
│   └── Pipeline configuration error (duplicate IDs)
├── DependangerProcessingException     -- ошибки в pipeline
│   ├── Processor failed (с phase)
│   └── Unknown processing error
├── DependangerValidationException     -- ошибки валидации
│   └── Validation errors list
└── DependangerGenerationException     -- ошибки генерации артефактов
    └── Generator failed (с generatorId)
----

=== Стратегия по методам

[cols="1,2"]
|===
| Метод | Стратегия

| `build()` | Выбрасывает `DependangerConfigurationException` при ошибках конфигурации
| `process()` | Выбрасывает `DependangerProcessingException` при необработанных ошибках pipeline. Ошибки валидации -- в diagnostics результата
| `validate()` | Никогда не выбрасывает (кроме fatal). Все ошибки -- в diagnostics результата
| `fromJson()` | Выбрасывает `DependangerConfigurationException` при невалидном JSON
| `generate()` / `writeTo()` (DependangerResult) | Выбрасывает `DependangerProcessingException` если effective = null, иначе делегирует генератору
|===

== Тестовые сценарии

=== Юнит-тесты для DependangerBuilder.build()

[cols="1,3"]
|===
| Тест | Описание

| `build from metadata` | DependangerBuilder(metadata).build() создаёт валидный Dependanger
| `build from DSL` | DependangerBuilder(dslBlock).build() вызывает toMetadata() и создаёт Dependanger
| `build without config throws` | DependangerBuilder без metadata и dslBlock -> DependangerConfigurationException
| `build with preset` | Preset сохраняется и используется в process()
| `build with environment` | Environment сохраняется и передаётся в ProcessingContext
| `build with additional processors` | Дополнительные процессоры добавляются в pipeline
| `build with disabled processors` | Отключённые процессоры исключаются из pipeline
| `build with pipeline customizer` | Customizer применяется к PipelineBuilder
| `metadata priority over DSL` | Если заданы оба -- metadata побеждает
|===

=== Юнит-тесты для Dependanger.process()

[cols="1,3"]
|===
| Тест | Описание

| `process with default preset` | Pipeline включает все mandatory процессоры
| `process with distribution` | Distribution передаётся в ProcessingContext
| `process with callback` | Callback получает события PhaseStarted/PhaseCompleted
| `process result has effective` | Успешная обработка возвращает DependangerResult с effective != null
| `process result has diagnostics` | Diagnostics из pipeline передаются в результат
| `process failure wraps exception` | Ошибка pipeline оборачивается в DependangerProcessingException
| `process pipeline config error` | PipelineConfigurationException -> DependangerConfigurationException
| `process with disabled processor` | Отключённый процессор не выполняется
| `process with additional processor` | Дополнительный процессор выполняется
| `process with STRICT preset` | Все optional процессоры включены
|===

=== Юнит-тесты для Dependanger.validate()

[cols="1,3"]
|===
| Тест | Описание

| `validate valid metadata` | isSuccess = true, diagnostics.errors пуст
| `validate invalid metadata` | isSuccess = false, diagnostics содержит ошибки
| `validate catches exceptions` | Исключения оборачиваются в diagnostics, не пробрасываются
| `validate uses MINIMAL preset` | Только базовые процессоры выполняются
| `validate ignores additional processors` | additionalProcessors из builder'а не используются

[cols="1,3"]
|===
| Тест | Описание
