# /doc-iterate

Помочь с следующей итерацией документа — перейти на следующий уровень зрелости.

## Использование

```
/doc-iterate [путь]               # файл или папка
/doc-iterate                      # итерировать активный документ
```

**Режимы:**

- **Файл:** итерация одного документа (интерактивно)
- **Папка:** последовательная итерация всех `.adoc` файлов, требующих доработки

## Примеры

```
/doc-iterate docs/02-features/F007-update-check.adoc   # один файл
/doc-iterate docs/02-features/                         # вся папка features
/doc-iterate docs/                                     # весь docs
```

## Инструкции для агента

$ARGUMENTS содержит путь к документу или папке. Если пустой — запросить путь.

### Определение режима

1. **Проверить, что указано в $ARGUMENTS:**
    - Если путь заканчивается на `.adoc` → режим одного файла
    - Если путь — директория → режим папки
    - Если пустой → запросить путь

2. **Режим папки:**
    - Сначала выполнить `/doc-check` на папке (внутренне, без вывода)
    - Отфильтровать документы с рекомендацией ITERATE
    - Отсортировать по приоритету (L0 → L1 → L2)
    - Показать план итераций
    - Обработать каждый файл по очереди (с подтверждением)

### Шаги выполнения

1. **Прочитать документ** и определить текущий уровень зрелости

2. **Определить целевой уровень** (следующий):
   - L0 → L1 (Minimal Viable)
   - L1 → L2 (Development Ready)
   - L2 → L3 (Complete)

3. **Показать критерии перехода:**

   **L0 → L1:**
   - [ ] Заполнена секция "Описание / Цель"
   - [ ] Есть хотя бы 1 User Story с приоритетом
   - [ ] Определены зависимости от других фич

   **L1 → L2:**
   - [ ] Все User Stories имеют Acceptance Criteria
   - [ ] AC записаны в формате Given/When/Then
   - [ ] Описаны Edge Cases и Error Scenarios
   - [ ] Указаны функциональные требования (FR)

   **L2 → L3:**
   - [ ] Все FR помечены как MUST/SHOULD/MAY
   - [ ] Каждое требование testable
   - [ ] Все stakeholder questions отвечены
   - [ ] Документ прошёл peer review

4. **Проверить текущее состояние** — что уже выполнено из критериев

5. **Предложить конкретные улучшения:**
   - Какие секции заполнить
   - Какие формулировки уточнить
   - Примеры как должно выглядеть

6. **Интерактивно помочь заполнить** недостающее:
   - Задавать вопросы
   - Предлагать формулировки
   - Редактировать документ

### Формат ответа

```markdown
## Итерация документа: [название]

**Текущий уровень:** L# ([название])
**Целевой уровень:** L# ([название])

### Критерии перехода L# → L#

| Критерий | Статус | Что сделать |
|----------|--------|-------------|
| [критерий] | ✅ / ❌ | [действие если ❌] |

### План итерации

**Шаг 1: [Название]**
[Что нужно сделать]

Пример:
```
[Пример как должно выглядеть]
```

**Шаг 2: [Название]**
...

---

Начнём с шага 1?
```

### Пример диалога

**User:** /doc-iterate docs/02-features/F007-update-check.adoc

**Agent:**
```markdown
## Итерация документа: F007-update-check.adoc

**Текущий уровень:** L1 (Minimal Viable)
**Целевой уровень:** L2 (Development Ready)

### Критерии перехода L1 → L2

| Критерий | Статус | Что сделать |
|----------|--------|-------------|
| Все US имеют AC | ❌ | US-002 без AC |
| AC в Given/When/Then | ❌ | Переписать AC-001, AC-002 |
| Edge Cases описаны | ❌ | Добавить 3-5 edge cases |
| FR указаны | ✅ | — |

### План итерации

**Шаг 1: Добавить AC для US-002**

US-002 описывает фильтрацию обновлений, но нет acceptance criteria.

Предлагаю такой AC:
```
| AC-002-1 | Включён флаг --major-only | Запустить update check | Показаны только major обновления |
| AC-002-2 | Библиотека в exclude списке | Запустить update check | Библиотека не проверяется |
```

**Шаг 2: Переписать AC-001 в Given/When/Then**

Текущий AC-001: "Версии резолвятся корректно"

Это не testable. Предлагаю:
```
| AC-001-1 | Library с versionRef("kotlin") | Pipeline выполнен | version = значение из versions["kotlin"] |
```

Начнём с шага 1?
```

### После каждого шага

- Обновить документ
- Проверить критерий
- Показать прогресс
- Перейти к следующему шагу или завершить

### После завершения итерации — Каскадная проверка

**ВАЖНО:** После изменения документа необходимо проверить связанные документы.

1. **Найти связанные документы:**
    - Поиск `xref:` на этот документ в других файлах
    - Проверить секцию "Зависимости" в этом документе
    - Определить документы того же слоя с пересекающейся функциональностью

2. **Для каждого связанного документа проверить:**
    - Ссылки актуальны?
    - Терминология согласована?
    - Нет противоречий в требованиях?

3. **Если найдены проблемы:**
    - Добавить в список "Каскадные действия"
    - Предложить запустить `/doc-check` на связанных документах

### Завершение итерации

```markdown
## Итерация завершена

**Новый уровень:** L2 (Development Ready)

### Что было сделано:
1. ✅ Добавлены AC для US-002
2. ✅ AC переписаны в Given/When/Then
3. ✅ Добавлены Edge Cases

### Следующие шаги:
- Документ готов для начала разработки
- При появлении вопросов — дополнить
- Для L3 нужен peer review

### Каскадные действия:

Проверить связанные документы на консистентность:
1. `docs/03-interfaces/cli.adoc` — использует команды из этой фичи
2. `docs/02-features/F003-processing-pipeline.adoc` — зависимость

Запустить: `/doc-check docs/03-interfaces/cli.adoc`

Рекомендация: **STOP** (достаточно для текущего этапа)
```

## Режим папки: пакетная итерация

### Шаги выполнения (режим папки)

1. **Анализ:** выполнить внутренний `/doc-check` на папке
2. **Фильтрация:** выбрать только документы с ITERATE
3. **Сортировка:** по уровню (L0 первые), затем по имени
4. **Показать план** и запросить подтверждение
5. **Итерировать по очереди** с паузой между документами

### Формат ответа (режим папки)

```markdown
## Пакетная итерация: [путь к папке]

### Анализ документов

| # | Документ | Уровень | Целевой | Проблем | Приоритет |
|---|----------|---------|---------|---------|-----------|
| 1 | F001-registry-dsl.adoc | L0 | L1 | 5 | Высокий |
| 2 | F003-pipeline.adoc | L1 | L2 | 3 | Средний |
| 3 | ADR-002-format.adoc | L1 | L2 | 2 | Средний |

**Пропущены (STOP/WAIT):**

- F002-parsing.adoc — L2, готов
- ADR-001-kotlinx.adoc — ждёт решения по формату

---

### План итераций

**Порядок обработки:**

1. `F001-registry-dsl.adoc` — поднять до L1 (базовый документ)
2. `F003-pipeline.adoc` — поднять до L2 (зависит от F001)
3. `ADR-002-format.adoc` — поднять до L2

**Оценка:** ~3 документа к обработке

---

Начать с документа 1 (`F001-registry-dsl.adoc`)?

Опции:

- `да` — начать с первого
- `[N]` — начать с документа N
- `все` — обработать все автоматически (без пауз)
- `отмена` — выйти
```

### Процесс итерации (режим папки)

После каждого документа:

```markdown
---

## ✅ Завершён: F001-registry-dsl.adoc (L0 → L1)

**Прогресс:** 1/3 документов

---

Следующий: `F003-pipeline.adoc` (L1 → L2)

Продолжить? (да / пропустить / завершить)
```

### Финальный отчёт (режим папки)

```markdown
## Пакетная итерация завершена

**Папка:** docs/02-features/

### Результаты

| Документ | Было | Стало | Статус |
|----------|------|-------|--------|
| F001-registry-dsl.adoc | L0 | L1 | ✅ Готов |
| F003-pipeline.adoc | L1 | L2 | ✅ Готов |
| ADR-002-format.adoc | L1 | L1 | ⏭️ Пропущен |

### Статистика

- **Обработано:** 2 документа
- **Пропущено:** 1 документ
- **Поднято уровней:** 2

### Каскадные действия

После изменений проверить:

1. `docs/03-interfaces/cli.adoc` — использует F001, F003
2. `docs/00-overview/index.adoc` — ссылается на features

### Следующие шаги

- Запустить `/doc-check docs/` для полной проверки
- Или `/doc-polish docs/02-features/` для финальной редактуры
```
