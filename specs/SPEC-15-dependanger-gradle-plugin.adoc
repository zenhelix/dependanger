[[overview]]
== Обзор

Модуль `dependanger-gradle-plugin` интегрирует Dependanger в Gradle build lifecycle.
Плагин предоставляет DSL-конфигурацию через `DependangerExtension` и 13 Gradle-задач для генерации, валидации и анализа зависимостей.

Plugin ID: `io.github.zenhelix.dependanger`

Зависимость: единственная зависимость -- `dependanger-api` (implementation), которая транзитивно тянет все необходимые модули.

[[common-pattern]]
== Общий паттерн задач

[[task-categories]]
=== Две категории задач

Задачи разделены на две категории с принципиально разной архитектурой:

**Категория 1: Генерирующие задачи (файловая цепочка)**

Задачи образуют цепочку через файловые inputs/outputs.
Pipeline выполняется **один раз** в `GenerateEffectiveTask`, результат записывается в `effective.json`.
Последующие задачи (`GenerateTomlTask`, `GenerateBomTask`, `ListVersionsTask`) **читают** `effective.json` и выполняют только генерацию артефактов без повторного запуска pipeline.

**Категория 2: Аналитические задачи (standalone)**

Каждая аналитическая задача самодостаточна: создаёт собственный pipeline с `ProcessingPreset.DEFAULT` и включает **только свой** feature-процессор через `configureProcessing { enableOptional("...") }`.
Не зависят от `GenerateEffectiveTask`.

[cols="1,2,1"]
|===
| Категория | Задачи | Pipeline

| Генерирующие
| GenerateMetadata, GenerateEffective, GenerateToml, GenerateBom, GenerateAll, ListVersions
| 1 раз (файловая цепочка)

| Аналитические
| CheckUpdates, Analyze, SecurityCheck, LicenseCheck, ResolveTransitives
| Свой pipeline + 1 feature

| Отчёт
| Report
| Свой pipeline (STRICT)

| Валидация
| Validate
| Свой pipeline (MINIMAL)
|===

=== Базовый шаблон задач

Все задачи следуют общему шаблону:

[source,text]
----
1. Получить DependangerExtension из project
2. В зависимости от категории:
   a. Генерирующие: читать effective.json (кроме GenerateEffectiveTask, который запускает pipeline)
   b. Аналитические: создать Dependanger с DEFAULT + enableOptional("feature-id")
3. Выполнить операцию (generate/analyze/validate)
4. Записать результат в extension.outputDirectory
5. Обработать ошибки с учётом extension.failOnError
----

=== Базовый код получения extension и metadata

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile
    val failOnError = extension.failOnError.get()

    outputDir.mkdirs()

    // ... специфичная логика задачи
}
----

=== runBlocking для suspend-функций

Все suspend-функции Dependanger API вызываются через `runBlocking`:

[source,kotlin]
----
import kotlinx.coroutines.runBlocking

val result = runBlocking {
    dependanger.process(distribution)
}
----

=== Обработка ошибок

[source,kotlin]
----
// Gradle-специфичная обработка ошибок
try {
    // ... операция
} catch (e: DependangerException) {
    if (failOnError) {
        throw GradleException("Dependanger: ${e.message}", e)
    } else {
        logger.warn("Dependanger: ${e.message}")
    }
}

=== Категория 1: Генерирующие задачи (файловая цепочка)

[source,text]
----

[source,text]

=== Регистрация зависимостей задач

[source,kotlin]
----
private fun registerTasks(project: Project, extension: DependangerExtension) {

        it.dependsOn("dependangerGenerateMetadata")

    project.tasks.register("dependangerGenerateToml", GenerateTomlTask::class.java) {
        it.dependsOn("dependangerGenerateEffective")

    project.tasks.register("dependangerGenerateBom", GenerateBomTask::class.java) {
        it.dependsOn("dependangerGenerateEffective")
    }
    project.tasks.register("dependangerGenerate", GenerateAllTask::class.java) {
        it.dependsOn("dependangerGenerateToml", "dependangerGenerateBom")
    }
    project.tasks.register("dependangerListVersions", ListVersionsTask::class.java) {
        it.dependsOn("dependangerGenerateEffective")

    // Категория 2: Аналитические задачи (standalone, без dependsOn)

    project.tasks.register("dependangerReport", ReportTask::class.java)

[[gradle-annotations]]
== Аннотации Gradle для кэширования

Для поддержки инкрементальных билдов и Gradle Build Cache задачи должны декларировать inputs и outputs.

=== Рекомендуемая структура задачи с аннотациями

[source,kotlin]
----
import org.gradle.api.DefaultTask
import org.gradle.api.file.DirectoryProperty
import org.gradle.api.file.RegularFileProperty
import org.gradle.api.provider.Property
import org.gradle.api.tasks.*

public abstract class GenerateMetadataTask : DefaultTask() {
    init {
        group = "dependanger"
        description = "Generate metadata.json from DSL"
    }

    @get:Internal
    public val extension: DependangerExtension
        get() = project.extensions.getByType(DependangerExtension::class.java)

    @get:OutputDirectory
    public val outputDirectory: DirectoryProperty
        get() = extension.outputDirectory

    @TaskAction
    public fun execute() {
        // реализация
    }
}
----

NOTE: DSL-конфигурация как `@Input` проблематична, т.к. `DependangerDsl` не `Serializable` для Gradle.
Для полноценного кэширования можно использовать `@Internal` для extension и объявить hashCode/fingerprint DSL через кастомный `@Input`:

[source,kotlin]
----
@get:Input
public val metadataFingerprint: String
    get() {
        val metadata = extension.dsl.toMetadata()
        return metadata.hashCode().toString()
    }
----

На первом этапе реализации можно использовать `@Internal` для extension и не кэшировать задачи.
Кэширование -- отдельная задача оптимизации.

=== Аннотации для генерирующих задач (файловая цепочка)

Генерирующие задачи, читающие `effective.json`, должны декларировать `@InputFile`:

[source,kotlin]
----
// Пример для GenerateTomlTask
public abstract class GenerateTomlTask : DefaultTask() {
    @get:Internal
    public val extension: DependangerExtension
        get() = project.extensions.getByType(DependangerExtension::class.java)

    @get:InputFile
    public val effectiveFile: RegularFileProperty
        get() = project.objects.fileProperty().fileValue(
            extension.outputDirectory.get().asFile.resolve("effective.json")
        )

    @get:OutputFile
    public val tomlFile: RegularFileProperty
        get() {
            val metadata = extension.dsl.toMetadata()
            val filename = metadata.settings?.toml?.filename ?: "libs.versions.toml"
            return project.objects.fileProperty().fileValue(
                extension.outputDirectory.get().asFile.resolve(filename)
            )
        }

    @TaskAction
    public fun execute() { /* ... */ }
}
----

NOTE: `@InputFile` на `effective.json` обеспечивает Gradle up-to-date check: если `effective.json` не изменился, TOML не перегенерируется.

[[dsl-fingerprinting]]
=== DSL fingerprinting: подход первой итерации

В первой итерации для up-to-date проверок используется простой hashCode от metadata.
Это не идеально для incremental builds (hashCode может давать коллизии, а Gradle предпочитает детерминированную сериализацию inputs), но работает корректно для определения изменений DSL-конфигурации.

[source,kotlin]
----
@Input
public fun getDslHashCode(): Int = dsl.toMetadata().hashCode()
----

NOTE: Это решение первой итерации.
В будущем для более надёжной поддержки incremental builds рекомендуется перейти на полноценный fingerprint -- например, сериализовать `DependangerMetadata` в JSON-строку и использовать её как `@Input`, что даст Gradle точную информацию об изменениях без коллизий hashCode.
Пометить как area для улучшения.

[[spec-generate-metadata]]
== SPEC-15.1: GenerateMetadataTask

=== Назначение

Конвертировать DSL-конфигурацию из Gradle build скрипта в сериализованный `metadata.json`.

=== Алгоритм

[source,text]
----
1. Получить extension из project
2. Вызвать extension.dsl.toMetadata() -> DependangerMetadata
3. Создать JsonSerializationFormat
4. Записать metadata в outputDir/metadata.json
5. Логировать результат
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile

    outputDir.mkdirs()

    val format = JsonSerializationFormat()
    val outputFile = outputDir.resolve("metadata.json").toPath()
    format.write(metadata, outputFile)

    logger.lifecycle("Dependanger: Generated metadata.json -> $outputFile")
    logger.lifecycle("  Versions: ${metadata.versions.size}")
    logger.lifecycle("  Libraries: ${metadata.libraries.size}")
    logger.lifecycle("  Plugins: ${metadata.plugins.size}")
    logger.lifecycle("  Bundles: ${metadata.bundles.size}")
}
----

=== Edge cases

[cols="1,2"]
|===
| Ситуация | Поведение

| Пустой DSL | Создаёт metadata.json с пустыми списками (валидный сценарий)
| DSL с ошибками (невалидные координаты) | Ошибка в toMetadata(), оборачивается в GradleException
| Output-директория не существует | Создаётся через mkdirs()
|===

[[spec-generate-effective]]
== SPEC-15.2: GenerateEffectiveTask

=== Назначение

Обработать metadata через pipeline и создать effective-metadata с разрешёнными версиями, применёнными фильтрами и собранной диагностикой.

=== Алгоритм

[source,text]
----
1. Получить extension
2. Построить metadata из DSL
3. Создать Dependanger через fromMetadata(metadata)
4. Вызвать process() через runBlocking
5. Сериализовать effective metadata в outputDir/effective.json
6. Вывести диагностику (warnings/errors)
7. Если failOnError && diagnostics.hasErrors -> throw GradleException
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val outputDir = extension.outputDirectory.get().asFile
    val failOnError = extension.failOnError.get()

    outputDir.mkdirs()

    val dependanger = Dependanger.fromMetadata(metadata).build()
    val result = runBlocking { dependanger.process() }

    // Вывод диагностики
    result.diagnostics.errors.forEach { logger.error("Dependanger ERROR: ${it.message}") }
    result.diagnostics.warnings.forEach { logger.warn("Dependanger WARN: ${it.message}") }
    result.diagnostics.infos.forEach { logger.info("Dependanger INFO: ${it.message}") }

    if (result.diagnostics.hasErrors && failOnError) {
        throw GradleException(
            "Dependanger processing failed with ${result.diagnostics.errors.size} error(s). " +
            "Set dependanger { failOnError = false } to continue despite errors."
        )
    }

    if (result.effective != null) {
        // Сериализация effective metadata
        // TODO: Зависит от наличия EffectiveJsonFormat или аналога
        val outputFile = outputDir.resolve("effective.json")
        // effectiveFormat.write(result.effective, outputFile.toPath())
        logger.lifecycle("Dependanger: Generated effective.json -> $outputFile")
    }
}
----

[[spec-generate-toml]]
== SPEC-15.3: GenerateTomlTask

=== Назначение

Сгенерировать TOML Version Catalog из effective metadata.
**Не запускает pipeline** -- читает `effective.json`, записанный `GenerateEffectiveTask`.

=== Алгоритм

[source,text]
----
1. Получить extension, outputDir
2. Прочитать effective.json из outputDir (записан GenerateEffectiveTask)
3. Десериализовать EffectiveMetadata через EffectiveJsonFormat
4. Создать DependangerResult из EffectiveMetadata
5. Извлечь TomlConfig из settings metadata (DSL)
6. Вызвать result.writeTomlTo(path, config)
----

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val outputDir = extension.outputDirectory.get().asFile

    // Читаем effective.json, записанный GenerateEffectiveTask
    val effective = DependangerTaskHelper.readEffective(outputDir)

    val result = DependangerResult(
        effective = effective,
        diagnostics = effective.diagnostics,
    )

    // TomlConfig из DSL settings
    val metadata = extension.dsl.toMetadata()
    val tomlSettings = metadata.settings?.toml
    val tomlConfig = TomlConfig(
        includeComments = tomlSettings?.includeComments ?: true,
        sortSections = tomlSettings?.sortSections ?: true,
    )

    val tomlFilename = tomlSettings?.filename ?: "libs.versions.toml"
    val tomlPath = outputDir.resolve(tomlFilename).toPath()
    result.writeTomlTo(tomlPath, tomlConfig)

    logger.lifecycle("Dependanger: Generated $tomlFilename -> $tomlPath")
}
----

NOTE: Pipeline НЕ запускается.
`GenerateTomlTask` зависит от `GenerateEffectiveTask` через Gradle `dependsOn`, что гарантирует наличие `effective.json`.

[[spec-validate-task]]
== SPEC-15.6: ValidateTask

=== Назначение

Валидация DSL-конфигурации без полной обработки.

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()

    val dependanger = Dependanger.fromMetadata(metadata).build()

    val warnCount = result.diagnostics.warnings.size

        throw GradleException(
            "Dependanger validation failed with $errCount error(s)."
        )
    }
}

[[spec-check-updates-task]]
== SPEC-15.8: CheckUpdatesTask

=== Назначение

Проверка доступных обновлений зависимостей.
Standalone задача: запускает собственный pipeline с `DEFAULT` preset и включает **только** `UpdateCheckProcessor`.

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val failOnError = extension.failOnError.get()

    // DEFAULT preset + только update-check feature
    val dependanger = Dependanger.fromMetadata(metadata)
        .configureProcessing { enableOptional("update-check") }
        .build()

    val result = runBlocking { dependanger.process() }

    if (!result.isSuccess && failOnError) {
        throw GradleException("Dependanger: Failed to check updates")
    }

    val effective = result.effective ?: return
    val updates = effective.updates

    if (updates.isEmpty()) {
        logger.lifecycle("Dependanger: All dependencies are up to date.")
    } else {
        logger.lifecycle("Dependanger: Available updates:")
        updates.forEach { update ->
            logger.lifecycle("  ${update.alias}: ${update.currentVersion} -> ${update.availableVersion} (${update.type})")
        }
        logger.lifecycle("  Total: ${updates.size} updates available")
    }
}
----

NOTE: Используется `DEFAULT` preset (12 core-процессоров) + `enableOptional("update-check")` для включения только UpdateCheckProcessor.
НЕ используется `STRICT`, чтобы не запускать SecurityCheck, LicenseCheck и другие ненужные feature-процессоры.

[[spec-analyze-task]]
== SPEC-15.9: AnalyzeTask

=== Назначение

Расширенный анализ совместимости зависимостей (CustomRule handlers через SPI).
Standalone задача: запускает собственный pipeline с `DEFAULT` preset и включает **только** `CompatibilityCheckProcessor`.

NOTE: Базовый анализ совместимости (JdkRequirement, MutualExclusion, VersionConstraint) выполняется core-процессором `CompatRulesProcessor` (order=66), который включён в DEFAULT preset.
`CompatibilityCheckProcessor` (order=110) добавляет CustomRule handlers.

=== Реализация

[source,kotlin]
----
@TaskAction
public fun execute() {
    val extension = project.extensions.getByType(DependangerExtension::class.java)
    val metadata = extension.dsl.toMetadata()
    val failOnError = extension.failOnError.get()

    // DEFAULT preset + только compatibility-analysis feature
    val dependanger = Dependanger.fromMetadata(metadata)
        .jdkVersion(Runtime.version().feature())
        .configureProcessing { enableOptional("compatibility-analysis") }
        .build()

    val result = runBlocking { dependanger.process() }

    // Compatibility issues из обоих процессоров (CompatRulesProcessor + CompatibilityCheckProcessor)
    val errors = result.diagnostics.errors.filter { it.code?.startsWith("COMPAT") == true }
    val warnings = result.diagnostics.warnings.filter { it.code?.startsWith("COMPAT") == true }

    if (errors.isEmpty() && warnings.isEmpty()) {
        logger.lifecycle("Dependanger: No compatibility issues found.")
    } else {
        errors.forEach { logger.error("Dependanger COMPAT ERROR: ${it.message}") }
        warnings.forEach { logger.warn("Dependanger COMPAT WARN: ${it.message}") }
        logger.lifecycle("Dependanger: Analysis complete: ${errors.size} errors, ${warnings.size} warnings")
    }

    if (errors.isNotEmpty() && failOnError) {
        throw GradleException("Dependanger: Compatibility analysis found ${errors.size} error(s).")
    }
}

[[helper-methods]]
== Общие вспомогательные методы

Рекомендуется вынести в utility object:

[source,kotlin]
----
// DependangerTaskHelper.kt (новый файл)
internal object DependangerTaskHelper {

    fun getExtension(project: Project): DependangerExtension =
        project.extensions.getByType(DependangerExtension::class.java)

    fun buildMetadata(extension: DependangerExtension): DependangerMetadata =
        extension.dsl.toMetadata()

    /**
     * Читает effective.json из outputDir.
     * Используется генерирующими задачами (TomlTask, BomTask, ListVersionsTask),
     * которые зависят от GenerateEffectiveTask через Gradle dependsOn.
     */
    fun readEffective(outputDir: File): EffectiveMetadata {
        val effectiveFile = outputDir.resolve("effective.json")
        if (!effectiveFile.exists()) {
            throw GradleException(
                "effective.json not found in $outputDir. " +
                "Ensure dependangerGenerateEffective has been executed."
            )
        }
        val format = EffectiveJsonFormat()
        return format.read(effectiveFile.toPath())
    }

    fun handleProcessingErrors(
        result: DependangerResult,
        failOnError: Boolean,
        logger: Logger,
    ) {
        result.diagnostics.errors.forEach { logger.error("Dependanger ERROR: ${it.message}") }
        result.diagnostics.warnings.forEach { logger.warn("Dependanger WARN: ${it.message}") }

        if (result.diagnostics.hasErrors && failOnError) {
            throw GradleException(
                "Dependanger processing failed with ${result.diagnostics.errors.size} error(s)."
            )
        }
    }

    fun ensureOutputDir(extension: DependangerExtension): File {
        val dir = extension.outputDirectory.get().asFile
        dir.mkdirs()
        return dir
    }
}

[[incremental-builds]]
== Поддержка инкрементальных билдов

=== Генерирующие задачи (файловая цепочка)

Архитектура с файловой цепочкой уже обеспечивает базовый incremental build для генерирующих задач:

1. **GenerateMetadataTask**: `@Input` = fingerprint DSL, `@OutputFile` = metadata.json
2. **GenerateEffectiveTask**: `@Input` = fingerprint DSL, `@OutputFile` = effective.json
3. **GenerateTomlTask**: `@InputFile` = effective.json, `@OutputFile` = libs.versions.toml
4. **GenerateBomTask**: `@InputFile` = effective.json, `@OutputFile` = bom.pom.xml
5. **ListVersionsTask**: `@InputFile` = effective.json

Если DSL не изменился -> effective.json не перегенерируется -> TOML/BOM UP-TO-DATE.

=== Аналитические задачи

Аналитические задачи (CheckUpdates, SecurityCheck, LicenseCheck, ResolveTransitives) выполняются заново при каждом запуске.
Это корректно, т.к. они зависят от внешнего состояния (Maven Central, OSV API, ClearlyDefined) и кэширование их результатов на уровне Gradle нежелательно.

=== Будущая оптимизация

Для более надёжной поддержки incremental builds рекомендуется заменить `hashCode()` fingerprint на JSON-сериализацию `DependangerMetadata` как `@Input`:

[source,kotlin]
----
@get:Input
public val metadataJson: String
    get() = JsonSerializationFormat().serialize(extension.dsl.toMetadata())
----

Это даёт Gradle точную информацию об изменениях без коллизий hashCode.
