Сериализация модели `DependangerMetadata` в промежуточный формат (JSON) через расширяемый SPI.

== Метаданные

[cols="1,2"]
|===
| ID | F002
| Название | Metadata Serialization
| Приоритет | P1 (критично для MVP)

== Описание

Metadata Serialization преобразует `DependangerDsl` (модель в памяти) в сериализуемый `DependangerMetadata` для:

* Хранения между запусками
* Передачи между этапами pipeline
* Кеширования для инкрементальной обработки

== User Stories

=== US-001: Сохранение метаданных в JSON

[quote]
____
Как пользователь Gradle plugin, я хочу чтобы результат DSL сохранялся в файл, чтобы другие таски могли использовать его без повторного парсинга DSL.

| Вызываю `dsl.toMetadata()` и `format.write(metadata, path)`
| Создаётся файл `metadata.json` с валидным JSON

| AC-001-2

| Возвращается `DependangerMetadata` с теми же данными

| Сериализую и десериализую обратно

| AC-001-4

| AC-001-5
| Gradle task `dependangerGenerateMetadata`
| Запускаю task
| Файл создан в `build/dependanger/metadata.json`
|===

== SPI: SerializationFormat

=== Описание интерфейса

Интерфейс `SerializationFormat<T>` предоставляет точку расширения для добавления новых форматов сериализации.

**Основные возможности:**

* `formatId` — уникальный идентификатор формата (например: "json", "yaml")
* `fileExtension` — расширение файла (например: ".json", ".yaml")
* `serialize()` / `deserialize()` — преобразование модели в формат и обратно
* `write()` / `read()` — работа с файловой системой

=== Реализации

[cols="1,2,2",options="header"]
|===
| Format ID | Модуль | Технология

| `json`
| `dependanger-metadata-json`
| kotlinx.serialization JSON

| `yaml`
| (планируется)
| kaml

| `toml`
| (планируется)
| ktoml
|===

== Модель сериализации

=== DependangerMetadata

Корневая структура модели сериализации содержит:

* `schemaVersion` — версия схемы для обратной совместимости
* `versions` — именованные версии с fallback условиями
* `libraries` — библиотеки с метаданными (включая deprecation)
* `plugins` — Gradle plugins
* `bundles` — группы библиотек
* `bomImports` — импорт версий из внешних Maven BOM
* `targetPlatforms` — целевые платформы (JDK, Android, KMP)
* `distributions` — профили фильтрации
* `compatibility` — правила совместимости
* `presets` — именованные presets настроек
* `settings` — глобальные настройки

NOTE: `bomImports` и `targetPlatforms` — это разные сущности. `bomImports` используется для импорта версий из внешних BOM, а `targetPlatforms` — для определения целевых платформ и проверки совместимости.

=== Вложенные структуры

[cols="1,2",options="header"]
|===
| Структура | Описание

| `VersionMetadata`
| Версия с value и fallback условиями

| `FallbackCondition`
| Условие для fallback версии: `type` + параметры (jdkBelow, kotlinVersionAtLeast, etc.)

| `LibraryMetadata`
| Maven координаты, теги, требования (`requires`), deprecation info, тип (library/platform)

| `DeprecationInfo`
| Информация об устаревании: `replacedBy`, `message`, `since`, `removalVersion`

| `RequirementsMetadata`
| Требования к окружению: JDK версия, Kotlin версия

| `PluginMetadata`
| Gradle plugin ID и версия

| `BundleMetadata`
| Список библиотек и включаемых бандлов (extends)

| `BomImportMetadata`
| Maven координаты BOM для импорта версий

| `TargetPlatformMetadata`
| Constraints для JDK, Android, поддерживаемые Kotlin targets

| `DistributionMetadata`
| Spec фильтрации с типизированными фильтрами

| `FilterSpec`
| Набор фильтров: `byTags`, `byGroups`, `byAliases`, `byBundles`, `byDeprecated`

| `CompatibilityRuleMetadata`
| Правило совместимости: тип, libraries, severity, message

| `PresetMetadata`
| Именованный preset с bundles, distributions, settings

| `SettingsMetadata`
| Глобальные настройки валидации и генерации
|===

=== Детальные структуры

==== FallbackCondition

Условие для выбора fallback версии.

[source,json]
----
// Простое условие
{ "type": "jdkBelow", "version": 17 }

// Составное условие (AND)
{
  "type": "all",
  "conditions": [
    { "type": "jdkBelow", "version": 17 },
    { "type": "kotlinVersionBelow", "version": "2.0" }
  ]
}

// Составное условие (OR)
{
  "type": "any",
  "conditions": [
    { "type": "jdkBelow", "version": 11 },
    { "type": "environment", "key": "LEGACY", "value": "true" }
  ]
}

// Инверсия
{ "type": "not", "condition": { "type": "distribution", "name": "production" } }
----

**Типы условий:**

[cols="1,2",options="header"]
|===
| Type | Параметры

| `jdkBelow` | `version: Int`
| `jdkAtLeast` | `version: Int`
| `jdkBetween` | `min: Int`, `max: Int`
| `kotlinVersionBelow` | `version: String`
| `kotlinVersionAtLeast` | `version: String`
| `gradleVersionBelow` | `version: String`
| `gradleVersionAtLeast` | `version: String`
| `distribution` | `name: String`
| `environment` | `key: String`, `value: String`
| `all` | `conditions: List<FallbackCondition>`
| `any` | `conditions: List<FallbackCondition>`
| `not` | `condition: FallbackCondition`
|===

==== FilterSpec

Спецификация фильтрации для distribution.

[source,json]
----
{
  "filters": [
    {
      "type": "byTags",
      "include": { "anyOf": ["web", "server"] },
      "exclude": { "anyOf": ["deprecated", "legacy"] }
    },
    {
      "type": "byGroups",
      "include": { "matching": ["org.springframework:*"] },
      "exclude": { "matching": ["com.legacy:*"] }
    },
    {
      "type": "byBundles",
      "include": ["spring-backend", "kotlin-core"]
    },
    {
      "type": "byDeprecated",
      "exclude": true
    }
  ]
}
----

**Логика фильтрации:**

* Несколько `include` в одном фильтре — **OR**
* Несколько `exclude` в одном фильтре — **OR**
* `include` + `exclude` — include применяется первым, затем exclude
* Несколько фильтров разных типов — **AND**

== Gradle Task

=== dependangerGenerateMetadata

Задача `dependangerGenerateMetadata` сериализует DSL модель в файл.

**Inputs:**

* `DependangerMetadata` из DSL extension

**Outputs:**

* `build/dependanger/metadata.json`

[[not-serialized]]
== Что НЕ сериализуется

Следующие элементы DSL являются **runtime-only** и не включаются в `metadata.json`:

[cols="1,2",options="header"]
|===
| Элемент | Причина

| `processing { }`
| Конфигурация pipeline — каждый интерфейс (Gradle Plugin, CLI) настраивает pipeline самостоятельно

| `ProcessingPreset`
| Preset pipeline (DEFAULT, MINIMAL, STRICT) — runtime настройка

| `withBomImport()`, `withUpdateCheck()`, etc.
| Builder методы для включения процессоров — runtime настройка

| `disabledProcessors`
| Список отключённых процессоров — runtime настройка

| Resolved versions
| Резолвинг версий происходит в xref:F003-processing-pipeline.adoc[Processing Pipeline], не на этапе сериализации
|===

IMPORTANT: `metadata.json` хранит **декларативное описание** зависимостей.
Все runtime решения (какие процессоры включить, какую версию использовать) принимаются позже в xref:F003-processing-pipeline.adoc[Processing Pipeline].

=== Пример: DSL vs Metadata

[source,kotlin]
----
// DSL (в памяти)
dependanger {
    versions { version("kotlin", "2.1.20") }
    libraries { library("kotlin-stdlib", "org.jetbrains.kotlin:kotlin-stdlib", version = versionRef("kotlin")) }

    // Runtime-only: НЕ сериализуется
    processing {
        preset = ProcessingPreset.STRICT
        withBomImport()
        withUpdateCheck()
    }
}
----

В `metadata.json` будут только `versions` и `libraries`.
Блок `processing` игнорируется при сериализации.

== Версионирование схемы

[cols="1,2",options="header"]
|===
| Версия | Изменения

| 1.0
| Начальная версия

| 1.1
| (планируется) Добавлены kmpTargets в Library
|===

При десериализации:

* Проверяется `schemaVersion`
* Поддерживается миграция между версиями
* Неизвестные поля игнорируются (forward compatibility)

== Edge Cases

[cols="1,3,2",options="header"]
|===
| Сценарий | Описание | Ожидаемое поведение

| Невалидный JSON
| Файл `metadata.json` содержит синтаксическую ошибку JSON
| ERROR: "Failed to parse metadata.json: [parse error details]"

| Неизвестная версия схемы
| `schemaVersion` в файле = "99.0", текущая поддерживаемая = "1.1"
| ERROR: "Unsupported schema version '99.0'. Supported: 1.0, 1.1"

| Старая версия схемы
| `schemaVersion` = "1.0", текущая = "1.1"
| Успешная миграция, WARNING: "Migrating from schema 1.0 to 1.1"

| Отсутствует schemaVersion
| JSON файл без поля `schemaVersion`
| ERROR: "Missing required field 'schemaVersion'"

| Неизвестные поля в JSON
| JSON содержит поле `"unknownField": "value"`
| Поле игнорируется (forward compatibility), десериализация успешна

| Пустой JSON файл
| Файл существует, но содержит `{}`
| ERROR: "Missing required field 'schemaVersion'"

| Файл не существует
| Путь указывает на несуществующий файл
| ERROR: "File not found: [path]"

| Нет прав на запись
| Директория существует, но нет прав на запись
| ERROR: "Cannot write to [path]: Permission denied"

| Очень большой файл
| metadata.json содержит 10000+ библиотек
| Успешная обработка (возможно медленнее)

| Специальные символы в значениях
| Библиотека с description содержащим unicode, emoji, переносы строк
| Корректная сериализация с экранированием

| Циклические ссылки в модели
| (не должно быть возможно по дизайну DSL)
| Валидация в DSL предотвращает, но если попало — ERROR при сериализации

| Null значения
| Опциональные поля (tags, requires) = null
| Поля опускаются в JSON или сериализуются как null (configurable)

| Дубликаты ключей
| JSON с дубликатами: `{"kotlin": {...}, "kotlin": {...}}`
| ERROR: "Duplicate key 'kotlin' in libraries"

| Кодировка файла
| Файл в UTF-16 вместо UTF-8
| ERROR или автодетект (implementation-specific)

| Concurrent writes
| Два процесса пишут в один файл одновременно
| Использовать file locking или atomic write (rename)
