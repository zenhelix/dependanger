= Обзор проекта: Dependanger
:toc: left
:toclevels: 3
:sectnums:
:icons: font

[abstract]
--
Dependanger — инструмент для централизованного управления зависимостями Kotlin/Java проектов.
Позволяет создавать каталоги библиотек с метаданными, формировать готовые наборы (бандлы), импортировать версии из внешних BOM и генерировать стандартные артефакты (Gradle Version Catalog, Maven BOM).
--

== Цели и видение

=== Проблема

Разработчики Kotlin/Java сталкиваются с комплексом проблем при управлении зависимостями:

**1. Разрозненность**

* Каждый проект самостоятельно определяет версии библиотек
* Нет единого источника истины для команды/организации
* Дублирование усилий по исследованию и выбору библиотек

**2. Сложность выбора**

* Новым разработчикам трудно понять, какие библиотеки использовать
* Нет контекста: почему выбрана именно эта библиотека
* Отсутствует информация о совместимости между библиотеками

**3. Устаревание и совместимость**

* Трудно отслеживать выход новых версий
* Сложно понять, какие версии библиотек совместимы друг с другом
* Обновление одной библиотеки может сломать совместимость с другими

=== Решение

Dependanger предлагает:

**Централизованный каталог с метаданными**

* Единое место для определения всех зависимостей через Kotlin DSL
* Описания, теги, информация о совместимости для каждой библиотеки
* Версии с условными fallback (JDK версия, Kotlin версия, Distribution)

**Ролевые бандлы и Distributions**

* **Bundles** — именованные группы библиотек (`kotlin-core`, `spring-web`)
* **Distributions** — профили генерации с фильтрацией (`android`, `backend`, `minimal`)
* Проверенные комбинации версий через compatibility rules

**Импорт внешних BOM**

* Импорт версий из внешних Maven BOM (Spring Boot BOM, Kotlin BOM)
* Автоматическое наследование версий без дублирования
* Приоритет: явно указанные версии > импортированные из BOM

**Два интерфейса с разными сценариями**

* **Gradle Plugin** — Kotlin DSL в `build.gradle.kts`, интеграция с Gradle lifecycle
* **CLI** — standalone команды для CI/CD и скриптовой автоматизации

**Многоэтапный pipeline обработки**

[mermaid]
----
flowchart LR
    DSL["Kotlin DSL"]
    META["metadata.json"]
    PIPE["Pipeline"]
    EFF["EffectiveMetadata"]
    OUT["TOML / BOM"]

    DSL -->|"toMetadata()"| META
    META -->|"Process"| PIPE
    PIPE -->|"Generate"| EFF
    EFF --> OUT
----

Pipeline выполняет: резолвинг версий, импорт из BOM, фильтрацию, валидацию, проверку обновлений.
См. xref:02-features/F003-processing-pipeline.adoc[F003: Processing Pipeline] для деталей.

[[glossary]]
== Глоссарий

Краткие определения ключевых понятий.
Полное описание с атрибутами и примерами: xref:01-concepts/domain-model.adoc[Доменная модель].

[cols="1,3",options="header"]
|===
| Термин | Краткое определение

| DependangerDsl
| Точка входа Kotlin DSL. Содержит блоки `versions`, `libraries`, `plugins`, `bundles`, `bomImports`, `targetPlatforms`, `distributions`, `compatibility`, `settings`, `presets`.

| DependangerMetadata
| Сериализуемое представление конфигурации. Промежуточный формат (`metadata.json`).

| EffectiveMetadata
| Результат обработки pipeline. Все версии разрешены, фильтры применены. Готов для генерации (`effective-\{distribution\}.json`).

| Library
| Зависимость с Maven координатами (`group:artifact`), версией и метаданными.

| Plugin
| Gradle plugin с alias, plugin ID и версией.

| Bundle
| Именованная группа библиотек для переиспользования.

| BomImport
| Импорт версий из внешнего Maven BOM/Platform (Spring Boot BOM, Kotlin BOM).

| Platform Library
| Библиотека типа BOM для экспорта в TOML с `type=pom`. Потребители подключают через `platform()`.

| TargetPlatform
| Целевая платформа с ограничениями (JDK version, Kotlin targets).

| Distribution
| Профиль генерации артефактов с фильтрацией библиотек.

| Version Alias
| Именованная версия с поддержкой fallback условий.

| Version Source
| Источник версии: DECLARED (явно в DSL) или BOM_IMPORT (из BOM).

| Compatibility Rule
| Правило проверки совместимости между библиотеками или с окружением.

| Settings
| Глобальные настройки валидации и генерации.

| Preset
| Именованный пресет конфигурации (bundles, distributions, settings) для переиспользования.
|===

== Границы системы

=== В области (In Scope)

**Определение зависимостей (Kotlin DSL):**

* Libraries — библиотеки с Maven координатами, тегами, метаданными
* Plugins — Gradle plugins с версиями
* Versions — именованные версии с fallback условиями
* Bundles — группы библиотек для переиспользования
* BomImports — импорт версий из внешних Maven BOM
* TargetPlatforms — целевые платформы с ограничениями
* Distributions — профили генерации с фильтрацией
* Compatibility rules — правила проверки совместимости
* Settings — глобальные настройки

**Генерация артефактов:**

* Gradle Version Catalog (TOML)
* Maven BOM (XML)

**Автоматизация:**

* Update Check — проверка доступных обновлений
* Compatibility Analysis — анализ совместимости
* Report Generation — единый отчёт в JSON

**Интерфейсы:**

* Gradle Plugin — Kotlin DSL, интеграция с Gradle
* CLI — standalone команды

**Расширяемость (SPI):**

* `SerializationFormat<T>` — форматы сериализации
* `EffectiveMetadataProcessor` — процессоры pipeline
* `ArtifactGenerator<T>` — генераторы артефактов

=== Вне области (Out of Scope)

* Автоматическое обновление версий (только отчет)
* Хостинг/публикация артефактов (только генерация)
* Графический интерфейс
* Поддержка не-JVM языков
* Разрешение конфликтов транзитивных зависимостей

== Quick Start Checklist

Чек-лист для быстрого старта с Dependanger:

=== Для разработчика (5 минут)

. [ ] Добавить Gradle plugin в `build.gradle.kts`
. [ ] Создать DSL конфигурацию с библиотеками
. [ ] Запустить `./gradlew dependangerGenerate`
. [ ] Подключить сгенерированный `libs.versions.toml`

=== Для Team Lead (15 минут)

. [ ] Определить стратегию: личный каталог или командный
. [ ] Выбрать distributions (android, backend, minimal)
. [ ] Настроить bundles для типовых проектов
. [ ] Добавить CI проверку через CLI

=== Для DevOps (10 минут)

. [ ] Интегрировать CLI в CI/CD pipeline
. [ ] Настроить `check-updates` для отчёта об обновлениях
. [ ] Добавить `validate` как pre-merge check
