= ADR-004: SPI-based расширяемость системы
:icons: font

== Метаданные

[cols="1,2"]
|===
| Дата | 2026-01-15
| Авторы | Dependanger Team
| Затронутые компоненты | Все модули
|===

== Контекст

Dependanger должен поддерживать расширение функциональности без модификации core кода:

* Новые форматы сериализации (YAML, TOML input)
* Новые генераторы артефактов (Gradle Catalog, Bazel)
* Кастомные процессоры pipeline
* Альтернативные стратегии фильтрации

Необходим механизм расширения, который:

1. Не требует форка проекта
2. Позволяет добавлять функциональность через зависимости
3. Имеет стабильные контракты

== Решение

Выбран **Service Provider Interface (SPI)** подход с явными интерфейсами расширения.

[mermaid,target=adr004-spi-architecture,format=svg]
----

== Обоснование

=== Рассмотренные альтернативы

**1. Plugin system с runtime discovery:**

[source,kotlin]
----
ServiceLoader.load(DependangerPlugin::class.java)
----

*Недостаток:* Сложность отладки, неявные зависимости, проблемы с classpath.

**2. Callback/Hook based:**

[source,kotlin]
----
dependanger.onLibraryProcessed { library -> ... }
----

*Недостаток:* Ограниченные возможности, сложность управления порядком.

**3. Inheritance based:**

[source,kotlin]
----
class MyDependanger : Dependanger() { override fun ... }
----

*Недостаток:* Fragile base class problem, сложность версионирования.

=== Преимущества выбранного подхода

[cols="1,2"]
|===
| Критерий | Обоснование

| Explicit contracts
| Интерфейсы явно определяют точки расширения

| Compile-time safety
| Ошибки обнаруживаются при компиляции

| Testability
| Легко создавать моки и тестовые реализации

| Versioning
| Интерфейсы могут эволюционировать с deprecation

| No magic
| SPI контракты явные. Runtime discovery через ServiceLoader — опциональный механизм наряду с явной регистрацией через `DependangerBuilder`
|===

== Последствия

=== Положительные

* Чёткие контракты для расширения
* Простое тестирование с mock реализациями
* Возможность создания сторонних расширений
* Стабильный API для публичного использования

=== Отрицательные

* Больше интерфейсов для поддержки
* Изменения в интерфейсах требуют careful versioning
* ServiceLoader discovery добавляет неявность — реализация в classpath автоматически активируется

=== Нейтральные

* Default реализации предоставляются в core модулях
* Builder pattern для конфигурации расширений

== Реализация

=== Основные SPI интерфейсы

==== SerializationFormat<T>

Формат сериализации для метаданных.

[source,kotlin]
----
interface SerializationFormat<T> {
    val formatId: String
    val fileExtension: String
    val description: String
    fun serialize(metadata: DependangerMetadata): T
    fun deserialize(input: T): DependangerMetadata
    fun write(metadata: DependangerMetadata, path: Path)
    fun read(path: Path): DependangerMetadata
}
----

Реализации:

* `JsonSerializationFormat` — JSON для DependangerMetadata

==== EffectiveMetadataProcessor

Процессор в pipeline обработки.

[source,kotlin]
----
interface EffectiveMetadataProcessor {
    val id: String
    val phase: ProcessingPhase
    val order: Int
    val isOptional: Boolean
    val description: String
    fun supports(context: ProcessingContext): Boolean
    suspend fun process(metadata: EffectiveMetadata, context: ProcessingContext): EffectiveMetadata
}
----

Реализации:

* 12 core процессоров (MetadataConversion, VersionResolver, Validation и др.)
* `CompatibilityCheckProcessor` — анализ совместимости (feature module)

==== ArtifactGenerator<T>

Генератор выходных артефактов.

[source,kotlin]
----
interface ArtifactGenerator<T> {
    val generatorId: String
    val description: String
    val fileExtension: String
    fun generate(effective: EffectiveMetadata): T
    fun write(artifact: T, output: Path)
}
----

Реализации:

* `TomlGenerator` — TOML Version Catalog
* `BomGenerator` — Maven BOM

==== LibraryFilter

Стратегия фильтрации библиотек.

[source,kotlin]
----
interface LibraryFilter {
    val filterId: String
    fun filter(libraries: List<Library>): List<Library>
}
----

=== Регистрация расширений

Два способа регистрации:

[source,kotlin]
----
// 1. Через DependangerBuilder (явная регистрация)
val dependanger = Dependanger.fromDsl { /* ... */ }
    .addProcessor(MyCustomProcessor())
    .build()

// 2. Через ServiceLoader (автоматическое обнаружение)
// Файл: META-INF/services/io.github.zenhelix.dependanger.effective.pipeline.EffectiveMetadataProcessor
// Содержимое: com.example.MyCustomProcessor
----

=== Пример кастомного процессора

[source,kotlin]
----
class SecurityScanProcessor(
    private val vulnerabilityDb: VulnerabilityDatabase
) : EffectiveMetadataProcessor {
    override val id = "security-scan"
    override val phase = ProcessingPhase.SECURITY_CHECK
    override val order = 120
    override val isOptional = true
    override val description = "Checks libraries for known vulnerabilities"

    override fun supports(context: ProcessingContext): Boolean = true

    override suspend fun process(
        metadata: EffectiveMetadata,
        context: ProcessingContext
    ): EffectiveMetadata {
        val diagnostics = metadata.libraries.values.flatMap { library ->
            val vulns = vulnerabilityDb.findVulnerabilities(
                library.group, library.artifact, library.version?.value
            )
            vulns.map { vuln ->
                Diagnostics.warning(
                    code = "VULNERABILITY_FOUND",
                    message = "Library $\{library.alias\} has vulnerability: $\{vuln.id\}",
                    processorId = id,
                )
            }
        }.fold(Diagnostics.EMPTY, Diagnostics::plus)

        return metadata.copy(diagnostics = metadata.diagnostics + diagnostics)
    }
}

// Использование
val dependanger = Dependanger.fromDsl { /* ... */ }
    .addProcessor(SecurityScanProcessor(MyVulnDb()))
    .build()
----

== Версионирование API

=== Stability markers

[source,kotlin]
----
@RequiresOptIn(
    message = "This API is experimental and may change",
    level = RequiresOptIn.Level.WARNING
)
annotation class ExperimentalDependangerApi

@ExperimentalDependangerApi
interface NewFeatureProcessor : EffectiveMetadataProcessor
----

=== Deprecation policy

1. Deprecated API сохраняется минимум 2 minor версии
2. Deprecated API помечается `@Deprecated` с указанием замены
3. Удаление только в major версиях
