== Обзор раздела

Architecture Decision Records (ADR) фиксируют важные архитектурные решения вместе с их контекстом и последствиями.
Это позволяет понять ПОЧЕМУ система устроена определенным образом.

== Статусы решений

* 🟡 **Предложено** — решение на обсуждении
* 🟢 **Принято** — решение утверждено и применяется
* 🔴 **Отклонено** — решение рассмотрено и отклонено
* ⚫ **Устарело** — решение было заменено другим

== Когда создавать ADR

Создавайте ADR для решений, которые:

* Влияют на архитектуру системы
* Сложно отменить после реализации
* Требуют обсуждения между участниками команды
* Могут вызвать вопросы "почему сделано так?" в будущем

=== Примеры решений для ADR

* Выбор библиотеки/фреймворка
* Структура модулей/пакетов
* Паттерны проектирования
* Стратегии версионирования
* Форматы данных и протоколы

=== НЕ создавайте ADR для

* Локальных рефакторингов
* Исправления багов
* Добавления тестов
* Обновления зависимостей (если не меняется мажорная версия)

== Как создать ADR

. Скопируйте шаблон: `docs/_templates/adr-template.adoc`
. Переименуйте: `ADR-###-краткое-название.adoc`
. Заполните все секции
. Отправьте на ревью
. После утверждения обновите статус и добавьте в реестр выше
