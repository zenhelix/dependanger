# /doc-polish

Финальная редактура документа — очистка от паразитных слов, упрощение формулировок, повышение читаемости.

## Использование

```
/doc-polish [путь]               # файл или папка
/doc-polish                      # редактировать активный документ
```

**Режимы:**

- **Файл:** редактура одного документа
- **Папка:** последовательная редактура всех `.adoc` файлов уровня L2+

## Примеры

```
/doc-polish docs/02-features/F001-registry-dsl.adoc   # один файл
/doc-polish docs/02-features/                         # вся папка features
/doc-polish docs/                                     # весь docs
```

## Инструкции для агента

$ARGUMENTS содержит путь к документу или папке. Если пустой — запросить путь.

### Определение режима

1. **Проверить, что указано в $ARGUMENTS:**
    - Если путь заканчивается на `.adoc` → режим одного файла
    - Если путь — директория → режим папки
    - Если пустой → запросить путь

2. **Режим папки:**
    - Найти все `.adoc` файлы в папке (рекурсивно)
    - Отфильтровать только документы уровня L2+ (готовы к polish)
    - Отсортировать по имени
    - Показать план и запросить подтверждение
    - Обработать каждый файл по очереди

### Когда применять

- После достижения уровня L2 или L3
- Перед финальным STOP
- Когда документ "готов по содержанию", но текст можно улучшить

### Шаги выполнения

1. **Прочитать документ** полностью

2. **Найти слова-паразиты:**
    - "в целом", "в принципе", "по сути", "как бы", "собственно"
    - "достаточно", "довольно", "весьма", "очень", "крайне"
    - "некоторый", "определённый", "соответствующий", "надлежащий"

3. **Найти канцеляризмы:**
    - "осуществлять" → конкретный глагол
    - "производить" → конкретный глагол
    - "является" → тире или убрать
    - "данный", "указанный", "вышеупомянутый" → "этот" или убрать
    - "в случае если" → "если"
    - "с целью" → "чтобы"
    - "в связи с тем что" → "потому что"

4. **Найти проблемы структуры:**
    - Предложения > 20 слов → разбить
    - Абзацы > 5 предложений → разбить
    - Пассивный залог → активный
    - Отглагольные существительные → глаголы

5. **Найти неконкретные формулировки:**
    - "быстро", "медленно" → конкретное время
    - "много", "мало" → числа
    - "большой", "маленький" → размер
    - "правильно", "корректно" → критерии

6. **Найти повторения (DRY):**
    - Похожие абзацы в разных секциях
    - Описания, которые есть в других документах
    - Копипаста определений терминов
    - Дублирование примеров кода

   **Заменять на:**
    - `<<anchor>>` — ссылка внутри документа
    - `xref:path.adoc#anchor[текст]` — ссылка на другой документ
    - `include::path[]` — вставка переиспользуемого блока

7. **Сформировать отчёт** с предложениями замен

8. **Применить изменения** (с подтверждением пользователя)

### Словарь замен

| Найти                     | Заменить на     |
|---------------------------|-----------------|
| осуществлять проверку     | проверять       |
| производить вычисление    | вычислять       |
| выполнять обработку       | обрабатывать    |
| является X                | — X / это X     |
| данный/указанный          | этот / (убрать) |
| в настоящее время         | сейчас          |
| в случае если             | если            |
| с целью                   | чтобы           |
| в связи с тем что         | потому что      |
| представляет собой        | это             |
| оказывает влияние         | влияет          |
| принимает участие         | участвует       |
| имеет возможность         | может           |
| предоставляет возможность | позволяет       |

```markdown
## Редактура документа: [название]

#### Слова-паразиты (N шт)

| Строка | Было | Стало |

| 12 | "является обязательным" | "обязательно" |

#### Длинные предложения (N шт)

| Строка | Слов | Рекомендация |
|--------|------|--------------|

#### Пассивный залог (N шт)

| Строка | Было | Стало |
|--------|------|-------|
| 45 | "данные будут обработаны" | "система обработает данные" |

#### Повторения (N шт)

| Строка | Что повторяется | Решение |
|--------|-----------------|---------|
| 52-58 | Описание DSL (есть в F001) | `xref:F001-registry-dsl.adoc[DSL]` |

### Итого

- **Слова-паразиты:** N → убрать
- **Канцеляризмы:** N → заменить

- **Пассивный залог:** N → переписать
- **Повторения:** N → заменить на ссылки
```

```markdown

### Статистика читаемости:

| Метрика | Было | Стало |

```

### Примеры трансформаций

**Пример 1: Канцеляризм**

```
❌ "Система осуществляет проверку валидности входных данных
    и производит их обработку в соответствии с установленными правилами"

✅ "Система проверяет и обрабатывает входные данные по правилам"
```

**Пример 2: Длинное предложение**

```
❌ "В случае если пользователь указывает версию библиотеки явно в build.gradle.kts,
    то данная версия имеет приоритет над версией из BOM и будет использована
    при резолюции зависимостей вне зависимости от других настроек."

✅ "Явно указанная версия в build.gradle.kts приоритетнее версии из BOM.
    Она используется при резолюции независимо от других настроек."
```

**Пример 3: Пассивный залог**

```
❌ "Конфигурация должна быть загружена до начала обработки"

✅ "Система загружает конфигурацию перед обработкой"
```

**Пример 4: Неконкретность**

```
❌ "Обработка выполняется достаточно быстро"

✅ "Обработка занимает <100ms для типичного проекта (50 зависимостей)"
```

**Пример 5: Повторение → ссылка**

```asciidoc
❌ "Registry DSL — это Kotlin DSL для декларативного описания зависимостей.
    Он позволяет определять библиотеки, версии и BOM в типобезопасном формате.
    [ещё 10 строк описания DSL, которое уже есть в F001]"

✅ "Зависимости описываются через xref:F001-registry-dsl.adoc[Registry DSL]."
```

**Пример 6: Повторение внутри документа → якорь**

```asciidoc
❌ // В секции "Обзор":
    "Резолюция версий — процесс определения конкретной версии..."

    // В секции "Детали":
    "Резолюция версий — процесс определения конкретной версии..." (копипаста)

✅ // В секции "Обзор":
    [[version-resolution]]
    "Резолюция версий — процесс определения конкретной версии..."

    // В секции "Детали":
    "Подробнее о <<version-resolution,резолюции версий>>..."
```

## Режим папки: пакетная редактура

### Шаги выполнения (режим папки)

1. **Найти файлы:** все `.adoc` в папке (рекурсивно)
2. **Фильтрация:** только документы уровня L2+ (готовы к polish)
3. **Анализ:** подсчитать проблемы в каждом файле
4. **Показать план** с приоритетами
5. **Обработать по очереди** (с подтверждением)

### Формат ответа (режим папки)

```markdown
## Пакетная редактура: [путь к папке]

### Анализ документов

| # | Документ | Уровень | Паразиты | Канцеляризмы | Длинные | Повторы |
|---|----------|---------|----------|--------------|---------|---------|
| 1 | F001-registry-dsl.adoc | L2 | 5 | 8 | 3 | 2 |
| 2 | F003-pipeline.adoc | L2 | 2 | 4 | 1 | 0 |
| 3 | architecture.adoc | L3 | 0 | 2 | 0 | 1 |

**Пропущены (уровень < L2):**

- F002-parsing.adoc — L1, сначала нужен `/doc-iterate`
- ADR-001-kotlinx.adoc — L0, черновик

**Всего проблем:** 28

---

### План редактуры

**Порядок обработки** (по количеству проблем):

1. `F001-registry-dsl.adoc` — 18 проблем
2. `F003-pipeline.adoc` — 7 проблем
3. `architecture.adoc` — 3 проблемы

---

Начать с документа 1 (`F001-registry-dsl.adoc`)?

Опции:

- `да` — начать с первого
- `[N]` — начать с документа N
- `все` — применить все изменения автоматически
- `отчёт` — только показать проблемы, не исправлять
- `отмена` — выйти
```

### Процесс редактуры (режим папки)

После каждого документа:

```markdown
---

## ✅ Завершён: F001-registry-dsl.adoc

**Исправлено:**
- Слов-паразитов: 5
- Канцеляризмов: 8
- Длинных предложений: 3
- Повторений: 2

**Прогресс:** 1/3 документов | Исправлено проблем: 18/28

---

Следующий: `F003-pipeline.adoc` (7 проблем)

Продолжить? (да / пропустить / завершить)
```
