# /doc-iterate

Помочь с следующей итерацией документа — перейти на следующий уровень зрелости.

## Использование

```
/doc-iterate [путь к документу]
/doc-iterate                      # итерировать активный документ
```

## Примеры

```
/doc-iterate docs/02-features/F007-update-check.adoc
/doc-iterate docs/05-decisions/ADR-001-kotlinx-serialization.adoc
```

## Инструкции для агента

$ARGUMENTS содержит путь к документу. Если пустой — запросить путь.

### Шаги выполнения

1. **Прочитать документ** и определить текущий уровень зрелости

2. **Определить целевой уровень** (следующий):
   - L0 → L1 (Minimal Viable)
   - L1 → L2 (Development Ready)
   - L2 → L3 (Complete)

3. **Показать критерии перехода:**

   **L0 → L1:**
   - [ ] Заполнена секция "Описание / Цель"
   - [ ] Есть хотя бы 1 User Story с приоритетом
   - [ ] Определены зависимости от других фич

   **L1 → L2:**
   - [ ] Все User Stories имеют Acceptance Criteria
   - [ ] AC записаны в формате Given/When/Then
   - [ ] Описаны Edge Cases и Error Scenarios
   - [ ] Указаны функциональные требования (FR)

   **L2 → L3:**
   - [ ] Все FR помечены как MUST/SHOULD/MAY
   - [ ] Каждое требование testable
   - [ ] Все stakeholder questions отвечены
   - [ ] Документ прошёл peer review

4. **Проверить текущее состояние** — что уже выполнено из критериев

5. **Предложить конкретные улучшения:**
   - Какие секции заполнить
   - Какие формулировки уточнить
   - Примеры как должно выглядеть

6. **Интерактивно помочь заполнить** недостающее:
   - Задавать вопросы
   - Предлагать формулировки
   - Редактировать документ

### Формат ответа

```markdown
## Итерация документа: [название]

**Текущий уровень:** L# ([название])
**Целевой уровень:** L# ([название])

### Критерии перехода L# → L#

| Критерий | Статус | Что сделать |
|----------|--------|-------------|
| [критерий] | ✅ / ❌ | [действие если ❌] |

### План итерации

**Шаг 1: [Название]**
[Что нужно сделать]

Пример:
```
[Пример как должно выглядеть]
```

**Шаг 2: [Название]**
...

---

Начнём с шага 1?
```

### Пример диалога

**User:** /doc-iterate docs/02-features/F007-update-check.adoc

**Agent:**
```markdown
## Итерация документа: F007-update-check.adoc

**Текущий уровень:** L1 (Minimal Viable)
**Целевой уровень:** L2 (Development Ready)

### Критерии перехода L1 → L2

| Критерий | Статус | Что сделать |
|----------|--------|-------------|
| Все US имеют AC | ❌ | US-002 без AC |
| AC в Given/When/Then | ❌ | Переписать AC-001, AC-002 |
| Edge Cases описаны | ❌ | Добавить 3-5 edge cases |
| FR указаны | ✅ | — |

### План итерации

**Шаг 1: Добавить AC для US-002**

US-002 описывает фильтрацию обновлений, но нет acceptance criteria.

Предлагаю такой AC:
```
| AC-002-1 | Включён флаг --major-only | Запустить update check | Показаны только major обновления |
| AC-002-2 | Библиотека в exclude списке | Запустить update check | Библиотека не проверяется |
```

**Шаг 2: Переписать AC-001 в Given/When/Then**

Текущий AC-001: "Версии резолвятся корректно"

Это не testable. Предлагаю:
```
| AC-001-1 | Library с versionRef("kotlin") | Pipeline выполнен | version = значение из versions["kotlin"] |
```

Начнём с шага 1?
```

### После каждого шага

- Обновить документ
- Проверить критерий
- Показать прогресс
- Перейти к следующему шагу или завершить

### Завершение итерации

```markdown
## Итерация завершена

**Новый уровень:** L2 (Development Ready)

### Что было сделано:
1. ✅ Добавлены AC для US-002
2. ✅ AC переписаны в Given/When/Then
3. ✅ Добавлены Edge Cases

### Следующие шаги:
- Документ готов для начала разработки
- При появлении вопросов — дополнить
- Для L3 нужен peer review

Рекомендация: **STOP** (достаточно для текущего этапа)
```
